# 195. Replace Rd Executable Alias With Rndn

## Problem

The codebase previously introduced `rd` as a short executable alias for `rundown`.

That decision is no longer the desired shorthand. The supported short invocation should be `rndn` instead of `rd`.

If the product keeps documenting or shipping `rd`, the shorthand surface becomes inconsistent with the intended branding and with future examples.

## Desired behavior

After this migration:

- `rndn` is the supported short executable alias for `rundown`.
- `rd` is no longer the preferred alias in docs, help, examples, or package bin mappings.
- Runtime behavior parity remains the same: `rndn <args...>` behaves exactly like `rundown <args...>`.

This migration changes the shorthand executable name only. It does not rename the canonical command from `rundown`.

## Behavior contract

### 1. `rndn` becomes the only supported short executable alias

The package should expose `rndn` as the short alias to the same CLI entrypoint used by `rundown`.

Expected behavior:

- `rndn <args...>` routes to the same compiled entrypoint as `rundown <args...>`.
- All commands, flags, exit codes, help behavior, and side effects are identical.

### 2. `rd` is removed as the documented alias

Any user-facing mention of `rd` as the preferred shorthand should be replaced with `rndn`.

This includes:

- CLI docs,
- README/examples,
- help text snippets,
- packaged bin expectations,
- and tests that currently assert `rd` parity.

### 3. The canonical command remains `rundown`

This migration is not a product rename.

`rundown` remains the canonical executable and command name in help and internal command ownership.

`rndn` is only a convenience alias.

### 4. Packaging-level aliasing remains the correct integration layer

Executable aliasing should continue to be implemented through package bin/shim configuration rather than through argument rewriting.

The same rule that previously applied to `rd` should apply to `rndn`.

## Scope

In scope:

- replace `rd` with `rndn` in package bin exposure,
- replace docs/examples/help references to `rd`,
- update executable-parity tests to use `rndn`,
- and preserve `rundown` behavior unchanged.

Out of scope:

- renaming the canonical executable from `rundown`,
- introducing `rndn`-specific semantics,
- or changing command-level alias rewriting.

## Implementation plan

Primary implementation shape:

1. Update package bin mapping so `rndn` points to the same entry script as `rundown`.
2. Remove `rd` from the supported alias surface.
3. Replace documentation/examples that refer to `rd` with `rndn`.
4. Update parity tests so they compare `rndn` with `rundown` instead of `rd` with `rundown`.
5. Keep `program.name("rundown")` and all canonical help/usage text centered on `rundown`.

## Implementation boundaries

Primary files likely involved:

- `implementation/package.json`
- `implementation/src/presentation/cli.ts`
- `implementation/README.md`
- `README.md`
- docs/examples/help surfaces referencing `rd`
- CLI parity tests referencing executable alias behavior

## Acceptance criteria

- `rndn` is exposed as an executable alias alongside `rundown`.
- `rndn` and `rundown` execute the same CLI entrypoint and produce matching behavior for equivalent inputs.
- User-facing docs and examples stop presenting `rd` as the supported shorthand and use `rndn` instead.
- Existing canonical `rundown` behavior is unchanged.
