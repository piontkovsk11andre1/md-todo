# 178. Remove Memory-Result and Clarify Memory Task Completion

## Problem

The current `memory:` workflow mixes three different concepts without a clear ownership boundary:

- persisted source-local memory artifacts (`.rundown/<source>.memory.md` and `.rundown/memory-index.json`),
- legacy inline task-local `memory-result:` child bullets written into the Markdown source,
- and normal task completion/checkmark handling owned by rundown itself.

That ambiguity showed up directly in the failed run for `migrations/163. Profiles scene.md`:

- the execute worker reported that it marked the `memory:` task complete and added `memory-result:` lines,
- the memory writer also persisted the memory body and index entry successfully,
- and then rundown attempted to mark the same task checkbox complete and failed with `Could not find unchecked checkbox on line 119` because the worker had already changed `[ ]` to `[x]`.

There is also a second inconsistency in the same flow: the verify prompt still showed memory status as unavailable even though the memory file and index had already been written during execute. That means prompt-time memory context is stale relative to the execute-phase write.

Today `memory-result:` behaves like a ghost API surface: it looks canonical in task output, but unlike `get-result:` it is not the authoritative memory store and is not consumed by the memory reader/resolver path. The result is duplicated output, confusing prompts, and a fragile completion contract.

This migration resolves that ambiguity by removing `memory-result:` from the `memory:` workflow entirely.

## Desired behavior

After this migration, the `memory:` workflow must have one clear source of truth and one clear completion owner.

- Persisted memory artifacts remain the authoritative storage for memory capture.
- Rundown owns checkbox completion for `memory:` tasks.
- The execute worker for a `memory:` task must not edit the source task file just to mark completion or inject ad hoc memory annotations.
- Verification and later prompt generation must reflect freshly written memory state instead of stale pre-execute metadata.
- Inline `memory-result:` bullets are no longer part of `memory:` task execution and must not be emitted, refreshed, or interpreted by the runtime.

## Behavior contract

### 1. Persisted memory is the authoritative memory state

The authoritative memory state for a source file is:

- body file: `<source-dir>/.rundown/<source-basename>.memory.md`
- index file: `<source-dir>/.rundown/memory-index.json`

Memory readers, memory validation, prompt memory context, and any memory views must derive state from those artifacts.

Inline `memory-result:` bullets in the source Markdown are deprecated legacy output and must not be treated as authoritative persisted memory.

### 2. Rundown owns `memory:` task completion

For `memory:` / `memorize:` / `remember:` / `inventory:` tasks:

- the worker executes the payload,
- the runtime persists memory through the memory writer,
- and rundown marks the task complete through the normal checkbox lifecycle.

The worker must not check the task box itself.

Any prompts or worker-facing instructions for memory capture must explicitly forbid changing checkbox state or editing the source Markdown task file unless the selected task text itself requested such edits.

### 3. `memory-result:` is removed from `memory:` execution

This migration makes the product decision explicit instead of leaving the current ambiguous state in place.

- `memory-result:` is not emitted during `memory:` task execution.
- Runtime code does not insert, refresh, or preserve `memory-result:` child bullets as part of memory capture.
- Prompt, help, and test documentation must stop describing `memory-result:` as a supported output surface for `memory:` tasks.
- Any leftover `memory-result:` bullets in older Markdown files are treated as non-authoritative legacy content and ignored by memory readers and completion logic.

Leaving `memory-result:` as an implicit side effect of worker output is not acceptable, and preserving it as a supported runtime annotation layer is out of scope for this migration.

### 4. Verification/prompt memory context must refresh after execute writes

When a `memory:` task writes new memory during execute, any later verify or repair prompt for that same task iteration must see the updated memory metadata.

The system must not render verify prompt memory context from a stale pre-execute snapshot when freshly written memory artifacts already exist.

### 5. No double-completion failure on successful memory capture

If memory capture succeeds and persistence succeeds, the task must be able to complete cleanly in one pass.

The runtime must not fail with an unchecked-checkbox error caused by its own execute phase already mutating the task line through worker-authored source edits.

## Scope

In scope:

- Clarify the runtime contract for `memory:` task output versus persisted memory artifacts.
- Remove `memory-result:` behavior from `memory:` task execution and related documentation.
- Ensure `memory:` task completion is owned only once.
- Fix stale memory context in verify/repair prompts after execute writes.
- Add regression coverage for the reproduced failure where a `memory:` task is checked by execute output and then fails during runtime completion.

Out of scope:

- Redesigning the overall memory storage format.
- Inlining full memory bodies into prompts.
- Broad changes to unrelated `get:`, `for:`, or general checkbox semantics beyond what is required to fix the `memory:` path.
