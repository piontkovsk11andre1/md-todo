# 159. TUI status probes

## Problem

The main menu from 158 ships hardcoded right-column placeholders. To make the menu *useful* (not just navigable) every row needs a live one-line status probe sourced from the existing `app.*` API and on-disk state. Probes must be cheap, cached, and never block menu rendering.

## Goal

Add a `status-probes.js` module that supplies a one-line status string per main-menu row. The main menu calls these probes lazily, renders cached values immediately on entry, and silently re-fetches in the background.

## Probe specifications

Each probe returns `{ text: string, tone: "ok" | "warn" | "error" | "muted" }` and never throws. On error the probe returns `{ text: "?", tone: "muted" }`.

| Row | Probe | Source |
|---|---|---|
| Continue | `app.listTasks(...)` to count unchecked tasks across the migrations source; render `"<N> tasks · <M> migrations pending"`. Tone = `warn` if `N > 0`, `ok` if `N === 0`. | already used by current materialize cockpit ([cli-sketch.js#L_estimateTasksWithList](implementation/src/cli-sketch.js)) |
| New Work | check `.rundown/agent.md` exists. If missing, `"agent.md missing!"` tone `error`. Else also check `app.viewWorkerHealthStatus({json:true})` for default worker eligibility; render `"worker ready"` ok, or `"<reason>"` warn. | filesystem + `viewWorkerHealthStatus` |
| Workers | summarize the worker config: `"default ✓ · tui ✓ · <N> fallbacks"` plus a `cooling`/`unavailable` count when present. Tone = `warn` if any cooling, `error` if any unavailable, `ok` otherwise. | `WorkerConfig` (loaded via `workerConfigPort.load`) + worker-health entries |
| Profiles | `<N> defined`. Optionally `· last used: <name>` if a tracked usage record exists; otherwise omit. Tone = `muted` if `N === 0`, else `ok`. | `WorkerConfig.profiles` |
| Settings | count `commands.<name>` and `profiles.<name>` keys present in **local** config. Render `"local + global · <N> effective overrides"` (skip the `global` chunk if no global file). Tone = `muted`. | `rundown config list --scope local --json` (in-process equivalent) |
| Help | static `"docs · website · changelog · keys"`, tone `muted`. | static |

Caching:

- Each probe declares a TTL: Continue 2s, New Work 5s, Workers 10s, Profiles 10s, Settings 30s, Help ∞.
- `getProbeStatus(row)` returns the cached value immediately. If stale, it fires a background refresh and resolves the next render.
- `refreshAllProbes()` is exposed and bound to `r` in the main menu.

Render contract:

- Probe text is right-aligned within its row.
- Long text is truncated with `…` to fit the available width.
- The tone maps to the existing `operationColor` palette (`ok` → green, `warn` → yellow, `error` → red, `muted` → dim).

## Implementation boundaries

Files expected to change:

- new `implementation/src/presentation/tui/status-probes.js`.
- `implementation/src/presentation/tui/scenes/main-menu.js` — wire the probe into row rendering.
- `implementation/src/presentation/tui/index.js` — bind `r` to `refreshAllProbes()`.

Reuse existing `App` capabilities; do not invent new ports.

## Acceptance

- Main menu opens immediately (no probe blocks the first paint).
- Each row renders its real status within one render tick after first paint.
- Pressing `r` triggers a re-fetch and updates the rows on next render.
- A probe failure renders `?` instead of crashing the TUI.
- A repeated open of the main menu within TTL renders cached values without firing the probe.
