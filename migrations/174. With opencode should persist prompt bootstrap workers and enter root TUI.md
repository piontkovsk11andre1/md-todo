rundown with opencode should persist prompt bootstrap workers and enter root TUI

## Intent

Document the expected onboarding contract for `rundown with opencode` when the user wants OpenCode wired for prompt-based execution and then returned to Rundown's own TUI.

This is a behavior/spec migration only. It does not change runtime code by itself.

## Problem statement

Current `rundown with opencode` behavior is producing unexpected worker commands and prompt wiring. The user expectation for the persisted local config is:

```json
{
  "workers": {
    "default": ["opencode", "run", "$bootstrap"],
    "tui": ["opencode", "--prompt", "$bootstrap"]
  }
}
```

Additionally, after `rundown with opencode` completes successfully, Rundown should open the Rundown root TUI instead of just exiting or launching a different direct discuss flow.

## Desired behavior

- `rundown with opencode` writes the OpenCode worker split exactly as:
  - `workers.default = ["opencode", "run", "$bootstrap"]`
  - `workers.tui = ["opencode", "--prompt", "$bootstrap"]`
- The command should preserve unrelated local config keys while updating only the keys owned by this onboarding flow.
- The onboarding flow should finish by entering the Rundown root TUI.
- The post-config handoff should be the same root interface users get from running bare `rundown` in an interactive terminal.
- The command should not silently substitute a different OpenCode prompt transport contract such as `--file $file` unless that contract is explicitly changed in the migration/spec.

## Current discrepancy to resolve

The observed output indicates that `rundown with opencode` is currently generating the wrong command shape and/or wrong prompt flags for this onboarding path.

There is also a UX mismatch in the handoff phase:

- expected: configure workers, then open Rundown TUI
- not expected: configure workers and exit
- not expected: configure workers and jump directly into a discuss worker path that bypasses the root Rundown TUI

## Scope

In scope:
- define the canonical `opencode` onboarding payload for this flow
- define the expected post-configuration handoff to root TUI
- add/update tests and docs so persisted config and runtime handoff match this migration

Out of scope:
- changing unrelated worker presets
- redesigning the rest of the TUI navigation model
- changing non-OpenCode onboarding behavior unless needed for consistency with root-TUI handoff semantics

## Acceptance criteria

1. Running `rundown with opencode` writes `workers.default` as `["opencode", "run", "$bootstrap"]`.
2. Running `rundown with opencode` writes `workers.tui` as `["opencode", "--prompt", "$bootstrap"]`.
3. Unrelated local config keys remain unchanged.
4. Re-running `rundown with opencode` is idempotent.
5. On successful completion in an interactive terminal, the command opens the Rundown root TUI.
6. The flow does not route into a direct discuss session instead of the root TUI.
7. CLI/docs/tests all describe the same OpenCode onboarding contract.

## Implementation notes

- Treat the persisted worker arrays as the source of truth for this onboarding path.
- The TUI handoff should reuse the same root-TUI entry path used by bare `rundown` rather than duplicating a separate interactive launch flow.
- If existing docs or tests currently assume a different OpenCode payload, those surfaces should be reconciled to this migration when the implementation change is taken.
