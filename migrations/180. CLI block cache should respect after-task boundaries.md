# 180. CLI block cache should respect after-task boundaries

## Problem

`--cache-cli-blocks` currently memoizes CLI fenced-block results by `(command, cwd)` only via [implementation/src/application/cached-command-executor.ts](c:/Work/md-todo/implementation/src/application/cached-command-executor.ts). That is intentionally broad enough to reuse the same command output across source expansion, task template expansion, and verify template expansion.

That broad reuse breaks down when the same command appears in a ` ```cli ` fence **after** the selected task in the source document.

Current execution order in [implementation/src/application/prepare-task-prompts.ts](c:/Work/md-todo/implementation/src/application/prepare-task-prompts.ts):

1. Expand CLI fences in the full source document.
2. Re-parse the expanded source and compute `expandedContextBefore` using lines before the selected task.
3. Render the task template from `context`, `task`, and other vars.
4. Expand CLI fences in the task template.
5. Optionally render and expand the verify template.

The default execute template in [implementation/src/domain/defaults.ts](c:/Work/md-todo/implementation/src/domain/defaults.ts) is explicitly scoped to source content *up to but not including the selected unchecked task*. So a CLI fence that sits after the selected task usually does not contribute to the prompt the worker sees during execution.

Despite that, step 1 still executes those after-task CLI fences, and when caching is enabled they populate the shared cache. A later task-template or verify-template fence with the same command then reuses that result even though the source fence was not part of the reached pre-task context.

In other words: the cache currently answers "has this command been executed anywhere in this run?" when the desired question is closer to "has this command occurred in a phase or source region that is already reachable for the current task?"

## Concrete failure mode

Given a source document like this:

````md
# Roadmap

- [ ] Build release

```cli
echo version-from-after-task
```
````

And `.rundown/execute.md` like this:

````md
## Execute

```cli
echo version-from-after-task
```

{{task}}
````

With `run roadmap.md --cache-cli-blocks`, the source-phase fence runs first and warms the shared cache. The execute-template fence then hits the cache instead of executing in the execute phase.

That is wrong for two reasons:

- The after-task source fence was not part of the pre-task context contract.
- It lets a later source region influence execute/verify prompt materialization earlier than intended.

## Goal

Keep cross-phase CLI block caching, but only for CLI blocks that have actually been *reached* for the current task.

After this migration:

- A source CLI fence located before the selected task can still warm the shared cache and be reused by the task template or verify template.
- A source CLI fence located after the selected task may still be expanded for full-source rendering, but it must **not** warm the shared cross-phase cache before the run reaches a phase that legitimately uses it.
- Task-template and verify-template CLI fences continue to use the shared cache exactly as today.
- Cache reuse remains based on command identity and cwd for phases that are already reached; we do **not** fragment the cache by prompt type unless the boundary cannot be modeled any other way.

## Non-goal

- Do not disable source CLI fence expansion entirely.
- Do not change the output format of expanded CLI fences.
- Do not remove the existing benefit where a pre-task source fence can satisfy the same command in the execute or verify template.
- Do not introduce a new CLI option.

## Root cause

The cache layer in [implementation/src/application/cached-command-executor.ts](c:/Work/md-todo/implementation/src/application/cached-command-executor.ts#L1) has no notion of task position, prompt reachability, or phase boundary. It memoizes at the command-executor level, below the point where source offsets and selected-task boundaries are known.

The boundary is only known in [implementation/src/application/prepare-task-prompts.ts](c:/Work/md-todo/implementation/src/application/prepare-task-prompts.ts#L91), where the full source is expanded and only afterward reduced to `expandedContextBefore`.

So the cache is currently implemented one layer too low to respect "after task" semantics.

## Recommended model

Do **not** solve this by adding `phase` to the cache key.

That would avoid the bug, but it would also throw away the intended optimization where the same pre-task command result is reused across source, execute-template, and verify-template expansion.

Instead, keep a shared cache store keyed by `(command, cwd)`, but make cache **writes** conditional on whether the executed CLI block is inside the reached region.

### Reached-region rule

For source expansion, a CLI block is cache-eligible only when the block occurs fully before the selected task boundary.

One practical rule:

- Compute the selected task start offset in the raw source.
- Parse source CLI blocks with offsets via [implementation/src/domain/cli-block.ts](c:/Work/md-todo/implementation/src/domain/cli-block.ts).
- While expanding source CLI blocks, allow cache population only for blocks whose `endOffset` is before the selected task start offset.

For task-template and verify-template expansion, keep current behavior: all executed CLI blocks are cache-eligible because those phases are reached by definition.

### Why write-gating is the right default

This preserves the useful cases:

- pre-task source -> execute template reuse
- pre-task source -> verify template reuse
- execute template -> verify template reuse

And it fixes the incorrect case:

- after-task source -/-> execute template reuse

It also matches the user-facing intuition behind the existing prompt contract: if a source region has not been reached for the selected task, it should not precompute shared CLI state for later phases.

## Implementation outline

### 1. Replace the opaque executor-local cache with an addressable cache store

[implementation/src/application/cached-command-executor.ts](c:/Work/md-todo/implementation/src/application/cached-command-executor.ts) should stop hiding the cache map entirely inside the decorator.

Recommended shape:

```ts
export interface CachedCommandResultStore {
  get(command: string, cwd: string): CommandResult | null;
  set(command: string, cwd: string, result: CommandResult): void;
}

export function createCachedCommandResultStore(): CachedCommandResultStore;
export function createCachedCommandExecutor(
  delegate: CommandExecutor,
  store?: CachedCommandResultStore,
  options?: { shouldCacheResult?: (params: CacheDecisionParams) => boolean },
): CommandExecutor;
```

The important part is not the exact API. The important part is that source expansion can share the same lookup store as later phases while suppressing writes for after-task blocks.

### 2. Make source expansion aware of the selected-task boundary

[implementation/src/application/prepare-task-prompts.ts](c:/Work/md-todo/implementation/src/application/prepare-task-prompts.ts) already has both:

- the original task metadata (`task.line`, `task.file`)
- the raw source document before expansion

Add a small helper that computes the selected task start offset in the raw source. This can be line-based; it does not need a new parser contract.

Then expand source CLI blocks with a cache policy that only writes for blocks before that offset.

### 3. Thread cache policy through the source expansion path only

There are two reasonable implementation shapes. Either is acceptable; prefer the one that keeps `expandCliBlocksWithOptions` smallest.

Option A:

- Teach `expandCliBlocksWithOptions(...)` an optional `shouldCacheCommandResult(commandOrdinal, block)` callback.
- For source expansion, pass a callback that checks block offsets against the selected task boundary.
- For task-template and verify-template expansion, omit the callback so all writes are allowed.

Option B:

- Keep `expandCliBlocksWithOptions(...)` unchanged.
- Wrap the cached executor passed to source expansion with a source-phase executor that shares cache reads but conditionally suppresses cache writes based on the current block.

Option B keeps the expansion helper simpler. Option A keeps the behavior explicit at the expansion callsite. Either is fine; the migration does not require one exact internal API.

### 4. Preserve current cache semantics for reached template phases

Do not change how task-template and verify-template caching behaves. Those phases are already explicit and reached, and the existing test in [implementation/__tests__/application/run-task.test.ts](c:/Work/md-todo/implementation/__tests__/application/run-task.test.ts#L2226) should keep passing after adaptation.

## Notes

- The simplest incorrect fix would be to key the cache by `(command, cwd, phase)`. That masks the bug but loses the optimization that the cache was built to provide.
- The preferred fix is to keep the key broad and make cache warming more selective.
- This migration intentionally does **not** require changing prompt rendering semantics. It only constrains when a source-phase execution is allowed to become reusable shared cache state.
