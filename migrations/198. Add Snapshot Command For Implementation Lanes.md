# 198. Add Snapshot Command For Implementation Lanes

## Problem

Today rundown has no first-class filesystem snapshot history for the materialized implementation state.

`materialize` is currently a convenience wrapper over `run --all --revertable`, which means the durable reversal/history story is still tied to revertable run artifacts and git-oriented commit metadata rather than to an explicit implementation snapshot model.

That leaves a gap relative to the new prediction snapshot contract:

- prediction can persist lane-aware full-tree snapshots,
- implementation has only the live `implementation/` tree,
- and there is no dedicated CLI surface for saving implementation state at completed migration boundaries.

We want to start migrating away from git-dependent history for this area.

This slice should introduce a direct implementation snapshot system that is filesystem-backed, lane-aware, and numbered from completed migrations rather than from ad hoc user input.

## Desired behavior

Add a top-level CLI command:

- `rundown snapshot`

Its job is to save the current `implementation/` tree into implementation snapshot history.

Canonical layout:

- live implementation head remains `implementation/`
- root-lane snapshots live under `implementation/snapshots/root/<N>/`
- thread-lane snapshots live under `implementation/snapshots/threads/<thread>/<N>/`

Numbering rule:

- snapshot numbers are derived from completed migration boundaries,
- not from user-provided names or arbitrary integers,
- and a snapshot cannot be created between migrations.

There is no `implementation/latest/` directory in this model because `implementation/` itself is already the live head.

## Behavior contract

### 1. `snapshot` is a dedicated implementation-history command

`rundown snapshot` should save the current materialized implementation state as one or more full-tree snapshots under `implementation/snapshots/`.

This command is filesystem-oriented. It is not a worker-run wrapper and it must not require git commit metadata in order to succeed.

### 2. Snapshot numbering comes only from completed migrations

Snapshot names are not free-form.

The command must derive eligible snapshot numbers from migration completion state:

- root snapshots use the highest completed root-lane migration number,
- thread snapshots use the highest completed migration number inside each thread lane,
- and review/supplementary files sharing migration number `N` only count as boundary `N` after that whole lane-number batch is complete.

Users do not choose snapshot numbers manually.

### 3. You cannot snapshot between migrations

The command must reject snapshot creation when the workspace is between migration boundaries.

Interpretation:

- if the live implementation reflects a lane whose newest migration batch is not fully completed, `snapshot` must not write a new snapshot for that state,
- and if the command cannot determine a completed migration boundary for the current materialized state, it must fail with an explicit message rather than inventing a number.

This command exists to record completed migration states, not arbitrary intermediate implementation drift.

### 4. Snapshot paths are lane-aware, but the copied tree is global

Threads already exist as separate migration lanes under `migrations/threads/<thread>`.

That lane structure should be mirrored in implementation snapshot storage:

- `implementation/snapshots/root/<N>/`
- `implementation/snapshots/threads/<thread>/<N>/`

However, the copied filesystem payload is still the full global `implementation/` tree, not a per-thread subtree. Threads are lane labels for historical boundaries, not separate implementation roots.

### 5. `snapshot` must not depend on git support

This migration intentionally drops git as a prerequisite for implementation snapshots.

`snapshot` must not require:

- a git repository,
- a clean git worktree,
- commit creation,
- or `extra.commitSha` from revertable runs.

Git-backed revert/log flows can be migrated separately later. This slice is about introducing a direct filesystem snapshot history.

### 6. Existing snapshots are historical artifacts, not arbitrary overwrite targets

If a target snapshot path for lane boundary `N` already exists, the command should treat that snapshot as already recorded historical state.

Recommended initial behavior:

- do not silently overwrite an existing snapshot for the same lane/number,
- report that the snapshot already exists,
- and leave force-refresh semantics for a later migration if they are needed.

This keeps snapshot numbering tied to historical boundaries rather than turning `snapshot` into a mutable export command.

## Scope

In scope:

- add a top-level `snapshot` CLI command,
- introduce `implementation/snapshots/root/<N>/` and `implementation/snapshots/threads/<thread>/<N>/`,
- define lane-aware snapshot numbering from completed migrations,
- define failure behavior when invoked between migrations,
- and make the command independent from git support.

Out of scope:

- replacing `materialize` with `snapshot`,
- automatically taking implementation snapshots after every `materialize`,
- migrating `revert`, `undo`, or log eligibility away from git/commit metadata,
- supporting ad hoc named snapshots,
- or introducing snapshot diff/browse subcommands.

## Implementation plan

1. Add workspace path helpers for implementation snapshot roots.
2. Add a dedicated application task for saving implementation snapshots.
3. Add a top-level `snapshot` CLI surface and wire it to the new task.
4. Resolve completed migration boundaries for root and thread lanes from migration state.
5. Validate that snapshot creation is happening at completed migration boundaries only.
6. Copy the live `implementation/` tree into the appropriate snapshot directories without recursing into `implementation/snapshots/` itself.
7. Update docs and tests for the new command and numbering rules.

## Acceptance criteria

- `rundown snapshot` exists as a top-level command.
- Running `rundown snapshot` after completed root migration `2` writes `implementation/snapshots/root/2/` as a full copy of the current `implementation/` tree.
- Running `rundown snapshot` after completed thread `checkout` migration `3` writes `implementation/snapshots/threads/checkout/3/` as a full copy of the current `implementation/` tree.
- Snapshot numbering is inferred from completed migration boundaries and cannot be provided manually.
- If the workspace is between migrations, `rundown snapshot` exits with an explicit error and does not create a new snapshot.
- The command works without requiring git commits, git cleanliness, or revertable run metadata.
- Re-running `rundown snapshot` for an already-recorded lane boundary does not silently overwrite the existing snapshot.

## Current implementation context (research baseline)

- `materialize` currently exists as a run-wrapper command in [implementation/docs/cli-materialize.md](implementation/docs/cli-materialize.md) and is explicitly described in terms of revertable defaults rather than filesystem snapshots.
- `predict` now uses `prediction/latest/` plus lane-aware snapshot directories, which provides a concrete pattern for snapshot path helpers and lane naming.
- Thread lanes already exist in migration discovery through `.rundown/threads/*.md` and `migrations/threads/<thread>/`, but implementation state itself is still a single global tree.
- There is no dedicated implementation snapshot command or implementation snapshot directory contract today.

## Edge cases

- If no completed migration exists in any lane, `snapshot` should report that there is no eligible migration boundary to snapshot.
- If one or more lanes are between migration boundaries, the command should fail rather than recording a snapshot for a mixed intermediate state.
- Snapshot copy logic must exclude `implementation/snapshots/**` itself so snapshots do not recursively contain earlier snapshots.
- If a thread brief exists but no completed migration exists yet for that lane, no thread snapshot should be created for that lane.
- If a target snapshot directory already exists, the command should report it and leave it unchanged.
- If the implementation directory is mounted elsewhere via workspace routing, snapshot paths must respect the resolved implementation path rather than reconstructing paths manually.
