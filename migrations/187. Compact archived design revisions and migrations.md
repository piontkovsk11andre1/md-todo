# 187. Compact Archived Design Revisions and Migrations

## Problem

The project now has multiple long-lived history buckets that can grow without a practical bound:

- `design/current/` for the mutable draft,
- `design/revisions/rev.N/` for immutable released revision payloads,
- `migrations/` for root migration history,
- and `migrations/threads/<thread>/` for thread-scoped migration histories.

That growth is expected, but the runtime still treats all historical payloads as permanently hot filesystem state.

- Design revision discovery, diff generation, and migrate planning still assume historical revision payloads live directly on disk in the canonical revisions tree.
- Migration state discovery still reads active directories directly and has no concept of archived root or thread payloads.
- There is no first-class way to compact old history while preserving transparent access for commands that still need to read it.
- If projects keep many revisions and thread lanes, the managed workspace becomes noisy and increasingly expensive to inspect.

The runtime needs a storage lifecycle that distinguishes:

- hot history that should remain directly visible in the primary directories,
- cold history that can be archived,
- and minimal hot metadata that still lets commands resolve and reopen archived payloads transparently.

## Desired behavior

After this migration, rundown should provide an explicit `compact` command that archives cold design revision payloads and migration payloads without breaking normal history reads.

The target product behavior is:

- `rundown compact` performs a dry-run or execute pass over managed history.
- Design revision metadata stays hot and authoritative.
- Old revision payload directories can move out of the primary hot revisions tree into an archive root.
- Old migration payload files can move out of active migration directories into archive roots for both the root lane and thread lanes.
- `rundown migrate` and `rundown design diff` continue to resolve historical content transparently whether the payload is still hot or already archived.
- The first compact slice does not compress payloads with tar/gzip yet; it archives by managed relocation first.

The runtime should prefer transparent archive access over a manual restore workflow.

## Behavior contract

### 1. Introduce an explicit compact lifecycle command

The CLI should expose a dedicated top-level command:

- `rundown compact`

The command should support at least:

- dry-run reporting,
- scoped targets (`revisions`, `migrations`, `all`),
- keep-count style retention controls,
- and human-readable summary output of what would move or did move.

The compact command is responsible for applying archive policy. It should not require users to manually move directories or understand internal archive roots.

### 2. Archive by relocation first, not by compression

The first implementation should treat compaction as logical relocation into managed archive roots, not as tarball creation.

Reasons:

- current readers are path-oriented and expect ordinary files/directories,
- transparent history access is substantially simpler if archived payloads remain ordinary filesystem content,
- and cross-platform behavior on Windows should not depend on external `tar` tooling.

Compression can be introduced later as a second phase on top of the same archive contract if relocation alone does not reduce storage pressure enough.

### 3. Design revision metadata remains hot; only payloads become cold

Design revision metadata is the control plane for revision ordering and planned-state discovery.

That means:

- canonical revision metadata sidecars remain in the hot revisions root,
- `findLowestUnplannedRevision(...)` and revision discovery continue to rely on hot metadata,
- and archived revision payloads must still be resolvable from that metadata record or an archive manifest.

The canonical design layout after compaction should distinguish:

- `design/current/` for the mutable draft,
- `design/revisions/rev.N.meta.json` for hot metadata,
- `design/revisions/rev.N/` for hot payloads that are still retained in place,
- `design/archive/revisions/rev.N/` for archived revision payloads.

Compaction must not strand a revision by moving its payload without leaving a deterministic way to resolve it later.

### 4. Migration history must support hot and archived payload roots

Migration history currently exists in two shapes:

- root lane under `migrations/`,
- thread lanes under `migrations/threads/<thread>/`.

After this migration, both shapes need archive-aware enumeration.

Canonical archived locations should be:

- `migrations/archive/root/` for archived root-lane migration payloads,
- `migrations/archive/threads/<thread>/` for archived thread-lane payloads.

The migration reader should build state from:

- hot root files in `migrations/`,
- hot thread files in `migrations/threads/<thread>/`,
- archived root files in `migrations/archive/root/`,
- archived thread files in `migrations/archive/threads/<thread>/`.

The numbering and review/supplementary file semantics must remain unchanged. Archive location changes storage temperature, not migration identity.

### 5. Archive-aware reads must be transparent for current history consumers

The first commands that must remain archive-transparent are:

- `rundown migrate` when it reads released revision history and migration history,
- `rundown design diff` when it resolves `--from` and target revisions,
- and any shared revision-diff helpers those commands already consume.

Transparent means:

- no extra restore command is required,
- existing revision references such as `rev.3` remain valid,
- and migrated/archived payloads still produce the same prompt context and diff content.

### 6. Archive eligibility is policy-driven and count-based first

The first retention policy should be count-based rather than age-based.

Recommended behavior:

- keep the newest `N` revision payloads hot and archive older planned revision payloads,
- keep the newest `N` root migration numbers hot and archive older root payloads,
- keep the newest `N` migration numbers hot per thread lane and archive older thread payloads.

Age-based filters can be added later, but the initial contract should optimize for predictability and easier tests.

### 7. Compaction writes a manifest for deterministic lookup and reporting

The runtime should persist an archive manifest under the managed config space, for example:

- `.rundown/archive-index.json`

The manifest should at minimum record:

- content kind (`revision-payload`, `migration-primary`, `migration-review`, future `migration-thread-*`),
- original logical path,
- archive path,
- archived timestamp,
- and any lane identity such as root vs thread slug.

The manifest is not a replacement for filesystem discovery, but it gives the runtime one stable source of archive bookkeeping and user-visible compact reporting.

### 8. This migration does not compact `.rundown/runs/`

Finished run artifacts remain out of scope for this first compact slice.

That is required to avoid regressions in finished-run `discuss`, which still scans `.rundown/runs/<run-id>/` directly for `run.json` and phase directories.

If run artifact compaction is added later, it should happen in a separate migration after discuss gains an archive-aware artifact reader.

## Scope

In scope:

- Add the explicit `rundown compact` command.
- Define canonical archive roots for design revisions and migration payloads.
- Keep design revision metadata hot while archiving old revision payload directories.
- Make migration state discovery enumerate hot and archived root/thread lanes.
- Make design diff and migrate resolve archived revision payloads transparently.
- Add an archive manifest and dry-run reporting.
- Add focused regression coverage for archive-aware reads and compact execution.

Out of scope:

- Compressing archived payloads as tarballs or gzip streams.
- Compacting `.rundown/runs/` or other run artifacts.
- Introducing a manual restore-only workflow.
- Changing revision naming (`rev.N`) or migration naming (`N. Title.md`).
- Changing thread semantics beyond making archived thread histories readable.
