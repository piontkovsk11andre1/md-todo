# 157. Make root rundown open the TUI

## Problem

After 156, the agent flow is reachable as `rundown agent`. We can now repurpose the no-args entry point. Today the root `.action(...)` in [src/presentation/cli.ts](implementation/src/presentation/cli.ts) calls `createHelpCommandAction`, which opens the agent on a TTY and prints help otherwise. We want the no-args entry to open the new TUI scaffold (created in 155) instead.

Backward compatibility is critical: scripts that pipe `rundown` or invoke it inside CI must keep getting help-style output, exit code 0. Only true interactive terminals get the TUI.

## Goal

Wire the no-args `rundown` invocation to the TUI scaffold from 155, while preserving non-TTY behavior.

## Model

Add `createRootTuiAction` next to `createHelpCommandAction` in [src/presentation/cli-command-actions.ts](implementation/src/presentation/cli-command-actions.ts):

```ts
export function createRootTuiAction({
  getApp,
  getWorkerFromSeparator,
  outputHelp,
  cliVersion,
  isInteractiveTerminal,
  getInvocationArgv,
}: HelpActionDependencies): () => CliActionResult {
  return async () => {
    const argv = getInvocationArgv?.() ?? process.argv.slice(2);
    if (resolveInvocationCommand(argv) !== "rundown") {
      outputHelp();
      return EXIT_CODE_SUCCESS;
    }
    if (hasCliFlag(argv, "--agents")) {
      process.stdout.write(getAgentsTemplate());
      return EXIT_CODE_SUCCESS;
    }
    if (!(isInteractiveTerminal ?? defaultIsInteractiveTerminal)()) {
      outputHelp();
      return EXIT_CODE_SUCCESS;
    }
    return runRootTui({
      app: getApp(),
      workerPattern: resolveWorkerPattern(undefined, getWorkerFromSeparator),
      cliVersion,
      argv,
    });
  };
}
```

`runRootTui` is the entry exported from `src/presentation/tui/index.js` after 155. It boots the existing menu (still materialize-only at this stage; the new menu lands in 158). It returns an exit code, defaulting to `EXIT_CODE_SUCCESS` on clean quit.

Replace the root `.action(...)` registration in [src/presentation/cli.ts](implementation/src/presentation/cli.ts):

```ts
program
  .name("rundown")
  // …
  .action(withCliAction(createRootTuiAction({ ... })));
```

`-c, --continue` and `--agents` stay declared on the root for backward compatibility. `-c` becomes a no-op for the TUI path in this migration (the TUI does not yet have a "resume" affordance); we keep the flag accepted so old scripts do not error. A follow-up migration can wire `-c` into the TUI's Continue scene if desired.

## Implementation boundaries

Files expected to change:

- `implementation/src/presentation/cli-command-actions.ts` — add `createRootTuiAction`.
- `implementation/src/presentation/cli.ts` — swap root `.action(...)`.
- `implementation/src/presentation/tui/index.js` — export `runRootTui({ app, workerPattern, cliVersion, argv })` adapter that wraps the existing Commander-driven entry. Goal is in-process invocation (no `spawn` child).
- `implementation/docs/cli-help.md` (or root README usage section) — update the description of bare `rundown`. Mention `rundown agent` for users who want the previous behavior.
- `implementation/__tests__/presentation/...` — tests:
  - non-TTY `rundown` returns help and exit 0.
  - `rundown --agents` (still) prints canonical agents and exit 0.
  - TTY `rundown` invokes `runRootTui` exactly once with the constructed dependencies (mock `runRootTui`).

## Compatibility and constraints

- Exit code on clean Quit must be 0.
- Exit code on Ctrl-C must be 130 (or whatever rundown uses today for SIGINT in interactive flows).
- `process.stdin.isTTY === false` paths must never enter the TUI.
- The existing `--config-dir`, `--trace` global options continue to work.

## Acceptance

- `rundown` in an interactive terminal opens the TUI.
- `rundown` piped (`rundown | cat`) prints help and exits 0.
- `rundown --agents` still prints canonical AGENTS.md.
- `rundown agent` continues to behave like the previous `rundown` no-args (verified in 156).
