# 160. Continue scene wires materialize cockpit

## Problem

After 158 the materialize cockpit is reachable from `Continue`, but the scene is still the original sketch with prototype-only behaviors:

- form-style input that mixes "select source" and "confirm" in a single screen,
- task-count estimation only on confirm, not on entry,
- no live phase badge tied to the new structured progress events ([verify-repair-loop.ts](implementation/src/application/verify-repair-loop.ts), [plan-task.ts](implementation/src/application/plan-task.ts), [research-task.ts](implementation/src/application/research-task.ts), [discuss-task.ts](implementation/src/application/discuss-task.ts)),
- no preview of the next task before the user commits to running.

This migration polishes the cockpit so it feels like a deliberate scene, not a prototype.

## Goal

Refine the Continue scene to:

1. Show the next unchecked task from the configured source the moment the scene opens, before any user input.
2. Display the task count and a 3-line preview (next, after-next, after-after-next).
3. On Enter, transition into the live run view that consumes `progress`, `group-start`, `group-end` events and renders the operation badge in real time.
4. Provide hotkeys: `Space` pause/resume hint (no-op placeholder is fine for this migration; design follow-up will wire actual lock release), `s` stop (sends graceful shutdown, calls `app.releaseAllLocks` if needed), `Esc` back to main menu after the run completes.

## Model

Scene states:

- `previewing`: shows next-N tasks, count, source path, profile/worker that will run.
- `running`: shows live operation badge, current task, completed/total, recent messages tape (max 6).
- `done`: shows summary — counts, duration, failures, repairs, resolves — and waits for Esc.

Phase badge mapping (already wired via `progress` events):

```
scan       cyan
execute    yellow
verify     blue
repair     red
resolve    magenta
resolveRepair magenta
plan       cyan
research   cyan
discuss    yellow
finalize   green
```

Right-side mini-panel during `running` shows phase counters parsed from `progress.current` / `progress.total` (`attempt 2/3`, `scan 1/4`, etc.) when the event payload includes them.

## Implementation boundaries

Files expected to change:

- `implementation/src/presentation/tui/scenes/continue.js` — refactor.
- `implementation/src/presentation/tui/output-bridge.js` — already consumes `progress`, just confirm `resolveRepair`/`plan`/`research`/`discuss` paint the badge.
- `implementation/src/presentation/tui/components/badge.js` — extract status badge if reuse is needed.
- `implementation/__tests__/presentation/tui/...` — golden test that feeds a recorded sequence of `ApplicationOutputEvent` values and asserts the rendered frame at key moments. Using a snapshot is acceptable.

Out of scope:

- Real pause semantics. `Space` shows a tooltip "Pause not yet supported" and continues rendering.
- Multi-source selection UX. Current source resolution stays as-is.

## Acceptance

- Opening Continue immediately shows the next task and the count, with no extra keystroke.
- Starting a run shows the operation badge changing across phases without flicker.
- A failing run lands in `done` showing the failure reason and a non-zero indicator; pressing Esc returns to main menu.
- The Continue right-column status probe (159) reflects the post-run task count when re-entering the menu.
