# 163. Profiles scene

## Problem

Profiles in rundown are **named full worker command arrays** under `profiles.<name>` in `.rundown/config.json`. Per [docs/configuration.md](implementation/docs/configuration.md), they are selected from one of three places in Markdown:

- file frontmatter `profile: <name>`,
- a parent directive list item `- profile=<name>`,
- a `profile=<name>` prefix modifier on a checkbox task.

Users today have no quick way to:

- list what profiles are defined in the merged config,
- see *where* in the workspace each profile is referenced,
- change a profile without hand-editing JSON.

Worse, profiles are easy to misunderstand as "the active worker preset". They are not. There is no global "set this profile as default" toggle in the schema — the default worker is `workers.default`, and a profile is only active when invoked from a Markdown construct.

## Goal

Build a Profiles scene that lists `profiles.<name>` entries, shows workspace usage attribution per profile, and offers to edit `.rundown/config.json` through `$EDITOR` (same approach as 162 — no bespoke form, no schema duplication).

The scene must NOT offer a "set as default" action, because that concept does not exist in the schema. Teaching users a wrong mental model is worse than the missing convenience.

## Model

Scene layout:

```
┌─ Profiles ──────────────────────────────── .rundown/config.json ─┐
│                                                                  │
│  complex   opencode run --file $file $bootstrap --model opus-4.6 │
│            used by: 0 frontmatter · 2 directives · 0 prefix      │
│                                                                  │
│  fast      opencode run --file $file $bootstrap --model gpt-5.3  │
│            used by: 1 frontmatter · 4 directives · 0 prefix      │
│                                                                  │
│  opus      claude -p --model opus                                │
│            used by: (no references found)                        │
│                                                                  │
│  sonnet    claude -p --model sonnet                              │
│            used by: (no references found)                        │
│                                                                  │
│  [↵] inspect references   [e] edit config.json   [u] full scan   │
└──────────────────────────────────────────────────────────────────┘
```

Workspace usage scan:

- "frontmatter" matches lines with `^profile:\s*<name>` inside YAML frontmatter blocks (`---` … `---`) of any markdown file in the workspace.
- "directive" matches plain list items `- profile=<name>` (case-sensitive, exact match on the profile name).
- "prefix" matches `profile=<name>` inside checkbox text, respecting the canonical prefix-chain separators `, ` / `; ` documented in [docs/configuration.md](implementation/docs/configuration.md).
- The scan respects the workspace root and the same ignore patterns rundown already uses for source resolution. Do not invent a new ignore policy.
- The first scan happens lazily on scene entry and caches results for 30 seconds. `[u]` forces a full re-scan.

Inspect references (`[↵]` on a profile row):

- Drills into a list of `<file>:<line>` references with kind tags (`frontmatter` / `directive` / `prefix`).
- Pressing `Enter` on a reference opens that file in `$EDITOR` at the reported line if `$EDITOR` supports `+N` (vim/nvim) or `:N` (`code -g <file>:<line>`) syntax; otherwise opens the file at the top.
- `Esc` returns to the profile list.

`[e]` opens `<config-dir>/config.json` in `$EDITOR`, identical to 162. After the editor exits, the scene reloads the config and re-renders. JSON validation errors surface in a banner above the list.

Empty / edge cases:

- `profiles` absent or `{}` → render a single line "No profiles defined." plus the `[e]` action and a one-line hint pointing at the docs section "Profiles" in [docs/configuration.md](implementation/docs/configuration.md).
- Profile defined but referenced nowhere → the row clearly shows `used by: (no references found)`.
- Profile referenced in Markdown but missing from `profiles.<name>` → render a separate "Unknown references" group at the bottom listing `<name> · <count>` with the file/line list available via `Enter`. Rationale: this is the same diagnostic users get from rundown ("Unknown profile name") but at workspace scale.

## Implementation boundaries

Files expected to change:

- new `implementation/src/presentation/tui/scenes/profiles.js`.
- new `implementation/src/presentation/tui/bridges/workspace-scan.js` — frontmatter + directive + prefix scan, returns `{ profile: string, file: string, line: number, kind: "frontmatter" | "directive" | "prefix" }[]`.
- reuse `editor-launch.js` from 162.
- reuse `config-bridge.js` from 162 for the profile list and reload-after-edit flow.

Out of scope:

- Setting a profile as default. There is no such concept in the schema; profiles are invoked from Markdown, not toggled globally. The scene must NOT offer this action.
- Pass-rate / verification stats per profile. Useful, but requires reading trace artifacts and is deferred to a later migration.
- Creating a new profile from an interactive form. Use `[e]` and the JSON editor.
- Renaming a profile across all workspace references. Out of scope; would require a multi-file refactor that belongs in its own migration.

## Acceptance

- Profiles scene lists every `profiles.<name>` from the loaded effective config.
- Each row shows accurate usage counts from the workspace scan against a fixture workspace seeded by the test harness (170).
- Drilling into a row lists individual `<file>:<line>` references with kind tags.
- `[e]` opens `<config-dir>/config.json` in `$EDITOR`, blocks while editing, and reloads on return.
- An invalid JSON edit surfaces the validator error in a banner; the scene does not crash.
- A workspace with no `profiles` section shows the "No profiles defined." hint and the `[e]` action.
- A profile referenced nowhere clearly shows `used by: (no references found)`.
- A reference to an unknown profile name is grouped under "Unknown references" with the file/line attribution.
- The scene does not expose any "set default" action.
