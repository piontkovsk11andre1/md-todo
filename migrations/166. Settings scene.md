# 166. Settings scene

## Problem

Per [docs/cli-config.md](implementation/docs/cli-config.md) and [docs/configuration.md](implementation/docs/configuration.md), rundown layers config across `built-in defaults → global → local → CLI overrides`, and `rundown config list --scope effective --show-source` reports per-key provenance. This is the answer to the question users actually ask when something behaves unexpectedly: *"where is this value coming from?"*

The TUI today has no equivalent. Workers (162) and Profiles (163) cover narrow slices, but the merged config view with provenance is not exposed.

## Goal

Build a Settings scene that wraps `rundown config list --scope <scope> --show-source --json` for `effective`, `local`, and `global` scopes. Read-first. Edits go through `$EDITOR` opening the underlying file (delegates to `editor-launch.js` from 162).

This scene must NOT reinvent JSON validation, schema awareness, or merge semantics — it only renders what `rundown config` returns.

## Model

Scene layout:

```
┌─ Settings ─ scope: effective ─ provenance shown ────────────────────┐
│                                                                     │
│  workers.default          ["opencode","run","--file","$file",…]     │
│                           ◀ local                                   │
│  workers.tui              ["opencode"]                       ◀ local│
│  workers.fallbacks        [["claude","-p"], ["codex",…]]     ◀ local│
│  workerTimeoutMs          120000                             ◀ local│
│                                                                     │
│  commands.plan            ["opencode",…,"--model","opus"]    ◀ local│
│  commands.verify          ["opencode",…,"--model","gpt-5.3"] ◀ local│
│                                                                     │
│  healthPolicy.unavailableReevaluation.mode      "cooldown" ◀ built-in│
│  healthPolicy.unavailableReevaluation.probeCo…  300        ◀ built-in│
│                                                                     │
│  traceStatistics.enabled  true                              ◀ local │
│  traceStatistics.fields   ["total_time","tokens_estimated"] ◀ local │
│                                                                     │
│  [s] scope ▸ effective | local | global                             │
│  [e] edit current scope file in $EDITOR                             │
│  [o] reveal config file path                                        │
│  [↵] inspect value (full text + raw JSON)                           │
└─────────────────────────────────────────────────────────────────────┘
```

Behavior:

- On entry, default scope is `effective`; provenance is always shown when scope is `effective`. For `local`/`global`, provenance is hidden because every key in those scopes has trivially one source.
- `[s]` cycles the scope. The header reflects the active scope.
- `[e]` opens the file for the current scope:
  - `local` → `<config-dir>/config.json`
  - `global` → path returned by `rundown config path --scope global`. If the global file does not exist, prompt to create an empty `{}` first.
  - `effective` → disabled with a tooltip "effective is a merged read-only view; switch to local or global to edit."
- `[o]` prints the resolved path in a banner without opening anything (useful when the user wants to grep or open from another tool).
- `[↵]` opens a single-key inspector overlay showing the full key path, the value formatted as pretty JSON, and the source attribution (when scope is `effective`). Esc closes the overlay.

Long values (any array of length > N or string longer than the row width) render as truncated single line in the list view; the inspector shows them in full.

Filter/search is **out of scope** for this migration. Cycling scopes is enough for v1.

## Implementation boundaries

Files expected to change:

- new `implementation/src/presentation/tui/scenes/settings.js`.
- extend `implementation/src/presentation/tui/bridges/config-bridge.js` (created in 162) with `listEffective`, `listLocal`, `listGlobal`, and `getGlobalPath` operations. These wrap the same code paths as `rundown config list` and `rundown config path`. Prefer in-process App API where available; otherwise call the loader in [src/infrastructure/adapters/worker-config-adapter.ts](implementation/src/infrastructure/adapters/worker-config-adapter.ts) and the global-path resolver under [src/infrastructure/adapters/](implementation/src/infrastructure/adapters/).
- reuse `editor-launch.js` from 162.

Out of scope:

- Editing values from the TUI through `rundown config set`. Editor-on-file is the only write path in this migration.
- Search/filter UX.
- Diffing local against global side by side.

## Acceptance

- Opening Settings on a fresh workspace with empty `{}` local and no global file shows only built-in defaults under scope `effective`, all attributed `◀ built-in`.
- `[s]` cycles `effective → local → global → effective`.
- Switching to scope `local` and pressing `[e]` opens `<config-dir>/config.json` in `$EDITOR`; on return the values reload.
- Switching to scope `global` when no global file exists prompts to create one, then opens it.
- `[↵]` shows the full value of a long array.
- Provenance markers match what `rundown config list --scope effective --show-source` would produce for the same workspace.
