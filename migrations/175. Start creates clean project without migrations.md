# 175. Start Creates Clean Project Without Migrations

## Problem

`rundown start` currently carries bootstrap behavior that tries to infer too much from the target directory.

- In some flows it seeds an initial migration.
- In non-empty directories it switches to a research-oriented migration template.
- In other flows it can leave only the `.rundown/` scaffold behind.

That makes Start inconsistent and harder to reason about. The command should not inspect whether the directory is empty or non-empty and then change the meaning of initialization. Start should do one thing: create a clean rundown project surface.

## Desired behavior

After this migration, `rundown start` initializes the workspace the same way regardless of whether the target directory is empty or already contains files.

- Create the rundown scaffold and configured directories.
- Do not create any migration files.
- Do not seed research tasks.
- Do not derive different templates from directory contents.
- If design is not predefined externally, create `design/current/Target.md` as an empty file.

The purpose of Start becomes simple and deterministic: establish the project structure, not author the first change.

## Behavior contract

### 1. Directory contents do not change Start semantics

Whether the target directory is empty or already contains project files must not affect Start behavior.

- No "existing project" migration seed.
- No research-first onboarding template.
- No automatic revision-0 planning document.

Existing non-rundown files in the target directory remain untouched.

### 2. Start creates structure, not migrations

Start should initialize the rundown workspace buckets and config, but it must not create an initial migration document.

Expected outputs from a normal local Start include:

- `.rundown/config.json`
- `design/current/`
- `design/current/Target.md`
- `migrations/` directory present but empty
- `specs/`
- `prediction/`
- any other standard scaffold files that still belong to the clean-project baseline

The key rule is that migrations begin only when the user explicitly creates or materializes one later. Start does not pre-author migration `1`.

### 3. Empty local design file by default

If the design target is managed locally by rundown, Start must create `design/current/Target.md` as an empty file.

- No starter prose.
- No placeholder description.
- No copied project summary.

If design is predefined externally, Start should respect that and not create a local replacement file.

### 4. Idempotent clean initialization

Re-running Start must remain safe.

- Do not overwrite an existing `Target.md`.
- Do not create duplicate migration seeds.
- Do not rewrite user-authored design content.

## Scope

In scope:

- Simplify Start to one deterministic clean-project initialization path.
- Remove automatic initial-migration creation from Start.
- Remove directory-content-based branching for Start seeding behavior.
- Ensure local design initialization uses an empty file when no external design source is provided.
- Add tests for empty and non-empty target directories proving Start behaves the same.

Out of scope:

- Changing how later commands create or materialize migrations.
- Changing external design resolution when `--from-design` is used.
- Generating any starter design text.

## Implementation boundaries

Primary files likely involved:

- `implementation/src/application/start-project.ts`

The root fix should be in the owning Start implementation, not in UI messaging.
