with opencode should confirm before overwriting existing workers

## Intent

Document a safety guard for `rundown with opencode`: if worker configuration already exists in local config, the command should ask for confirmation before mutating it.

This is a behavior/spec migration only. It does not change runtime code by itself.

## Problem statement

`rundown with opencode` is currently treated as a one-step onboarding command, but that becomes unsafe once a workspace already has configured workers. Re-applying the onboarding flow can overwrite or reshape existing worker commands without an explicit user decision.

The expected behavior is to prompt before changing worker-related config when the workspace already contains worker settings.

## Desired behavior

- When `rundown with opencode` detects existing local worker config, it should ask before making changes.
- The confirmation should happen before any config mutation is written.
- If the user declines, the command should exit without changing config.
- If the user accepts, the command may proceed with the normal preset application flow.
- The guard should apply when any worker-related keys already exist locally, including:
  - `workers.default`
  - `workers.tui`
  - `workers.fallbacks`
- The prompt should clearly state that existing worker config will be replaced or updated.

## Scope

In scope:
- define the confirmation requirement for `rundown with opencode`
- clarify what counts as existing worker config
- require no-op behavior when the user declines

Out of scope:
- changing the actual opencode preset payload
- redesigning the broader `with <harness>` onboarding UX unless needed for consistency
- changing unrelated config sections outside worker keys

## Behavioral notes

- The check should be based on local config that `with` is about to mutate, not merely effective inherited values from global config.
- The confirmation should be skipped when there are no existing local worker keys to overwrite.
- Idempotent no-op application may still avoid prompting if the existing local worker config already matches the intended target exactly.
- If the command is running in a non-interactive context, it should fail with an actionable message instead of silently overwriting worker config.

## Acceptance criteria

1. If local config already contains worker keys, `rundown with opencode` asks for confirmation before writing changes.
2. If the user declines, no worker config is modified.
3. If the user accepts, the normal `with opencode` flow continues.
4. If local worker config is absent, the command proceeds without an overwrite confirmation.
5. If the target worker config is already present exactly, the command may report no-op semantics without prompting.
6. Non-interactive execution does not silently overwrite existing worker config.
7. Tests and docs describe the same overwrite-confirmation contract.

## Implementation notes

- Treat this as an overwrite guard around worker-key mutation planning rather than a late warning after config has already changed.
- Keep the prompt narrow and explicit: the user should know the command is about to replace/update existing worker configuration.
- The result path for a declined confirmation should be a clean cancel/no-op outcome, not a partial mutation.
