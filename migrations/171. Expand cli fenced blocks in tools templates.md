# 171. Expand cli fenced blocks in tools templates

## Problem

Every prompt-producing pipeline in rundown expands ` ```cli ` fenced blocks before the prompt reaches the worker:

- task source + task template + verify template via [src/application/prepare-task-prompts.ts](implementation/src/application/prepare-task-prompts.ts#L99)
- repair template via [src/infrastructure/repair.ts](implementation/src/infrastructure/repair.ts#L176)
- verification prompt via [src/infrastructure/verification.ts](implementation/src/infrastructure/verification.ts#L292)
- plan, help, discuss, research via their `*-task.ts` callers (e.g. [help-task.ts](implementation/src/application/help-task.ts#L178))

The one exception is `.rundown/tools/<name>.md` template-tool expansion. [src/domain/builtin-tools/template-tool.ts](implementation/src/domain/builtin-tools/template-tool.ts) renders the template with `renderTemplate(...)` and hands the result straight to `workerExecutor.runWorker(...)`. There is no `expandCliBlocks` step, no `--ignore-cli-block` honouring, no `--cli-block-timeout`, no trace hook, and no artifact phase label.

That means a tool author who writes a ` ```cli ` block inside `tools/post-on-gitea.md` (for example, to inject the current `gh` repo state into the prompt) sees the literal fence reach the worker — inconsistent with every other template surface.

## Goal

Make tool template rendering expand ` ```cli ` fenced blocks the same way task and verify templates do, with the same options, trace, artifact, and failure-policy contract.

After this migration:

- A ` ```cli ` block in `.rundown/tools/<name>.md` is executed before the worker is invoked, and its XML output replaces the fence in the rendered prompt.
- `--ignore-cli-block` leaves tool-template fences raw (passed through to the worker verbatim).
- `--cli-block-timeout <ms>` applies to tool-template commands.
- With `--keep-artifacts`, tool-template CLI block stdout/stderr are written under a new artifact phase label `cli-tool-template` with prompt type `tool-template`.
- Trace records tool-template CLI block executions like other phases.
- A non-zero CLI block exit (or executor error) aborts the tool with a clear failure message and a non-zero exit code, mirroring `withTemplateCliFailureAbort` for task templates. (Author-controlled template, so abort — not warn — is the right default.)

## Non-goal

- No change to the syntax or semantics of `expandCliBlocks` itself.
- No change to non-template tool handlers (`memory`, `question`, `get`, `for`, `end`, `terminal`, `include`). They do not render Markdown templates.
- No new CLI flag. The existing `--ignore-cli-block` and `--cli-block-timeout` are reused.
- No change to how `tools/` files are discovered or loaded.

## Model

### Plumb CLI-block context into `ToolHandlerContext`

`ToolHandlerContext` at [src/domain/ports/tool-handler-port.ts](implementation/src/domain/ports/tool-handler-port.ts#L72) gains optional fields used only by template tools (other handlers ignore them):

```ts
export interface ToolHandlerContext {
  // ...existing fields...
  cliBlockExecutor?: CommandExecutor;
  cliExpansionEnabled?: boolean;            // false when --ignore-cli-block
  cliExecutionOptions?: CommandExecutionOptions;
  traceWriter?: TraceWriterPort;
  cliTraceRunId?: string;
  nowIso?: () => string;
}
```

These are populated in [src/application/task-execution-dispatch.ts](implementation/src/application/task-execution-dispatch.ts#L265) where `toolContext` is built.

Scope/plumbing confirmation for that build site:

- In scope there already: `ignoreCliBlock` (dispatch param), `artifactContext` (dispatch param, includes `runId`), `dependencies.traceWriter`, and `dependencies.cliBlockExecutor`.
- **Not** currently in scope there: generic `cliExecutionOptions` and `nowIso`.
- Entry points for missing values (no dependency-shape restructure required):
  - `cliExecutionOptions` is created in [src/application/run-task-execution.ts](implementation/src/application/run-task-execution.ts#L727), then passed through `runTaskIteration(...).prompts.cliExecutionOptions` in [src/application/run-task-iteration.ts](implementation/src/application/run-task-iteration.ts#L689), and can be forwarded into `dispatchTaskExecution(...)` as a new param.
  - `nowIso` is created in [src/application/run-task-execution.ts](implementation/src/application/run-task-execution.ts#L888), then passed through `runTaskIteration(...).prompts.nowIso` in [src/application/run-task-iteration.ts](implementation/src/application/run-task-iteration.ts#L693), and can be forwarded into `dispatchTaskExecution(...)` as a new param.
  - `cliBlockExecutor` and `traceWriter` continue to come from `RunTaskDependencies` via `dependencies`.

### Lift `expandCliBlocksWithOptions` out of `prepare-task-prompts.ts`

The helper at [prepare-task-prompts.ts](implementation/src/application/prepare-task-prompts.ts#L44-L116) is exactly what we need (artifact context, trace, failure-policy hook). Move it to a new shared module `src/application/cli-block-expansion.ts` and import it from both `prepare-task-prompts.ts` and the tool handler. No behavior change for existing callers.

Add a new artifact phase label / prompt type pair to its union:

```ts
artifactPhaseLabel: "cli-source" | "cli-task-template" | "cli-verify-template" | "cli-tool-template";
artifactPromptType: "source" | "task-template" | "verify-template" | "tool-template";
```

### Update `createTemplateToolHandler`

In [template-tool.ts](implementation/src/domain/builtin-tools/template-tool.ts#L12), after `renderTemplate(...)` and before `runWorker(...)`:

```ts
const renderedPrompt = renderTemplate(template, vars);

const expansion = await expandCliBlocksWithOptions({
  content: renderedPrompt,
  cliExpansionEnabled: context.cliExpansionEnabled ?? true,
  cliBlockExecutor: context.cliBlockExecutor ?? createCliBlockExecutor(),
  cwd: context.cwd,
  baseCliExpansionOptions: context.cliExecutionOptions,
  artifactContext: context.artifactContext,
  traceWriter: context.traceWriter,
  cliTraceRunId: context.cliTraceRunId,
  nowIso: context.nowIso ?? (() => new Date().toISOString()),
  artifactPhaseLabel: "cli-tool-template",
  artifactPromptType: "tool-template",
  wrapExecutionOptions: withTemplateCliFailureAbort,
  onCliExpansionFailure: async (error) => {
    return mapTemplateCliFailureToExitCode(error); // shared helper, same contract as task templates
  },
});

if ("earlyExitCode" in expansion) {
  return {
    exitCode: expansion.earlyExitCode,
    failureMessage: "Tool template CLI block expansion failed.",
    failureReason: "CLI fenced block in tool template exited non-zero.",
  };
}

const expandedPrompt = expansion.expandedContent;

const runResult = await context.workerExecutor.runWorker({
  // ...
  prompt: expandedPrompt,
  // ...
});
```

The handler stays a `ToolHandlerFn`, no signature change.

### Defaults for handlers that do not opt in

When `context.cliBlockExecutor` is missing (older test stubs, or non-template tools), behave as today: skip expansion. This keeps `__tests__/domain/builtin-tools/*.test.ts` green without forcing every test to wire a CLI executor.

## Implementation boundaries

Files expected to change:

- `implementation/src/domain/ports/tool-handler-port.ts` — add the optional fields described above.
- `implementation/src/application/cli-block-expansion.ts` (new) — extracted `expandCliBlocksWithOptions` plus its result type.
- `implementation/src/application/prepare-task-prompts.ts` — import from the new module; delete the local copy.
- `implementation/src/domain/builtin-tools/template-tool.ts` — render → expand → run.
- `implementation/src/application/task-execution-dispatch.ts` — populate the new context fields when building `toolContext`.
- `implementation/src/application/cli-block-handlers.ts` — export the abort wrapper / failure-mapping helper if not already exported.
- `implementation/__tests__/domain/builtin-tools/template-tool.test.ts` (new) — unit test covering: cli block expanded, `cliExpansionEnabled=false` leaves it raw, executor failure surfaces as non-zero exit code with failure message.
- `implementation/__tests__/integration/cli.test.ts` — add three tests:
  1. `run` against a task whose tool template (`tools/post.md`) contains a ` ```cli ` block — assert worker prompt contains the expanded XML, not the raw fence.
  2. Same workspace + `--ignore-cli-block` — assert the worker prompt still contains ` ```cli `.
  3. `--keep-artifacts` writes a stdout/stderr pair under the `cli-tool-template` phase.
- `implementation/docs/templates.md` — note that tool templates participate in CLI block expansion.

Out of scope:

- `tools/*.js` JavaScript tool handlers (they do not go through `createTemplateToolHandler`).
- Caching of tool-template CLI block output across runs.

## Acceptance

- `npx vitest run __tests__/domain/builtin-tools/template-tool.test.ts __tests__/integration/cli.test.ts` passes.
- A tool template containing a single ` ```cli ` block with `echo hello` produces a worker prompt that contains the canonical `<command>echo hello</command>...<stdout>hello</stdout>...` XML and does not contain the literal substring `\`\`\`cli`.
- The same template run with `--ignore-cli-block` produces a worker prompt that **does** contain `\`\`\`cli` and does not invoke the CLI block executor.
- A failing CLI block (`exit 1`) inside a tool template causes the run to exit non-zero with a failure message that mentions tool-template CLI expansion, and does not invoke the worker.
- Existing `__tests__/domain/task-intent.test.ts`, `__tests__/domain/builtin-tools/*.test.ts`, `__tests__/application/run-task-iteration-modules.test.ts`, and the full integration suite still pass without modification beyond imports.
