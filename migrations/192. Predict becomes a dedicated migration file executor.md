# 192. Predict Becomes A Dedicated Migration File Executor

## Problem

Prediction is no longer well-described as a generic "run migrations forward" shortcut.

The intended behavior is more specific:

- prediction consumes migration files,
- the worker sees the whole migration file,
- the worker updates `prediction/` to reflect that migration,
- and verification/repair run at whole-file prediction scope rather than at checkbox task scope.

That means prediction deserves its own execution contract instead of remaining implicit inside old `migrate up` language.

At the same time, tests need to say clearly whether they target live implementation state or predicted future state.

## Desired behavior

Recommended public surface:

- `rundown predict`
- `rundown test now`
- `rundown test future`

Interpretation:

- `predict` executes migration files into `prediction/`,
- `test now` validates against implementation/current state,
- `test future` validates against prediction state.

This makes the producer/consumer relationship explicit:

- `migrate` creates migrations,
- `predict` applies them to prediction,
- `materialize` applies resulting state to implementation,
- `test now` and `test future` choose which state they validate.

## Behavior contract

### 1. `predict` is incremental and migration-file-driven

Bare `rundown predict` should:

1. discover migration files in order,
2. determine which migrations have already been predicted,
3. process each remaining migration file in sequence,
4. update `prediction/` from that migration file,
5. and persist successful progress after each verified migration application.

It should not create migrations and should not own design revision release.

### 2. Prediction execution uses the whole migration file as the unit of work

The worker prompt for prediction should be built around the migration file, not around checklist items inside it.

For each unapplied migration:

- pass the whole migration file to the worker,
- instruct the worker to modify `prediction/` so it reflects that migration,
- verify the resulting prediction tree or affected files,
- and repair/re-verify if needed at that same migration-pass scope.

This keeps prediction distinct from `run`.

### 3. Prediction progress is explicitly tracked

Prediction must remember which migrations have been successfully applied to `prediction/`.

The implementation should use dedicated prediction progress state rather than overloading unrelated metadata.

That state should at minimum support:

- last valid predicted migration,
- per-migration predicted status,
- migration content hash or equivalent invalidation input,
- and enough metadata to detect stale prediction progress.

### 4. `test` becomes explicit about target state

`test` should stop encoding target choice through implicit default versus old flag language.

Recommended public surface:

- `rundown test now`
- `rundown test future`
- `rundown test new <assertion>`

This keeps `test` aligned with the new product vocabulary:

- `now` means implementation/current state,
- `future` means prediction state.

### 5. `predict` does not absorb rollback concerns in this migration

Rollback is not part of the primary command model here.

This migration is only about making prediction a first-class forward executor and making test targets explicit.

If prediction rollback or rebuild controls are needed later, they should be introduced separately without muddying the main command-family language.

## Scope

In scope:

- define `predict` as a dedicated migration-file-driven prediction executor,
- define prediction progress tracking requirements,
- define `test now` and `test future` as explicit target actions,
- and update docs/help around the new prediction model.

Out of scope:

- migrate source-reconciliation modes,
- start/bootstrap redesign,
- or rollback/rewind command design.

## Implementation plan

Primary implementation shape:

1. Introduce or rename the prediction execution command to `predict`.
2. Route prediction execution through a dedicated application path rather than old migrate-up semantics.
3. Build prediction prompts around whole migration files.
4. Add prediction progress state that records which migrations have been successfully predicted.
5. Invalidate stale prediction progress when prior predicted migration content changes.
6. Reshape `test` parsing so `now`, `future`, and `new` are explicit actions.
7. Keep `materialize` separate as the implementation-application step.

## Implementation boundaries

Primary files likely involved:

- `implementation/src/presentation/cli.ts`

- `implementation/src/application/migrate-task.ts`

## Acceptance criteria

- `rundown predict` is a first-class command that processes migrations into `prediction/` without creating new migrations.
- Prediction execution is defined in terms of whole migration files, not checkbox-level task execution.
- Prediction progress is stored explicitly enough to know which migrations have already been applied to `prediction/` and when that progress became stale.
- `rundown test now` and `rundown test future` are accepted as explicit target selectors.
- `test now` validates implementation/current state and `test future` validates prediction state.
- `materialize` remains a separate command and is not collapsed into `predict`.

## Current implementation context (research baseline)

### CLI surface today

- `cli.ts` currently registers `migrate`, `materialize`, and `test`, but no `predict` command.
- `materialize` already exists as an explicit wrapper over run-all/revertable semantics and is separate from migration planning concerns.
- `test` currently exposes `[action] [prompt]` where only `new` is treated as an action in `createTestCommandAction`; non-`new` positional input is treated as a directory fallback.
- Existing `test` docs still reference `--future`, but the current parser path does not map that flag into actionful `now/future` routing.

### `test` application behavior today

- `test-specs.ts` currently operates in a single mode with `TestContext.mode = "materialized"`.
- Included/excluded directory hints are currently built for materialized-state assertions; `prediction/` is explicitly ignored for verdict decisions in default materialized guidance.
- Template loading supports `test-materialized.md` and fallback `test-verify.md`, while built-in defaults also include a `DEFAULT_TEST_FUTURE_TEMPLATE` that is not currently wired to explicit action parsing.
- `test new <assertion>` file creation behavior already exists and should remain intact under the new action family.

### migration/prediction model today

- `migrate-task.ts` is revision-aware and focused on generating migration drafts and promoting migration files; it is not currently exposing public `up/down` action routing in the command action layer.
- `createMigrateCommandAction` currently rejects string actions as unknown, which means prior `migrate up` language is already effectively non-primary in this implementation line.
- `migrate-task.ts` already resolves and forwards workspace routing variables including `workspacePredictionPath`, which is useful groundwork for a dedicated prediction executor prompt contract.
- migration lanes/threads and archive-aware file enumeration already exist; prediction execution should preserve lane ordering assumptions where relevant.

### workspace and design context

- Managed design docs are expected at `design/current/**` with compatibility-only fallback to legacy layouts.
- Current workspace resolution logic already treats resolved absolute `workspace*Path` values as authoritative and includes prediction as a first-class logical root.
- This migration should keep that routing contract intact and reuse the same path resolution utilities for `predict` and `test future` context shaping.

## Proposed command and parsing contract detail

### `predict` command contract

- Add top-level `predict` command as a first-class sibling of `migrate` and `materialize`.
- Keep `predict` focused on applying existing migration files into `prediction/`; no migration drafting, no design revision release, no migration synthesis.
- Recommended options mirror existing command conventions where appropriate (`--dir`, `--workspace`, `--worker`, `--show-agent-output`, `--keep-artifacts`), while remaining minimal for the first migration.

### `test` action grammar

- Canonical actions become:
  - `test now`
  - `test future`
  - `test new <assertion>`
- Omitted action should be treated as compatibility behavior, with explicit docs signaling `now` as the canonical form.
- `test new` remains orthogonal to target-state choice and should continue to support `--run`.
- Parser validation should reject ambiguous `test <unknown-action>` with explicit guidance to valid actions.

## Prediction execution architecture detail

### unit of work and worker prompt shape

- For each unapplied migration file, prompt input should include:
  - migration file path and full content,
  - authoritative workspace routing (`workspacePredictionPath`, `workspaceImplementationPath`, etc.),
  - explicit instruction that only prediction state is being advanced.
- The migration file itself is the execution unit; checklist extraction from migration Markdown is not used to decide sub-work items.
- Verification and repair loops run once per migration pass, not per checkbox line in migration content.

### sequencing and persistence model

- Execution is strictly ordered by migration number and canonical file naming.
- Progress persistence should occur after each migration passes verify/repair, so partial forward progress survives interruptions.
- On rerun, the executor resumes from the next unapplied migration according to persisted prediction state.

### invalidation and stale-progress handling

- Progress entries should include migration content hash (or equivalent deterministic digest).
- If a previously-applied migration file changes content, all downstream predicted progress is stale.
- Stale detection should force replay from the first changed migration (inclusive) and should emit clear operator-facing diagnostics.

## Prediction progress state requirements

### storage boundary

- Persist dedicated prediction progress under `.rundown/` rather than migration planning metadata.
- Keep schema independent from run artifacts so state survives artifact pruning and remains command-authoritative.

### minimum state schema expectations

- `lastAppliedMigrationNumber` (or equivalent pointer).
- Per-migration records:
  - migration identifier (number + filename),
  - content hash,
  - applied timestamp/status,
  - optional run/verification metadata reference for diagnostics.
- Optional global invalidation fingerprint:
  - workspace routing fingerprint,
  - prediction root path,
  - schema version for forward migrations.

### compatibility notes

- If state file is absent: treat as fresh prediction and start from first migration.
- If state file is unreadable/corrupt: fail safely with guidance or auto-recover to replay-all behavior (decision to be explicit during implementation).
- If migration files are deleted/renumbered: treat as invalidated state and require replay from coherent baseline.

## Verification semantics for `test now` and `test future`

### `test now`

- Validates current implementation/materialized state.
- Keeps current include/exclude behavior semantics for implementation truth.
- Uses materialized template chain (`test-materialized.md` fallback behavior) as canonical.

### `test future`

- Validates prediction state.
- Should use prediction-aware template chain (`test-future.md` fallback to shared verify template).
- Must expose explicit mode/environment hints so worker instructions are deterministic and mode-aware.

### shared result contract

- Preserve existing verdict protocol:
  - `OK`
  - `NOT_OK: <reason>`
- Maintain per-spec pass/fail summary behavior.

  - `predict` wording must clearly say “apply migrations into prediction”.
  - `test` wording must mention explicit target actions (`now`, `future`, `new`).

## Integration and boundary notes

### keep `migrate` and `predict` decoupled

- `migrate` remains producer of migration files.
- `predict` remains consumer/executor of migration files.
- Do not add implicit prediction execution as a hidden `migrate` side effect in this migration.

### keep `materialize` decoupled

- `materialize` remains implementation-application execution path and must not be merged with `predict`.
- This preserves clean stage semantics:
  - planning (`migrate`)
  - future-state projection (`predict`)
  - implementation application (`materialize`)

## Edge cases and operational constraints

- No migrations present: `predict` exits cleanly with no-work/info result.
- Migrations present but already fully predicted: `predict` exits cleanly and reports up-to-date state.
- Mid-run failure on migration `N`: migrations `< N` remain committed in prediction progress; rerun resumes at `N`.
- Migration content edited after successful prediction: stale progress detected and replay begins at first changed migration.
- Workspace routing conflicts or invalid mount map: fail before mutation with actionable error.
- Threaded/archived migration lanes: ordering and de-dup semantics must remain deterministic and consistent with migration discovery rules.
