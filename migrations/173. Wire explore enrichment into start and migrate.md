Wire explore enrichment into start and migrate

## Intent

`rundown start` and `rundown migrate` currently claim to run explore enrichment on newly created Markdown files, but the application wiring never provides the programmatic explore runner those flows depend on. The result is deterministic warnings like `Explore integration is not configured; skipping enrichment for ...` and silently missing enrichment work.

This migration makes that behavior real instead of aspirational: newly created target and migration documents must be enriched through the same research+plan orchestration that the `explore` command exposes.

## Current behavior and root cause

Relevant implementation surfaces:
- `src/application/start-project.ts` defines optional dependency `runExplore?: (source: string, cwd: string) => Promise<number>` and calls it after creating the target document and first migration.
- `src/application/migrate-task.ts` defines the same optional dependency and calls it after creating each new migration file.
- `src/create-app.ts` constructs both use cases without passing any `runExplore` implementation, so the callback is always `undefined` in normal app execution.
- `src/presentation/cli-command-actions.ts` already contains real `explore` orchestration via `runExplorePhases()` that sequences `research` then `plan`.

Observed consequences:
- `start` creates the target file and initial migration, then logs `Explore integration is not configured; skipping enrichment for ...` instead of enriching them.
- `migrate` creates migration files, then logs the same warning for each file instead of enriching them.
- Existing migration/design documents already describe enrichment as part of the intended workflow, so current behavior is below spec rather than merely noisy.

## Path and workspace considerations

- The canonical primary design file is `Target.md` under the managed design current directory.
- The parent design directory is configurable via workspace directories.
- `start --from-design` may point `design/current` at an external absolute path.
- The fix must preserve current path resolution behavior; this migration is about missing orchestration wiring, not about renaming `Target.md` or changing workspace directory semantics.

## Desired behavior

- `rundown start` enriches newly created documents immediately after scaffolding them:
  - managed design current primary file when Rundown creates it;
  - initial migration file.
- `rundown migrate` enriches each newly created migration file before continuing to any confirm/prediction step.
- Enrichment uses the same effective semantics as `rundown explore <file>`: research first, then plan, on the same file.
- If explore execution fails, `start`/`migrate` should fail with a real command error rather than downgrade to a configuration warning.
- If a path is intentionally skipped because Rundown did not create that file in the current flow, the code should skip quietly and explicitly rather than emit a false configuration warning.

## Scope

In scope:
- Replace the unbound optional `runExplore` dependency path with actual application wiring for programmatic explore orchestration.
- Share or extract the existing explore sequencing logic so `start`, `migrate`, and the CLI `explore` command are behaviorally aligned.
- Add focused tests that prove start/migrate-created files are enriched rather than merely created.
- Remove or narrow the misleading `Explore integration is not configured` warning path from normal runtime behavior.

Out of scope:
- Renaming `Target.md` or changing canonical design workspace conventions.
- Redesigning the `explore` CLI interface.
- Changing migration planning semantics beyond making post-create enrichment actually run.

## Design notes

- Prefer a real use case or helper for programmatic explore execution over ad hoc shelling out to the CLI binary.
- The composition root should own the orchestration wiring so `start` and `migrate` receive a concrete runner.
- Reuse existing research/plan use cases or a shared helper derived from `runExplorePhases()` instead of duplicating sequencing logic in multiple places.
- Keep `start` behavior sensitive to whether Rundown created the target file itself versus using an external `design/current` directory.

## Acceptance criteria outline

- Running `start` in normal app execution no longer emits `Explore integration is not configured` for the scaffolded target or initial migration.
- Running `migrate` in normal app execution no longer emits that warning for newly created migration files.
- The code path that enriches created files is wired from `create-app.ts`, not left as an unbound optional callback in production composition.
- `start` still respects configured workspace directories and `workspace.design.currentPath` overrides.
- `migrate` still respects configured migrations directory resolution.
- Programmatic enrichment uses research+plan sequencing equivalent to the `explore` command.
