# 194. Migrate Supports File Input And New Shortcut Without Design Workspace

## Problem

The current migrate model assumes a managed design workspace:

- `design/current/` as mutable draft input,
- immutable released revisions under `design/revisions/`,
- and revision-aware planning over those design boundaries.

That works for full rundown workspaces, but it prevents a smaller ad hoc workflow where the user only has a single plan/design file and still wants rundown to generate migrations.

The product should be operable even when no `design/` directory exists.

At the same time, users need a fast way to create the next migration file manually without invoking the full planning pipeline.

## Desired behavior

Recommended new surfaces:

- `rundown migrate --from-file Plan.md`
- `rundown migrate new "File name basically"`

Interpretation:

- `migrate --from-file <file>` uses one explicit file as migrate input and creates migrations in `migrations/` without requiring a managed design revision workspace.
- `migrate new <title>` creates the next ordered migration file for the current scope as a convenience shortcut.

These surfaces should coexist with the existing managed-workspace migrate flow rather than replacing it.

## Behavior contract

### 1. `migrate --from-file <file>` is a design-less migrate mode

This mode is for lightweight migration authoring from one explicit source file.

The file should be treated as the authoritative planning input for that run.

Expected behavior:

1. read the file content directly,
2. build migrate planning context from that file,
3. create migration files in the selected migration scope,
4. do not require `design/current/`, released revisions, or revision metadata.

This is intentionally a file-input planning mode, not a hidden bootstrap of the full design revision system.

### 2. `migrate --from-file` does not depend on version-controlled design revisions

When this mode is used, migrate should not fail just because `design/` is absent.

It should also avoid manufacturing fake design revisions just to satisfy the current revision-aware planner contract.

Instead, this mode should define a direct file-input planning path that still produces canonical migration files in `migrations/`.

### 3. File-input migrate should still honor thread-aware drafting

If `.rundown/threads/*.md` exists, file-input migrate should still pass through the existing thread-aware drafting model.

That means:

- the explicit file content becomes the shared planning source,
- thread briefs still specialize that source,
- and migrations still promote either into the root lane or into thread lanes according to the existing thread rules.

Threads should not require a managed design revision archive in order to specialize a file-input migrate run.

### 4. `migrate new <title>` is only a shortcut for creating the next migration file

This command should:

1. determine the next migration number for the current scope,
2. create `N. <Title>.md` using canonical filename formatting,
3. and stop there.

It should not:

- release revisions,
- inspect design or implementation state,
- run planning,
- or invoke prediction/materialization.

This is a convenience authoring shortcut, not a planning workflow.

### 5. "Next number" is scope-local

`migrate new` must use the next number for the currently selected migration scope.

That means:

- root lane uses the next number in `migrations/`,
- thread lanes use the next number inside their own `migrations/threads/<thread>/` directory when a thread-scoped create path is later exposed or implied.

The shortcut should follow the same numbering locality as normal migrate promotion.

### 6. `migrate new` should use canonical migration filename formatting

The generated filename should use the same canonical dotted title-case filename style as the rest of the migrate system.

Example:

- `132. File name basically.md`

The shortcut should reuse the same formatting logic used by normal migrate promotion rather than inventing a parallel naming convention.

### 7. Managed design migrate remains unchanged

These new surfaces are additive.

The existing managed design workflow should still work as the canonical full-featured path:

- `rundown migrate`
- `rundown migrate --from implementation`
- `rundown migrate --from prediction`

`--from-file` is the design-less escape hatch, not the new default model.

## Scope

In scope:

- add a file-input migrate mode that does not require a `design/` folder,
- allow that mode to create migrations directly from one explicit file,
- keep thread-aware drafting available in file-input mode,
- add `migrate new <title>` as a convenience shortcut,
- and preserve canonical numbering/filename rules.

Out of scope:

- bootstrapping a synthetic design revision archive from the file,
- changing prediction or materialize semantics,
- redesigning thread mechanics,
- or adding path-scoped migrate for general directories.

## Implementation plan

Primary implementation shape:

1. Extend migrate CLI parsing with `--from-file <path>` as a mutually exclusive planning-input mode.
2. Add a file-input application branch that reads one explicit file and bypasses design-revision requirements.
3. Reuse the existing migration drafting engine as much as possible, but feed it a file-derived planning source instead of a released design diff.
4. Preserve thread discovery and thread-brief specialization for this branch.
5. Add `migrate new <title>` as a separate shortcut branch under migrate command routing.
6. Reuse existing migration parser/formatter utilities to compute next number and canonical filename.

## Acceptance criteria

- `rundown migrate --from-file <file>` is accepted even when no `design/` directory exists.
- In file-input mode, migrate reads the specified file as the planning source and creates canonical migration files in `migrations/`.
- File-input migrate does not require fake revision creation or a managed design archive.
- If thread documents exist, file-input migrate still uses thread-aware drafting and promotes migrations into the correct root or thread lane.
- `rundown migrate new <title>` creates exactly one new canonical migration file with the next number for the current scope.
- `migrate new` does not trigger planning, prediction, or revision release side effects.
- Canonical migration numbering and dotted filename formatting remain consistent with existing migrate behavior.
