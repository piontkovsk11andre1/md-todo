# 169. Port TUI to TypeScript

## Problem

The TUI lives under `implementation/src/presentation/tui/` as plain JavaScript (155). The rest of the source tree is TypeScript with strict typing through Pylance/tsc, and the App's output events (`ApplicationOutputEvent`) are precisely typed in [src/domain/ports/output-port.ts](implementation/src/domain/ports/output-port.ts). The recent regression where `currentOperation = "verify"` was effectively dead would have been caught by the type system if the TUI were typed.

Continuing to grow the TUI in JS will accumulate the same class of regression: silent label drift between domain events and the TUI's switch statements, untyped scene state machines, untyped key handlers.

## Goal

Convert every file under `implementation/src/presentation/tui/` from `.js` to `.ts`, with strict typing, and integrate it into the existing build (`tsup`) and test (`vitest`) configurations.

This is a mechanical port. Behavior must be identical before and after; review should be a noop diff modulo type annotations.

## Model

Phase 1 — module conversion:

- Rename every `.js` under `src/presentation/tui/` to `.ts`.
- Replace inline JSDoc with explicit TypeScript types.
- Replace `import { createApp } from "../../create-app.js"` (already adjusted in 155) with the typed import; let `App` and `ApplicationOutputEvent` flow.
- Add explicit types for scene state (`SceneId`, `MainMenuState`, `ContinueState`, etc.) and for the navigation stack from 168.

Phase 2 — domain alignment:

- Replace the `applyOutputEvent` switch with an exhaustive switch keyed on `event.kind`, using a `never` exhaustiveness check at the default case so future additions to `ApplicationOutputEvent` force a TUI update.
- Replace the operation-color lookup with a typed map keyed on the literal labels emitted by `progress` events (`"verify" | "repair" | "resolve" | "resolveRepair" | "plan" | "research" | "discuss" | "execute" | "scan" | "finalize"`). Unknown labels render dimmed without color; the type system flags missing entries.

Phase 3 — build / test:

- Confirm `tsup.config.ts` already picks up `src/presentation/**/*.ts`. If not, extend the entry list.
- Add a smoke test under `__tests__/presentation/tui/index.test.ts` that imports the TUI module and asserts it exports the expected entry function. Real interactive tests live in 170.
- Update `package.json` `scripts.tui-sketch` to `node ./dist/presentation/tui/index.js` if the entry point changes, or keep `tsx ./src/presentation/tui/index.ts` for dev iteration.

## Implementation boundaries

Files expected to change:

- every `.js` under `implementation/src/presentation/tui/` becomes `.ts`.
- new type modules under `src/presentation/tui/types/` if shared interfaces grow large enough to warrant a home (`scene.ts`, `keybinding.ts`, `status-probe.ts`).
- `tsup.config.ts`, `tsconfig.json`, `package.json` — only if the existing globs do not already cover the new files.

Out of scope:

- Behavior changes.
- Restructuring scenes.
- Type-driven refactors that change output (e.g., changing how event kinds are dispatched). The port preserves observable behavior.

## Acceptance

- `npx tsc --noEmit` passes.
- `npx vitest run` passes with no new test failures.
- `node ./dist/presentation/tui/index.js` (or the dev-mode equivalent) boots the TUI identically to before the port.
- Removing a literal from the `progress.label` union (or adding a kind to `ApplicationOutputEvent`) causes a compile error in the TUI dispatch — verify by introducing a temporary mismatch and confirming `tsc` flags it.
- The Pylance "no implicit any" check passes across the new `.ts` files.
