# 190. Fold Design Revision Lifecycle Into Migrate

## Problem

The current product splits one revision-aware workflow across two command surfaces:

- `rundown design ...` owns release and diff of immutable design revisions,
- `rundown migrate` owns migration planning against released revisions.

That split was introduced to separate revision lifecycle from migration lifecycle, but it now creates an unnecessary extra step in the normal authoring flow.

Today a user who updates `design/current/` must remember to:

1. release a new immutable design revision, and only then
2. run migrate so the tool can plan work for the next unplanned revision.

In practice, these are not independent workflows for the main path. Migration planning already depends on design revision state, revision diffs, and revision metadata. The extra command boundary adds friction without adding meaningful control for the default case.

The target direction is simpler: `migrate` should own the full revision-aware planning flow, including creating the next immutable design revision when the current draft changed.

## Desired behavior

After this migration, the canonical user flow is:

1. edit `design/current/**`,
2. run `rundown migrate`,
3. let migrate snapshot the current design into the next immutable revision when needed,
4. then let migrate plan against the next unplanned released revision.

This means the standalone `design` command surface disappears as a user-facing workflow.

The runtime should keep the existing design revision model:

- mutable draft state lives under `design/current/`,
- immutable released revisions live under `design/revisions/rev.N/`,
- and revision progress is still tracked through revision metadata sidecars, especially `plannedAt` and `migrations`.

The command boundary changes, but the underlying revision and planning model should remain stable.

## Behavior contract

### 1. `rundown migrate` becomes the only user-facing entrypoint for revision-aware planning

The default planning command should own the entire flow that previously required both `design` and `migrate`.

That means:

- `rundown migrate` remains the canonical planning command,
- users do not need to run a separate release command before planning,
- and revision synchronization becomes part of migrate orchestration rather than a separate user action.

This migration does not require introducing a new `migrate run` subcommand. The current top-level `rundown migrate` entrypoint should stay canonical unless another migration explicitly changes that CLI shape later.

### 2. Migrate performs a preflight design revision sync before planning

Before the planning loop looks for the next unplanned revision, migrate should reconcile the mutable current draft with released revisions.

The intended preflight behavior is:

- if `design/current/` differs from the latest released revision, migrate creates the next immutable revision under `design/revisions/rev.N/`,
- if `design/current/` is unchanged from the latest released revision, migrate does not create a new revision,
- if there are no released revisions yet, migrate creates the initial baseline revision from `design/current/` when the managed design workspace exists,
- and after that preflight, migrate continues with normal revision-aware planning using the released revision set.

This keeps the common path single-step while preserving immutable released revisions as the actual planning targets.

### 3. Revision metadata remains the source of truth for progress

This migration should not replace the existing revision progress model with a separate migrate cursor.

The current revision metadata already captures the planning boundary:

- `plannedAt` indicates whether a released revision has been planned,
- `migrations` records the migration files associated with that revision boundary,
- and the next planning target is still the lowest released revision whose metadata shows it is not yet planned.

The new preflight release step should feed that same model rather than bypassing it.

In other words:

- newly auto-created revisions begin unplanned,
- unchanged current drafts do not create fake revision entries,
- and migrate continues to walk released revision boundaries through the existing metadata contract.

### 4. Immutable revisions stay canonical; `current` remains only the draft workspace

Even though migrate now creates revisions automatically, planning should still be framed around immutable released snapshots rather than around the mutable draft directory itself.

That means:

- migration planning still targets released revisions,
- revision diffs still compare released revision pairs or a released revision against the current preflight result,
- and prompt context that is meant to describe the target state should continue to come from the relevant released revision payload under `design/revisions/rev.N/`.

This migration is about command ownership, not about collapsing the revision model into `design/current/`.

### 5. The standalone `design` command surface is removed

After this migration, release and diff are no longer separate end-user commands.

The product surface should stop exposing:

- `rundown design release`,
- `rundown design diff`,
- and any remaining compatibility surfaces that only exist to preserve the old split workflow.

Internal helpers for saving revisions, discovering revisions, and computing revision diffs may remain as implementation details. The change is about user-facing ownership, not about deleting useful internal primitives.

If implementation needs a temporary compatibility bridge, it should be a short-lived transition detail rather than a continued parallel workflow.

### 6. Messages and docs must describe migrate as owning revision creation

Any wording that currently tells the user to run `rundown design release` must be updated.

Examples of messaging that need to change:

- no-revision bootstrap guidance,
- caught-up guidance,
- CLI help text,
- README examples,
- and any prompt or template text that explains the revision lifecycle.

The user-facing story should become:

- edit `design/current/`,
- run `rundown migrate`,
- migrate will release a new immutable revision if needed,
- then migrate will plan the next revision boundary.

### 7. Linked-workspace and configured-design-root semantics remain unchanged

This migration should not change how the tool decides which workspace or design root it is operating on.

The existing rules for:

- linked workspace resolution,
- explicit `--workspace` selection,
- configured design directories,
- and external `design/current` overrides

should continue to apply.

The only behavior change is that migrate, once pointed at the correct workspace, performs revision release preflight itself.

### 8. Legacy flat revision layouts remain readable as compatibility paths

The canonical revision archive remains `design/revisions/rev.N/`, consistent with recent migration history.

Older workspaces that still rely on flat `design/rev.N/` layouts should continue to be readable where existing compatibility rules already support them.

This migration should not reopen the older flat layout as a new canonical write target. Automatic revision creation owned by migrate should still write to the canonical revisions archive location.

## Scope

In scope:

- move design revision release ownership into migrate orchestration,
- remove the standalone user-facing `design` command surface,
- preserve the existing revision metadata model as the planning cursor,
- update migrate messaging and help to describe the new single-step workflow,
- and update tests and docs around the new ownership boundary.

Out of scope:

- changing the revision diff algorithm,
- changing the immutable revision format or metadata schema,
- introducing a brand new revision progress store,
- changing linked-workspace resolution behavior,
- or automatically relocating old flat `design/rev.N/` repositories.

## Implementation boundaries

Primary code and docs surfaces likely involved:

- `implementation/src/application/migrate-task.ts`
- `implementation/src/application/design-context.ts`
- `implementation/src/application/docs-revision-task.ts`
- `implementation/src/presentation/cli.ts`

- `implementation/src/create-app.ts`
- `implementation/README.md`
- `README.md`
- tests under `implementation/__tests__/integration/` and `implementation/__tests__/presentation/`

Expected implementation shape:

- add a revision-release preflight step to migrate before it selects the next unplanned revision,
- reuse existing design revision save/discovery/diff helpers instead of inventing a parallel mechanism,
- remove presentation-layer command registration and routing for the old `design` flow,
- update migrate output strings so they no longer instruct the user to release revisions manually,
- and keep revision discovery/planning anchored to the existing `plannedAt` and `migrations` metadata semantics.

## Documentation and migration notes

Docs should explicitly describe this as a product simplification rather than a change to the underlying revision model.

That documentation should explain:

- `design/current/` is still the editable draft,
- `design/revisions/rev.N/` is still the immutable release archive,
- `rundown migrate` now performs the release step automatically when current design changed,
- revision progress is still tracked per released revision through metadata,
- and the previous standalone `design` commands are no longer the supported workflow.

Where old docs showed a two-step flow, examples should be rewritten to show the new single-step migrate flow.

## Acceptance criteria

- Running `rundown migrate` on a workspace whose `design/current/` differs from the latest released revision creates exactly one new immutable revision under `design/revisions/rev.N/` before planning continues.
- Running `rundown migrate` on a workspace whose `design/current/` matches the latest released revision does not create a new revision and still plans any existing lowest unplanned released revision using current metadata rules.
- When there are no released revisions yet and the managed design workspace exists, `rundown migrate` bootstraps the initial released revision itself instead of requiring a prior manual release command.
- Revision progress continues to be determined by released revision metadata, with `plannedAt` and `migrations` remaining the source of truth for which revision boundary should be planned next.
- User-facing CLI help, runtime messages, and docs no longer instruct users to run `rundown design release` or `rundown design diff` as part of the canonical workflow.
- The standalone `design` command surface is removed from CLI registration and routing, aside from any strictly temporary compatibility bridge explicitly chosen during implementation.
- Linked-workspace resolution, configured design-root handling, and external current-path behavior remain unchanged aside from migrate taking ownership of the release preflight.
- Canonical automatic revision writes continue to target `design/revisions/rev.N/`, while legacy flat `design/rev.N/` layouts remain readable only through existing compatibility behavior.
