# 181. Migrate Stages Drafted Migrations Before Promotion

## Problem

`rundown migrate` currently asks a worker to output migration names, parses those names, writes minimal placeholder files, and then runs `explore` on the real `migrations/` directory.

- The worker only contributes slugs, not actual migration seed content.
- The app writes generic migration bodies like `- [ ] Implement this migration`.
- There is no migrate-specific verification or repair loop for migration coverage.
- Partially useful or invalid outputs can reach the real `migrations/` directory before the system knows whether they actually cover the target design drift.

That makes `migrate` more of a naming pass than a migration-drafting flow.

## Desired behavior

After this migration, `rundown migrate` should draft candidate migration files in the current run artifact directory first, verify and repair those staged drafts against the target design diff, and only then promote them into the real `migrations/` directory.

The worker should author migration seed documents directly.

- `migrate` still discovers the next unreconciled design revision.
- `migrate` still shows design diff and migration history to the worker.
- The worker creates full candidate migration markdown files in a staging directory under `.rundown/runs/<run-id>/`.
- Verification checks only those staged candidate migrations against the current revision diff.
- Repair updates only the staged drafts.
- Promotion into `migrations/` happens only after verification succeeds.
- `explore` runs only on promoted migration files.

The result should be that `migrate` becomes a drafting-and-convergence loop for migration documents, not just a slug planner.

## Behavior contract

### 1. Drafting happens in run-artifact staging

For each migrate run, candidate migration documents are created in a staging directory inside the current artifact run directory.

- Use the active run artifact root under `.rundown/runs/<run-id>/`.
- Store candidate migration files in a dedicated subdirectory such as `drafted-migrations/`.
- The worker must not write directly into the real `migrations/` directory during drafting.

This staging area is the only place where migrate drafting and repair are allowed to mutate files before promotion.

### 2. Worker drafts migration files, not slug output

The migrate drafting worker should create candidate migration markdown files directly in the staging directory.

- Do not ask for kebab-case migration names on stdout.
- Do not parse list items as the primary migrate output contract.
- The worker may still use stdout for status text, but the app should determine drafted migrations from staged files on disk.

The drafting prompt should ask the worker to read the design diff and current migration history, then author one or more migration seed documents that together cover the uncovered design change.

### 3. Canonical filename rules stay enforced by the app

Drafted migration files must use canonical migration filenames.

- Migration files must be named `N. Title.md`.
- Drafted migration numbers must be greater than the current highest real migration number.
- Drafted files must form a continuous forward range without gaps.
- The worker must not modify existing real migrations.

If the staged result violates these rules, verification must fail before promotion.

### 4. Verification checks staged migration coverage

After drafting, `migrate` should verify whether the staged candidate migrations cover the current revision drift.

Verification should answer questions like:

- Do the drafted migrations collectively cover the relevant design diff?
- Are any important changed areas still uncovered?
- Are there duplicate or excessively overlapping drafted migrations?
- Are the drafted migrations materially meaningful, rather than empty or placeholder-only?

The verification target is the staged migration set for the current revision, not the full historical directory alone.

### 5. Repair mutates staged drafts only

If verification fails, `migrate` should run a repair loop against the staged migration files.

- Repair operates only on files inside the staging directory.
- Repair must not touch the real `migrations/` directory.
- Reverification runs after each repair attempt.
- If repair attempts are exhausted, the run fails and preserves staged artifacts for inspection.

This reuses the existing verify/repair model, but points it at staged migration documents.

### 6. Promotion happens only after verification passes

Once staged migrations pass verification, the app promotes them into the real `migrations/` directory.

- Promotion is app-controlled, not worker-controlled.
- Promoted files keep the exact canonical filenames that were validated in staging.
- Only newly promoted files are considered created by this migrate iteration.

After promotion, `explore` runs on the promoted migration files to enrich them further.

### 7. Revision planning state tracks promoted migrations only

When `migrate` marks a revision as planned, it should record only the migration files that were actually promoted.

If drafting or verification fails before promotion, the revision must remain unplanned.

## Scope

In scope:

- Replace slug-based migrate planning with staged migration drafting.
- Add migrate-specific staged verification and repair behavior.
- Keep using design diff plus migration history as worker context.
- Promote staged canonical migration files into `migrations/` only after verification succeeds.
- Run `explore` only after promotion.
- Preserve failed staged drafts in run artifacts for debugging.

Out of scope:

- Changing how `materialize` executes migrations later.
- Reworking general task execution verification contracts unrelated to `migrate`.
- Allowing workers to edit existing migration history directly.
- Changing the canonical migration filename convention.

## Acceptance criteria

- [ ] `rundown migrate` no longer depends on parsing migration slugs from worker stdout as the primary planning result.
- [ ] The drafting worker creates candidate migration files under the current run artifact directory, not directly in `migrations/`.
- [ ] Drafted migration files must use canonical `N. Title.md` filenames and numbers strictly above the current highest real migration.
- [ ] Migrate verifies staged drafted migrations against the target design diff before any promotion.
- [ ] Failed verification triggers repair against staged drafts only.
- [ ] Exhausted repair attempts leave the real `migrations/` directory unchanged and preserve staged artifacts for inspection.
- [ ] Successful verification promotes the staged migrations into the real `migrations/` directory.
- [ ] `explore` runs only on promoted migration files, not on failed staged drafts.
- [ ] Revision planning state records only promoted migrations.
- [ ] Focused tests cover success, verification failure, repair success, and no-promotion-on-failure behavior.
- [ ] verify: run migrate-focused tests covering staged drafting, verification, repair, and promotion boundaries.
