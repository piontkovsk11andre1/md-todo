# 172. Empty-root menu uses Start Workers Help

## Problem

The root TUI currently opens the same six-row main menu for every interactive workspace:

1. `Continue`
2. `New Work`
3. `Workers`
4. `Profiles`
5. `Settings`
6. `Help`

That works for initialized rundowns, but it is the wrong affordance in an empty directory. In that state there is nothing to continue, no project scaffold for new work, and the most important action is to bootstrap the workspace with `rundown start`.

Showing `Continue` and `New Work` first in an empty directory makes the UI feel broken or premature. The menu should collapse to the actions that are actually meaningful before initialization.

## Goal

When `rundown` starts in an empty directory, replace the default main menu with a compact bootstrap menu:

1. `Start`
2. `Workers`
3. `Help`

Selection behavior:

- `Start` runs the same flow as `rundown start`.
- `Workers` opens the Workers scene, but the row must show a warning that no workers are configured yet.
- `Help` opens the existing Help scene.

Outside the empty-directory case, the existing six-row main menu remains unchanged.

## Model

Treat this as a distinct main-menu variant, not as a cosmetic rename of existing rows.

### Empty-directory detection

Add a small bootstrap-state detector for the root TUI. The detector should answer whether the current working directory is effectively uninitialized for rundown.

For this migration, "empty directory" means:

- the current working directory has no user files or directories yet, or
- it does not contain initialized rundown workspace structure such that `Continue` and `New Work` are actionable.

Implementation should prefer a single explicit helper rather than scattering `fs.existsSync(...)` checks across menu rendering, probe code, and routing.

Suggested helper surface:

```ts
type RootWorkspaceState = {
  isEmptyBootstrap: boolean;
  hasWorkersConfigured: boolean;
};

function detectRootWorkspaceState(cwd: string): RootWorkspaceState;
```

The exact heuristic can be refined in implementation, but it must be deterministic and shared by both menu rendering and action routing.

### Menu variants

Extend the main-menu scene model in `implementation/src/presentation/tui/scenes/main-menu.ts` so it can render one of two row sets:

- default initialized menu: `Continue`, `New Work`, `Workers`, `Profiles`, `Settings`, `Help`
- empty bootstrap menu: `Start`, `Workers`, `Help`

This should remain keyboard-compatible with the current grammar:

- `Up` / `Down`, `j` / `k`
- `1` ... `9`
- `Enter`

The menu state should track the active variant so selection normalization and numeric jumps use the visible rows, not the full six-row set.

### Row semantics

`Start`

- New top-level route target for the empty bootstrap menu.
- Enter triggers the same application behavior as the CLI `start` command.
- The TUI should hand off cleanly to the existing start flow rather than duplicating scaffold logic inside the scene.

`Workers`

- Reuse the existing Workers scene.
- In the empty bootstrap variant, its status text should explicitly warn that no workers are configured yet.
- Opening Workers from an empty directory must still work and should preserve the scene's current empty-config behavior.

`Help`

- Reuse the existing Help scene without special behavior.

### Probe/status behavior

The existing main-menu probe registry assumes the initialized menu row set. Adjust it so status text is variant-aware:

- `Start` gets a bootstrap-oriented status, for example `scaffold design/ + migrations/`.
- `Workers` reports a warning such as `no workers configured yet` when the bootstrap detector says the workspace is empty or unconfigured.
- `Help` remains unchanged.

Do not keep stale `Continue` or `New Work` probes alive for a menu variant that does not render those rows.

### Routing

Update the root TUI router in `implementation/src/presentation/tui/index.ts` so main-menu selection dispatch can target `start` when the bootstrap variant is active.

Preferred behavior:

- selecting `Start` invokes `app.startProject(...)` through the existing application API
- after successful start, the TUI refreshes workspace detection, returns to the main menu, and now shows the initialized six-row variant
- if start fails or is cancelled, remain in the TUI and surface a banner/hint with the failure result

Avoid spawning a child `rundown start` process unless there is a concrete architectural blocker. The TUI already has in-process app access, and the start flow already exists as an application use case.

## Implementation boundaries

Files expected to change:

- `implementation/src/presentation/tui/scenes/main-menu.ts` - support menu variants, add `Start`, and compute rows from workspace state.
- `implementation/src/presentation/tui/status-probes.ts` - add bootstrap-aware probe logic for `Start` and `Workers`.
- `implementation/src/presentation/tui/index.ts` - detect empty bootstrap state, route `Start`, and refresh the menu after project initialization.
- `implementation/src/presentation/tui/layout.ts` - only if needed for menu copy or footer text.
- `implementation/__tests__/presentation/tui/*` - add coverage for the empty bootstrap menu, Start routing, and post-start refresh.

Optional extraction if the logic becomes noisy:

- `implementation/src/presentation/tui/root-workspace-state.ts` - shared detector for root workspace/bootstrap state.

Out of scope:

- Changing the initialized six-row menu.
- Reworking Workers scene internals.
- Rewriting `start-project` scaffolding behavior.
- Adding Profiles or Settings to the empty bootstrap menu.

## Acceptance

- `rundown` in an empty directory opens a three-row menu: `Start`, `Workers`, `Help`.
- `Continue` and `New Work` are not shown in that state.
- The `Workers` row displays a warning that no workers are configured yet.
- Pressing Enter on `Start` runs the existing start-project flow.
- After successful start, the main menu refreshes into the normal initialized six-row menu.
- Opening `Workers` from the empty bootstrap menu still works and preserves its empty-config messaging.
- `Help` behavior is unchanged.
- Non-empty, already initialized workspaces still open the existing six-row menu.

## Notes

This migration is intentionally about first-run UX, not about redefining rundown's initialized navigation model. The important design constraint is to make the empty root feel like a bootstrap surface with one obvious next action.
