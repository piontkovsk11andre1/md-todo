# 158. Replace materialize-only menu with retro main menu

## Problem

The current TUI (now under `src/presentation/tui/` after 155) has a flat menu of action keys (`m`, `a`, `o`, …) optimized for prototyping a single flow. The integrated TUI needs a discoverable retro-style main menu that frames rundown as a workshop you walk into, not a key-press cheatsheet.

## Goal

Replace the current menu surface with a vertical main menu of six top-level entries, keyboard-navigable, with placeholder right-column status text. Status probes themselves are wired in 159; this migration only stands up the menu shell and the routing to scenes.

The six entries (in this order):

1. `Continue`
2. `New Work`
3. `Workers`
4. `Profiles`
5. `Settings`
6. `Help`

## Model

A new module `src/presentation/tui/scenes/main-menu.js` owns the menu state. The TUI top-level (`src/presentation/tui/index.js`) becomes a thin scene router:

```
runRootTui()
  ├─ render banner + frame
  ├─ scenes.mainMenu          (default)
  │    on Enter ─▶ scenes.continue / newWork / workers / profiles / settings / help
  ├─ scenes.continue          (= existing materialize cockpit, renamed)
  ├─ scenes.newWork           (= existing agent flow placeholder)
  ├─ scenes.workers           (placeholder; landed in 162)
  ├─ scenes.profiles          (placeholder; landed in 163)
  ├─ scenes.settings          (placeholder; landed in 166)
  └─ scenes.help              (placeholder; landed in 167)
```

Navigation grammar (must work in every scene unless explicitly overridden):

- `↑ ↓` / `j k` — navigate
- `Enter` — select / drill in
- `Esc` / `Backspace` — back to parent
- `q` / `Ctrl-C` — quit (Ctrl-C exits with 130)
- `?` / `h` — open inline help overlay (overlay text is per-scene in later migrations; this migration ships a static placeholder)
- `1`…`9` — jump to nth menu item

Right column: each menu row reserves a right-aligned area for status text. In this migration the status string is a hardcoded placeholder per row (e.g. `"…"` or `"loading…"`). Wiring real probes lands in 159.

The existing materialize flow keeps working but is reachable only via `Continue`. Single-key shortcuts (`m`, `a`, `o`, …) are removed from the new main menu; they may be reintroduced as a command palette in a later migration.

## Implementation boundaries

Files expected to change:

- new `implementation/src/presentation/tui/scenes/main-menu.js`.
- new `implementation/src/presentation/tui/scenes/continue.js` (extracts the materialize cockpit from current `index.js`).
- new `implementation/src/presentation/tui/scenes/new-work.js` (extracts the `agent` and `discuss-agent` flows from current `index.js`).
- new placeholder files for `workers.js`, `profiles.js`, `settings.js`, `help.js` under `scenes/`. Each renders a "Coming in migration NNN — press Esc to go back" panel.
- `implementation/src/presentation/tui/index.js` becomes a router with the navigation grammar above. Existing render helpers (`buildFrame`, `applyOutputEvent`, layout utilities) move into `src/presentation/tui/layout.js` and `src/presentation/tui/output-bridge.js` so multiple scenes can share them.

Out of scope:

- Real status text (159).
- Real workers/profiles/settings/help content (162/163/166/167).
- Command palette.

## Acceptance

- `rundown` (TTY) opens directly on the main menu with all six rows.
- Pressing Enter on `Continue` lands in the materialize cockpit identical to today.
- Pressing Enter on `New Work` lands in the existing agent flow.
- Pressing Enter on any other row shows the placeholder panel.
- `↑ ↓`, `j k`, number keys, Esc, q, Ctrl-C all behave per the grammar.
- Resizing the terminal does not break the frame (existing layout helpers continue to work).
