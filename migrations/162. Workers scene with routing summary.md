# 162. Workers scene with routing summary

## Problem

The Workers scene must present rundown's actual worker model, not a simplified caricature. Per [docs/configuration.md](implementation/docs/configuration.md):

- The worker pool is `workers.default`, `workers.tui`, and ordered `workers.fallbacks[]`.
- Per-command overrides live under `commands.<name>` for `run, plan, discuss, help, research, reverify, verify, memory, tools.<toolName>`.
- Per-phase routing for `run`/`reverify` lives under `run.workerRouting.{execute, verify, repair, resolve, resolveRepair, reset}`, with attempt selectors (`attempt`, `fromAttempt`, `toAttempt`) on `repair`/`resolveRepair`.
- Worker health (`worker-health.json`) tracks `ready` / `cooling_down` / `unavailable` per candidate, governed by `healthPolicy`.

A naive "list of workers" UI hides three of those four axes. Editing arbitrary nested paths through a TUI form would also reinvent JSON schema validation that already exists in [src/infrastructure/adapters/worker-config-adapter.ts](implementation/src/infrastructure/adapters/worker-config-adapter.ts).

## Goal

Build a Workers scene that:

1. **Reads** the full worker picture (pool + per-command + routing summary + health) and renders it in a single screen.
2. Routes any **edit** through `$EDITOR` opening `.rundown/config.json` (or whichever scope the user picks). After the editor exits, the scene reloads and re-renders. No bespoke form fields, no partial writes, no validation duplication.

The "routing" subsection in this scene is intentionally a **summary**, not an editor. Editing routing rules in a form UI is too risky given attempt selectors and `useFallbacks` semantics; the editor-on-config approach delegates to the same JSON validator the rest of rundown uses.

## Model

Scene layout:

```
┌─ Workers ─────────────────────────────────── .rundown/config.json ─┐
│                                                                    │
│  Pool                                                              │
│    default      <command>                                  ✓ ready │
│    tui          <command>                                  ✓ ready │
│    fallbacks    1. <command>                               ✓ ready │
│                 2. <command>                               ⚠ cooling│
│    workerTimeoutMs  120000                                         │
│                                                                    │
│  Per-command overrides (commands.<name>)                           │
│    plan         <command>                                          │
│    verify       <command>                                          │
│    discuss      <command>                                          │
│    tools.post-on-gitea  <command>                                  │
│    (rows hidden for unset commands)                                │
│                                                                    │
│  Phase routing (run.workerRouting) — summary                       │
│    execute        ✓ inherits                                       │
│    verify         ⚙ overridden                                     │
│    repair         ⚙ overridden · 2 attempt rules                   │
│    resolve        ⚙ overridden                                     │
│    resolveRepair  ⚙ overridden · 1 attempt rule                    │
│    reset          ✓ not configured (semantic reset disabled)       │
│                                                                    │
│  [e] edit config.json   [E] edit global config   [r] reload        │
└────────────────────────────────────────────────────────────────────┘
```

Health column rules:

- ✓ `ready` — green
- ⚠ `cooling_down` — yellow, with `until <hh:mm:ss>` and the failure class
- ✗ `unavailable` — red

Discovery:

- `workers.tui` row hidden when key is absent.
- `workers.fallbacks` group hidden when array is empty.
- Per-command and routing groups hide rows where no override is present (so an empty `{}` config renders a clean "Pool only" view).

`[e]` opens `<config-dir>/config.json` in `$EDITOR` (Windows fallback: `notepad`; otherwise `vi` / `nano` / `code -w` based on `$EDITOR`/`$VISUAL`). Block until the editor exits, then re-run config load. Surface JSON validation errors in a banner above the scene if the new config is invalid.

`[E]` (shift) opens the global config file resolved by `rundown config path --scope global`. If the global file does not exist, the scene offers to create an empty `{}` first.

## Implementation boundaries

Files expected to change:

- new `implementation/src/presentation/tui/scenes/workers.js`.
- new `implementation/src/presentation/tui/components/editor-launch.js` — `$EDITOR` / `$VISUAL` / OS fallback wrapper that returns when the editor exits.
- new `implementation/src/presentation/tui/bridges/config-bridge.js` — thin wrapper around `WorkerConfig` load, `rundown config list`, and `rundown config path --scope global`. Uses the in-process `App` API where available, otherwise re-implements the same load path against [src/infrastructure/adapters/worker-config-adapter.ts](implementation/src/infrastructure/adapters/worker-config-adapter.ts).
- new `implementation/src/presentation/tui/bridges/health-bridge.js` — reads `worker-health.json` for the active config dir and returns per-candidate health entries.

Out of scope:

- Editing routing in-place. That stays editor-driven.
- Reset / probe actions on health entries. Those land in 165 (Health scene). Workers shows status; Health acts on it.

- Opening Workers on a freshly initialized empty config (`{}`) renders only the "Pool" section header with a hint that no defaults are configured.

- `[e]` opens the local config in `$EDITOR`, blocks while editing, and re-renders the scene with the updated values on return.
- An invalid JSON edit surfaces the validator error in a banner; the scene does not crash.
- Phase routing summary correctly reports attempt rules count for `repair` and `resolveRepair`.
