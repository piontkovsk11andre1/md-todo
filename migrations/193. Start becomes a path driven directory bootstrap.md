# 193. Start Becomes A Path-Driven Directory Bootstrap

## Problem

The current `start` command is trying to carry an old freeform description concept while the desired product direction is path-oriented onboarding.

The new goal is simpler:

- `start` should bootstrap/adopt a design directory,
- it should optionally place rundown control state in an outer workdir,
- and after setup it should hand off to the normal migrate flow.

At the same time, the command should avoid surprising in-place reorganization of existing non-empty directories.

Additional context from the current implementation:

- The CLI surface still requires `rundown start <description>` and routes through `createStartCommandAction` into `startProject`.
- `startProject` currently initializes `.rundown`, persists workspace config/mount metadata, scaffolds bucket directories, and creates a bootstrap commit (`rundown: start project`), but does not hand off to `migrate`.
- Existing mounted/adopted directory behavior already exists through `--mount` and `--from-design`, and should be reused rather than duplicated.

## Desired behavior

Recommended public surface:

- `rundown start`
- `rundown start <design-dir>`
- `rundown start <design-dir> <workdir>`

Interpretation:

- no args means local bootstrap in the current directory when safe,
- one positional directory means adopt that directory as design input,
- two positional directories mean use the first as design input and the second as the controlling rundown workspace.

The freeform description argument should be removed entirely.

Additional interpretation details for deterministic behavior:

- `<design-dir>` and `<workdir>` are directory paths (relative or absolute), never freeform text.
- Relative paths resolve from invocation directory.
- `start` remains responsible for workspace bootstrap/adoption only; planning/migration remains owned by `migrate`.

## Behavior contract

### 1. `start` is directory-oriented, not file-oriented

This redesign should not add `start <file>`.

Rationale:

- design revisions are directory-based,
- `design/current` is directory-shaped,
- and single-file onboarding would require extra satellite-directory semantics that are not necessary for the current product direction.

Compatibility and validation notes:

- If a provided positional path resolves to an existing file, fail with a clear path-type error.
- If a provided path does not exist, creation behavior follows directory semantics only (no file bootstrap path).

### 2. Empty directories may be initialized locally without an outer workdir

If the selected local target directory is empty, `start` may set up rundown in place.

That keeps greenfield and safe local bootstrap straightforward.

Operational safety definition:

- Empty means no existing user content in the selected local design directory.
- Safe local bootstrap is intended for greenfield directories and should not silently repartition non-empty working trees.

### 3. Non-empty directories require an explicit outer workdir

If the selected local design directory already has content and no outer workdir is provided, `start` should fail with a clear error.

The command should not silently reorganize an existing working directory.

The explicit adoption form is:

- `rundown start . ..\myproject-rundown`

That means:

- use the current directory as the design source,
- place rundown control/work buckets in `..\myproject-rundown`.

Behavioral guarantees:

- No hidden move/rename of existing design files.
- No implicit in-place mounting of non-empty project roots without explicit outer workdir intent.

### 4. Existing `--from-design` behavior should be reused where possible

The implementation should lean on the current directory-based design adoption model rather than inventing a second one.

`start <design-dir>` and `start <design-dir> <workdir>` should mostly be a cleaner positional surface over the same underlying capabilities:

- directory-based design adoption,
- mounted/linked workspace setup,
- and normal migrate handoff afterward.

Implementation note from current code:

- `startProject` already supports design adoption through `fromDesign` and persisted `workspace.design.currentPath`.
- Existing mount routing in `workspace-paths.ts` already treats resolved absolute mount paths as authoritative and supports nested logical mounts.

### 5. `start` hands off to migrate; it does not own migration logic itself

After bootstrap or adoption succeeds, `start` should invoke the normal migrate flow.

Thread-aware drafting, source modes, revision release, and migration creation remain owned by `migrate`.

`start` is only the onboarding shortcut.

Integration boundary:

- `start` should call the existing migrate application entrypoint after successful setup, not duplicate migration planner/revision-sync logic in `start-project.ts`.

### 6. `migrate <path>` is not introduced

Focused path-based onboarding belongs to `start`, not to a scoped migrate command.

This redesign should keep `migrate` workspace-level.

## Scope

In scope:

- remove the old freeform description argument from `start`,
- make `start` directory- and path-driven,
- define empty-versus-non-empty directory safety rules,
- define the optional outer workdir positional,
- and keep `start` as a handoff into migrate.

Out of scope:

- adding single-file start support,
- adding `migrate <path>`,
- or redesigning migration/prediction internals.

Additional in-scope clarifications:

- update CLI parsing and help text to positional directory semantics,
- preserve existing workspace mount/link infrastructure and config schema compatibility,
- update docs and TUI start copy to match path-first onboarding language.

## Existing implementation context

### CLI surface and parsing

Current `start` command in `implementation/src/presentation/cli.ts`:

- uses `.argument("<description>", "Project description")`,
- supports legacy/compatibility options (`--dir`, `--mount`, directory and placement overrides, `--from-design`),
- includes help examples centered on description-based calls.

Current action in `implementation/src/presentation/cli-command-actions.ts`:

- `createStartCommandAction` forwards `description`, `dir`, mounts, and legacy directory/placement/adoption options to `app.startProject(...)`.

This means positional parsing is the primary change point for public syntax.

### Application workflow (`start-project.ts`)

Current `createStartProject` behavior:

- resolves target directory from `--dir` or invocation cwd,
- ensures git repo exists,
- runs `initProject` to seed `.rundown`,
- persists workspace directories/placement/mount config,
- writes workspace link metadata for linked/satellite flows,
- scaffolds local bucket directories where paths resolve local,
- creates bootstrap commit (`rundown: start project`).

Important current gap relative to desired behavior:

- no automatic handoff into `migrate` flow yet.

### Workspace routing and compatibility

Current routing model in `workspace-paths.ts` already provides:

- logical roots (`design`, `implementation`, `specs`, `migrations`, `prediction`),
- legacy placement compatibility (`sourcedir`/`workdir`),
- explicit mount map support with normalized logical paths,
- nested mount-prefix override behavior,
- canonical absolute path resolution for runtime workspace variables.

This is the right foundation for positional `start` adoption behavior.

### TUI integration

Current TUI start action (`implementation/src/presentation/tui/index.ts`) still calls `app.startProject(...)` with a fixed description constant (`"Initialize workspace"`). That coupling will need to align with the description removal and path-driven semantics so root-TUI bootstrap behavior stays coherent with CLI behavior.

## Implementation plan

Primary implementation shape:

1. Replace the old description positional with directory path positionals.
2. Interpret the first positional as `design-dir` when present.
3. Interpret the second positional as explicit `workdir` when present.
4. Add a safety check: empty local directories may initialize in place; non-empty directories must provide an outer workdir.
5. Reuse existing start/adoption/mount machinery under the new surface.
6. After bootstrap/adoption completes, hand off into the normal migrate flow.
7. Update TUI/help/examples to reflect the new path-oriented start semantics.

Expanded technical steps:

1. **CLI contract update**
   - Change `start` command signature from `<description>` to `[design-dir] [workdir]`.
   - Remove description references from docs/help and argument parsing.
2. **Action mapping**
   - Map positional `design-dir` to existing adoption primitive (`fromDesign` or equivalent internal resolved path input).
   - Map positional `workdir` to controlling workspace directory target (`dir` equivalent).
3. **Safety gate**
   - Before in-place local setup, inspect selected design directory contents.
   - If non-empty and no explicit outer workdir, fail with actionable guidance.
4. **Adoption/bootstrap execution**
   - Continue using existing workspace config/mount/link persistence paths where possible.
5. **Migrate handoff**
   - On successful start setup, invoke existing migrate application flow with normal worker resolution and workspace context.
6. **Presentation alignment**
   - Update CLI help text, root docs, examples, and TUI phrasing to path-first onboarding.

## Error handling and UX contract

Required failure guidance:

- Non-empty local adoption without outer workdir must fail with a message that includes the explicit safe form:
  - `rundown start . ..\myproject-rundown`
- File path passed where directory is required must fail with a path-type error.
- Ambiguous/invalid positional combinations should fail fast and preserve workspace state.

Message quality goals:

- name the offending path,
- explain why it is blocked (non-empty local directory without explicit outer workdir),
- provide one working command pattern.

## Acceptance criteria

- The old freeform description argument for `start` is removed.
- `rundown start` remains valid for safe local bootstrap.
- `rundown start <design-dir>` is valid when the selected local directory is empty or otherwise safe for in-place setup.
- `rundown start <design-dir>` fails with clear guidance when that directory is non-empty and no outer workdir was provided.
- `rundown start <design-dir> <workdir>` adopts the first directory as design input and places rundown control/work buckets in the second directory.
- `start` remains a bootstrap/adoption shortcut and hands off to the normal migrate flow rather than implementing its own separate migration planner.
- No single-file `start <file>` support is introduced.

Expanded validation criteria:

- `rundown start`/`start <design-dir>`/`start <design-dir> <workdir>` help text and docs are aligned.
- TUI start behavior does not rely on removed description positional.
- Existing mount/link compatibility and resolved workspace path semantics remain intact for downstream commands.
- Failure modes are non-destructive: blocked unsafe in-place adoption does not modify existing non-empty design directories.

## Verification outline

Recommended verification scenarios for planning and implementation review:

1. **Greenfield local**
   - Empty cwd, run `rundown start`.
   - Expect successful bootstrap + migrate handoff.
2. **Explicit design dir empty**
   - `rundown start ./design-input` where dir exists and empty.
   - Expect safe in-place bootstrap.
3. **Non-empty design dir, no workdir**
   - `rundown start .` in an existing project root with files.
   - Expect fail with explicit outer-workdir guidance.
4. **Non-empty design dir with workdir**
   - `rundown start . ../control`.
   - Expect adoption success and control state in `../control`.
5. **Path-type guard**
   - `rundown start ./README.md`.
   - Expect directory-required error.
6. **Compatibility sanity**
   - Confirm subsequent `rundown migrate` behavior is unchanged and workspace path variables remain authoritative absolute targets.
