# 197. Predict Stores Latest Head And Thread-Aware Snapshots

## Problem

Today `rundown predict` treats `prediction/` as a single mutable tree.

That gives us only one visible future state: whatever the latest successful prediction pass produced. The exact migration-level progress is stored in `.rundown/prediction-progress.json`, but the predicted filesystem state after migration `N` is not preserved as a first-class artifact.

This creates a few concrete problems:

- operators cannot inspect or diff the predicted state at an earlier migration boundary without replaying work,
- predict has no directory contract for historical snapshots,
- and downstream tooling has no stable place to find saved historical prediction state.

We want prediction to expose both:

- a mutable head for the latest future state,
- and immutable snapshot directories that mirror migration lane structure.

## Desired behavior

`rundown predict` should manage prediction as a two-level structure:

- `prediction/latest/` - authoritative latest predicted tree.
- `prediction/snapshots/root/<N>/` - full copy of `prediction/latest/` after prediction has successfully advanced the root lane through migration number `N`.
- `prediction/snapshots/threads/<thread>/<N>/` - full copy of `prediction/latest/` after prediction has successfully advanced thread lane `<thread>` through migration number `N`.

Interpretation details:

- `prediction/latest/` is the only mutable prediction tree.
- Each lane-specific snapshot directory is an immutable historical checkpoint from the operator's perspective. The runtime may delete and recreate snapshots during rebuilds, but normal forward prediction does not mutate older snapshots.
- `root` and `threads/<thread>/` mirror the same lane split used by migration history.
- The directory name `<N>` is the applied migration number within that lane, while the lane path carries the root-versus-thread identity.
- If a lane contains multiple files with the same migration number `N` for review or supplementary shapes, snapshot creation follows the same lane-number boundary: write or refresh that lane's `N` snapshot only after prediction has completed the lane's `N` batch.
- If no migrations have been predicted yet, `prediction/latest/` may be absent or empty and `prediction/snapshots/` has no numbered children.

## Behavior contract

### 1. `prediction/latest/` becomes the predict head

`prediction/latest/` is the mutable output of `predict`.

This migration only defines how `predict` saves its output. It does not require changing other commands in the same slice.

The root `prediction/` directory becomes a container for prediction artifacts rather than the head itself.

### 2. `prediction/snapshots/root/<N>/` and `prediction/snapshots/threads/<thread>/<N>/` store full-tree snapshots

After prediction successfully finishes the work that advances a lane boundary, the runtime must create or refresh the matching lane snapshot as a full copy of `prediction/latest/`.

Examples:

- after root migration `2`, write `prediction/snapshots/root/2/`,
- after thread `checkout` migration `4`, write `prediction/snapshots/threads/checkout/4/`.

Snapshot semantics:

- the snapshot is a whole-tree snapshot, not a delta,
- file paths inside each snapshot mirror the relative layout of `prediction/latest/`,
- files removed from the latest prediction must also be absent from the snapshot,
- snapshots are written only after the relevant prediction pass has already verified successfully,
- and the same `prediction/latest/` contents may be copied into more than one snapshot path over time as different lanes advance.

### 3. Snapshot persistence follows successful predict progress

Prediction progress is already driven by path-based progress plus migration content hashes.

This migration keeps snapshot persistence intentionally narrow:

- after each successful predicted lane boundary, save a full-tree snapshot for that boundary,
- do not write or refresh a snapshot for a failed prediction pass,
- and keep `.rundown/prediction-progress.json` as the source of truth for exact applied migration identity.

How stale snapshots are cleaned up or rebuilt can be introduced in a follow-up migration. This slice is only about saving snapshots during successful `predict` execution.

### 4. Progress state remains separate from snapshot directories

`.rundown/prediction-progress.json` remains the authoritative machine-readable progress record.

The new snapshot directories do not replace progress tracking. They complement it.

Progress should continue to store exact migration identity and invalidation inputs such as:

- migration identifier,
- migration number,
- migration filename,
- and migration content hash.

Snapshot directories exist for durable filesystem history, not as the sole source of truth for which migration files have been applied.

### 5. Flat legacy `prediction/` trees need one compatibility path

Existing workspaces may already contain prediction files directly under `prediction/`.

This migration should define an upgrade path instead of requiring manual cleanup:

- if `prediction/latest/` does not exist but legacy files exist directly under `prediction/`, treat that flat tree as legacy latest-state input,
- on the next successful `predict`, rewrite the structure into the new layout,
- and after rewrite, `predict` should continue writing to `prediction/latest/` rather than back to the flat root.

The compatibility path is transitional only.

## Scope

In scope:

- introduce the `prediction/latest/` and thread-aware `prediction/snapshots/` layout,
- define snapshot write timing during successful `predict` execution,
- and define upgrade behavior from the existing flat `prediction/` tree.

Out of scope:

- changing `test future`, `migrate --from prediction`, or other prediction readers,
- adding rebuild cleanup or snapshot truncation rules,
- adding a new CLI for browsing or diffing prediction snapshots,
- optimizing snapshot storage via hard links, deduplication, or content-addressed blobs,
- changing migration numbering rules,
- or replacing `.rundown/prediction-progress.json` with snapshot discovery.

## Implementation plan

1. Add path-resolution helpers for prediction head and snapshot directories.
2. Update `predict` so migration execution writes into `prediction/latest/` rather than directly into `prediction/`.
3. After a successful prediction pass, materialize or refresh the appropriate lane snapshot under `prediction/snapshots/root/` or `prediction/snapshots/threads/<thread>/`.
4. Add compatibility handling for legacy flat `prediction/` roots during upgrade.
5. Update docs for the new predict output layout.

## Implementation boundaries

Files likely involved:

  - keep current per-migration progress contract,
  - and adjust only if additional snapshot metadata is needed.

  - expose canonical path(s) for `prediction/latest/` and `prediction/snapshots/`.

- docs
  - command docs that describe `predict` output layout.

- tests
  - `implementation/__tests__/application/predict-task.test.ts`
  - integration coverage for successful snapshot creation and legacy-layout upgrade.

## Acceptance criteria

- After predicting root migrations `1`, `2`, and thread `checkout` migration `1`, the workspace contains `prediction/latest/`, `prediction/snapshots/root/1/`, `prediction/snapshots/root/2/`, and `prediction/snapshots/threads/checkout/1/`, each representing the full predicted tree at that lane boundary.
- `prediction/latest/` always matches the newest successful predicted state.
- Root-lane snapshots and thread-lane snapshots mirror migration lane structure rather than sharing a flat numeric namespace.
- A workspace using the old flat `prediction/` layout upgrades on the next successful `predict` run without requiring manual file moves.

## Current implementation context (research baseline)

- `implementation/src/application/predict-task.ts` currently discovers migrations, replays from stale progress when needed, and writes progress after each successful migration pass, but it still treats `workspacePaths.prediction` as the direct mutable prediction root.
- The migration model already distinguishes root history from `migrations/threads/<thread>/` history, so prediction snapshots should mirror that lane split rather than collapsing all artifacts into one flat numeric namespace.
- This migration intentionally avoids broadening into reader behavior; it only changes how `predict` persists successful output.

## Edge cases

- If prediction fails during a root-lane migration `N`, `prediction/latest/` may contain partial work from that failed pass, but `prediction/snapshots/root/<N>/` must not be refreshed unless the pass completes successfully.
- If prediction fails during thread `checkout` migration `N`, `prediction/latest/` may contain partial work from that failed pass, but `prediction/snapshots/threads/checkout/<N>/` must not be refreshed unless the pass completes successfully.
- If a lane snapshot already exists from an earlier run, the rewrite should replace it so the snapshot is a clean full-tree copy rather than an overlay with stale files.
- Existing readers may continue to read the flat root until follow-up migrations move them to `prediction/latest/`; this migration does not require changing them yet.
