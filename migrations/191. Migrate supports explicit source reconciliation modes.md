# 191. Migrate Supports Explicit Source Reconciliation Modes

## Problem

The product now needs three closely related ways to create migrations:

- from design diffs,
- from implementation changes,
- and from prediction changes.

The existing `migrate` name still fits the high-level user intent better than `release`, but the current command shape only communicates the design-diff path. When the source of truth becomes implementation or prediction, the surface needs to say that explicitly without inventing awkward names like `release now` or `release future`.

The command should stay centered on "create migrations", while the source mode should explain where current design is being reconciled from.

## Desired behavior

`rundown migrate` remains the top-level orchestration command for creating migrations.

Recommended public surface:

- `rundown migrate`
- `rundown migrate --from implementation`
- `rundown migrate --from prediction`

Interpretation:

- no `--from` means the existing design-diff workflow,
- `--from implementation` means create migrations by reconciling current design against implementation changes,
- `--from prediction` means create migrations by reconciling current design against prediction changes.

This keeps one verb for one family of outcomes: create migrations in `migrations/`.

## Command contract

### 1. `migrate` stays the canonical migration-authoring entrypoint

The user-facing main path should remain:

1. update the relevant source of truth,
2. run `rundown migrate ...`,
3. let rundown create the required revision boundary and draft/promote migration files.

The surface should not split the primary authoring flow into different verbs for each source mode.

### 2. Source selection is explicit and source-oriented

The source distinction should be expressed as a dedicated option, not hidden in temporal language.

Supported values in this migration:

- `implementation`
- `prediction`

Default behavior remains the current design-diff mode when `--from` is omitted.

This keeps `migrate` readable while making source selection precise.

### 3. `migrate --from implementation` reconciles current design from implementation

This mode is for the case where implementation changed and the user wants that change to come back as migration history.

The implementation plan should:

- inspect implementation changes relative to the current design contract,
- produce an updated design boundary from those changes,
- release that boundary into the normal design revision model,
- and then reuse the normal migration drafting pipeline.

The outcome is still migration files in `migrations/`, not a direct design-only rewrite with no migration history.

### 4. `migrate --from prediction` reconciles current design from prediction

This mode is the same pattern, but with `prediction/` as the source of truth.

The implementation plan should:

- compare or interpret the current prediction state against design,
- derive the resulting design update,
- release that update into the normal design revision flow,
- and then create migrations from that revision boundary.

Again, the outcome is migration history, not just a silent design sync.

### 5. The default design-diff path remains unchanged

The current `migrate` behavior should remain the default and should not be weakened by the new modes.

The existing revision-aware planning flow still applies when `--from` is omitted:

- release current design revision if needed,
- select the next unplanned revision boundary,
- and create migrations from the released design diff.

### 6. Thread-aware drafting remains part of migrate in every source mode

Source mode should affect how the design boundary is derived, not how migration drafting lanes are owned.

Existing thread behavior should continue to apply after the source-specific reconciliation phase:

- thread discovery from `.rundown/threads/`,
- translated thread briefs,
- per-thread drafting lanes,
- staged promotion,
- and verification of drafted migrations.

### 7. Path scoping is not added to migrate in this redesign

`migrate <path>` is not part of this migration.

`migrate` remains workspace-level and source-mode-driven. Focused onboarding or adoption belongs to `start`, not to a path-scoped migrate surface.

## Scope

In scope:

- keep `migrate` as the authoring verb,
- add explicit `--from implementation|prediction` source modes,
- keep the no-flag design-diff mode as the default,
- route source-specific reconciliation back into the same revision-aware migration pipeline,
- and update help/docs/messages around the new source model.

Out of scope:

- changing `predict` execution semantics,
- adding rollback verbs,
- adding `migrate <path>` scoping,
- or redesigning thread mechanics.

## Implementation plan

Primary implementation shape:

1. Extend migrate CLI parsing with a constrained `--from` option.
2. Normalize the option into an explicit application-level source mode.
3. Keep the current design-diff path as the default branch.
4. Add an implementation-reconciliation branch that derives a design update from implementation.
5. Add a prediction-reconciliation branch that derives a design update from prediction.
6. After either reconciliation branch produces a released design boundary, hand off to the existing thread-aware migrate planner.
7. Keep the migration file generation, staging, verification, and promotion logic centralized in the current migrate engine.

## Acceptance criteria

- `rundown migrate` keeps the current design-diff behavior as the default migration-authoring flow.
- `rundown migrate --from implementation` is accepted by the CLI and routes into a distinct source-reconciliation branch before migration drafting.
- `rundown migrate --from prediction` is accepted by the CLI and routes into a distinct source-reconciliation branch before migration drafting.
- In both non-default source modes, the resulting workflow still creates migrations in `migrations/` through the normal migrate pipeline rather than bypassing migration history.
- Existing thread-aware drafting behavior remains part of migrate regardless of source mode.
- No `migrate <path>` surface is introduced as part of this redesign.

## Current implementation context (codebase baseline)

The current implementation already has a strong migration-authoring spine that should be reused rather than replaced.

### CLI surface and parsing today

- `implementation/src/presentation/cli.ts` defines `rundown migrate` with options like `--dir`, `--workspace`, `--confirm`, `--worker`, and `--slug-worker`.
- No `--from` option exists yet on `migrate`.
- `implementation/src/presentation/cli-command-actions.ts` routes `migrate` through `createMigrateCommandAction(...)`.
- That action rejects positional action tokens (for example, `migrate up`) by throwing `unknown action` when a positional string is present.
- The migrate handler currently forwards normalized options into `app.migrateTask(...)` with no source-mode field.

### Application entrypoint and option contract today

- `implementation/src/create-app.ts` wires `migrateTask` to `createMigrateTask(...)` as the application boundary.
- `implementation/src/application/migrate-task.ts` defines `MigrateTaskOptions` with fields like `dir`, `workspace`, `confirm`, worker settings, and artifact flags.
- `MigrateTaskOptions.action` is currently accepted at the type level but rejected at runtime if non-empty, preserving an actionless migrate surface.
- This makes `migrate` a single orchestration path already aligned with the desired command contract.

### Existing migrate engine behavior to preserve

`createMigrateTask(...)` and `runMigrateLoop(...)` already implement the revision-aware and thread-aware pipeline that new source modes should feed into:

1. Resolve effective workspace root and logical paths (including linked/mounted workspace resolution).
2. Resolve migrations directory and validate existence.
3. Run preflight revision release via `saveDesignRevisionSnapshot(...)`.
4. Select the next unplanned released revision via `findLowestUnplannedRevision(...)`.
5. Build design diff context with `prepareDesignRevisionDiffContext(...)`.
6. Discover and process thread lanes from `.rundown/threads/*.md`.
7. Draft staged migrations, verify/repair staged drafts, and only then promote into `migrations/`.
8. Mark the revision planned via `markRevisionPlanned(...)` with promoted migration metadata paths.

This is the central behavior to keep unchanged after source-mode reconciliation.

### Design/revision model dependencies

`implementation/src/application/design-context.ts` is the authoritative design revision API currently used by migrate:

- `saveDesignRevisionSnapshot(...)` handles immutable `design/revisions/rev.N` snapshots.
- `discoverDesignRevisionDirectories(...)` and metadata (`plannedAt`, `migrations`) control planning boundaries.
- `prepareDesignRevisionDiffContext(...)` builds comparison context for migration drafting.
- Legacy design paths are still supported as compatibility fallbacks when canonical managed paths are unavailable.

The new `--from` modes must integrate with this model, not bypass it.

### Workspace routing constraints that affect this migration

`implementation/src/application/workspace-paths.ts` resolves logical roots to authoritative absolute targets (`design`, `implementation`, `specs`, `migrations`, `prediction`) and validates conflicts.

For this redesign:

- Source reconciliation should read from resolved absolute workspace paths.
- Logic should not reconstruct locations from `workspaceDir + bucket` assumptions.
- Linked/mounted workspace behavior must remain consistent across all `migrate` source modes.

## Integration design for explicit source modes

### Proposed source-mode model

Introduce an explicit migrate source concept at the application boundary, with behavior:

- default (no flag): existing design-diff path.
- `implementation`: derive design update from implementation state.
- `prediction`: derive design update from prediction state.

A normalized internal mode (for example `"design" | "implementation" | "prediction"`) keeps CLI parsing concerns separate from application orchestration.

### Command flow after redesign

Keep one canonical flow:

1. Parse `migrate` options, including constrained `--from` values.
2. Normalize source mode.
3. Run source-specific reconciliation phase (or no-op default phase).
4. Release resulting design boundary into revision model.
5. Continue through existing `runMigrateLoop(...)` planning/drafting/promotion pipeline.

This preserves one verb for one outcome family: migration history in `migrations/`.

### Reconciliation phase boundaries

#### Default mode (`rundown migrate`)

- Keep `runDesignRevisionReleasePreflight(...)` and subsequent revision planning exactly as-is.
- No behavior change except plumbing of explicit default mode.

#### Implementation mode (`rundown migrate --from implementation`)

- Reconciliation phase should inspect implementation against current design contract.
- It should materialize resulting design updates into the managed design draft/revision flow.
- It should then release a design revision boundary and continue through normal migrate planning.
- It should not write migrations directly from implementation without revision release.

#### Prediction mode (`rundown migrate --from prediction`)

- Reconciliation phase should compare/interpret prediction state against current design.
- It should derive and release the resulting design update into revision history.
- It should then run the same migrate planner/drafter/promotion flow.
- It should not perform silent design sync that skips migration files.

## Edge cases and compatibility constraints

- Invalid `--from` values should fail fast with clear allowed values.
- Missing source root (implementation/prediction) in non-default mode should fail with actionable path-specific guidance.
- If source reconciliation yields no effective design boundary change, command should return a clear no-op/caught-up message and preserve success semantics.
- Thread lanes must remain source-agnostic post-reconciliation (same discovery, translation, staged promotion flow).
- No path-scoped `migrate <path>` variant should be introduced.
- Existing default no-flag behavior must remain stable for automation and existing users.

## Validation outline (acceptance-focused)

Functional acceptance should cover at least:

1. `rundown migrate` unchanged default branch and output semantics.
2. `rundown migrate --from implementation` accepted and routed through non-default reconciliation path before drafting.
3. `rundown migrate --from prediction` accepted and routed through non-default reconciliation path before drafting.
4. Both non-default modes still produce migration files in `migrations/` via normal planner pipeline.
5. Threaded migrate behavior still runs in every mode when `.rundown/threads/*.md` exists.
6. Linked/mounted workspace routing remains correct under all source modes.
7. No positional action surfaces or path-scoped migrate behavior are introduced.
