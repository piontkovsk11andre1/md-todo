# 189. Remove plan-prepend and plan-append guidance files

## Problem

Planning currently has a split customization model:

- `.rundown/plan.md` defines the main planner prompt,
- `.rundown/plan-prepend.md` provides advisory guidance for tasks that should appear early,
- and `.rundown/plan-append.md` provides advisory guidance for tasks that should appear late.

That split adds extra file loading, scaffolding, localization, template variables, and documentation surface for behavior that is fundamentally prompt authoring.

It also makes planner customization harder to reason about because the effective planning instructions are spread across multiple files instead of being owned directly by the plan template the user edits.

The target direction is simpler: planning behavior should be driven by `plan.md` itself rather than by separate prepend/append guidance files.

## Desired behavior

After this migration, planner ordering and coverage guidance should live directly inside the relevant planner templates.

The intended contract is:

- top-level planning behavior is authored in `.rundown/plan.md`,
- loop-specific planning behavior is authored in `.rundown/plan-loop.md`,
- deep planning behavior is authored in `.rundown/deep-plan.md`,
- and there are no special-purpose `.rundown/plan-prepend.md` or `.rundown/plan-append.md` files.

If a project wants the planner to prefer discovery work early or validation work late, that instruction should be written directly into the template text that drives that planning mode.

This keeps prompt behavior explicit and local instead of relying on an extra guidance channel.

## Behavior contract

### 1. Remove the dedicated prepend/append guidance channel

The runtime should stop treating `plan-prepend.md` and `plan-append.md` as planner inputs.

That means:

- no reading those files during planning,
- no fallback defaults for those files,
- no special template variables such as `planPrependGuidance` or `planAppendGuidance`,
- and no implicit propagation of prepend/append guidance into deep planning.

Planner behavior should come from the active plan template content itself.

### 2. `plan.md` owns top-level planning guidance directly

Any instruction about what should usually appear near the beginning or end of a plan should be written directly inside `.rundown/plan.md`.

Examples of guidance that should now live inline in `plan.md` rather than in sidecar files:

- when to add early discovery or inspection tasks,
- when to add setup or prerequisite checks,
- when to add final verification tasks,
- and when to skip those patterns.

The planner should not need a separate advisory payload to understand those preferences.

### 3. Other planning modes follow the same principle

If loop planning or deep planning needs different guidance, that guidance should be expressed in the template that actually drives that mode.

In practice:

- `.rundown/plan-loop.md` should carry its own wording for loop planning,
- `.rundown/deep-plan.md` should carry its own wording for child-task expansion,
- and no hidden cross-template guidance channel should be preserved just to share prepend/append intent.

This keeps each planning mode self-describing.

### 4. Existing sidecar files become unsupported inputs

Repositories may already contain `.rundown/plan-prepend.md` and `.rundown/plan-append.md`.

After this migration:

- those files must no longer affect planning behavior,
- planning should continue normally even if they remain on disk,
- and the user guidance should tell teams to move any useful content into `.rundown/plan.md` or the relevant sibling template.

Automatic merging of those files into `plan.md` is out of scope. Their contents are freeform prompt text, so there is no safe deterministic way to splice them into existing templates without changing meaning.

If the product wants migration help, the acceptable approach is a concise deprecation warning when those files are present, not implicit prompt rewriting.

### 5. Planning safety rules remain unchanged

This migration is about prompt ownership and template resolution. It must not change planner edit safety.

The following behavior must remain as-is:

- add-only planning validation,
- checkbox-state protection,
- convergence and scan-loop semantics,
- and post-processing rules for plan edits.

Removing prepend/append sidecars should simplify prompt wiring, not alter what file edits are permitted.

## Scope

In scope:

- remove planner loading and wiring for `plan-prepend.md` and `plan-append.md`,
- move any default ordering/coverage guidance into the default plan templates themselves,
- stop scaffolding/localizing/documenting those sidecar files as active planner inputs,
- and document the new rule that planning guidance belongs in `plan.md`, `plan-loop.md`, and `deep-plan.md` directly.

Out of scope:

- automatic migration of user-authored sidecar file contents into template files,
- changing plan convergence or edit validation behavior,
- introducing a new replacement guidance file format,
- or adding a separate planner metadata system.

## Implementation boundaries

Primary code and docs surfaces likely involved:

- `implementation/src/application/plan-task.ts`
- `implementation/src/application/project-templates.ts`
- `implementation/src/application/init-project.ts`
- `implementation/src/application/localize-project.ts`
- `implementation/src/domain/defaults.ts`
- template variable typing and render surfaces that currently carry prepend/append guidance
- `implementation/docs/templates.md`
- tests covering planner prompt construction, project template loading, initialization, and localization

Expected implementation shape:

- remove optional sidecar guidance resolution from planning orchestration,
- delete the dedicated guidance template variables from planner prompt rendering,
- fold any default advisory wording into the main built-in plan templates,
- stop creating `plan-prepend.md` and `plan-append.md` in project bootstrap/localization flows,
- and update documentation/examples to point users at `plan.md` itself.

## Documentation and migration notes

Docs should clearly state that planner customization is now template-driven rather than sidecar-driven.

That documentation should explain:

- where users should place top-level planning guidance,
- where loop and deep planning guidance should live,
- that old `plan-prepend.md` and `plan-append.md` files are no longer consulted,
- and that teams should manually move any retained guidance into the appropriate template.

Examples should show inline guidance blocks inside `.rundown/plan.md` rather than separate prepend/append files.

## Acceptance criteria

- Planning no longer reads `plan-prepend.md` or `plan-append.md` from the config directory.
- Planner prompt construction no longer depends on prepend/append-specific template variables.
- The default built-in planning prompts still express any desired ordering/coverage guidance directly in their own template text.
- `init` and localization flows stop generating or exposing `plan-prepend.md` and `plan-append.md` as supported planner files.
- Existing repositories continue to plan successfully even if old sidecar files remain on disk, but those files no longer influence planner behavior.
- Planning documentation shows `plan.md` as the place to encode planner guidance instead of pointing users at prepend/append sidecars.
