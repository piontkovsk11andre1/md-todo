# 176. Start Scaffolds Implementation Directory

## Problem

`rundown start` currently creates the rundown starter surface without an `implementation/` directory.

Today the default workspace buckets are:

- `design/`
- `specs/`
- `migrations/`
- `prediction/`

That leaves the initialized project missing the most obvious source bucket. A freshly started workspace should show `implementation/` alongside the other starter directories so the project shape is complete and explicit from the start.

This is especially confusing because the rest of the model already talks about current implementation, predicted implementation, and migrations between them, but the starter scaffold does not materialize an `implementation/` home.

## Desired behavior

After this migration, `rundown start` creates `implementation/` alongside the other starter directories whenever it initializes a local project scaffold.

Expected starter directories after a normal local Start:

- `design/`
- `implementation/`
- `specs/`
- `migrations/`
- `prediction/`

`implementation/` should be treated as a first-class workspace bucket, not a special-case directory created only by Start.

## Behavior contract

### 1. `implementation/` is part of the default workspace layout

The workspace directory model and persisted `.rundown/config.json` should include `implementation` alongside the existing buckets.

That means:

- default workspace directory names include `implementation: "implementation"`
- workspace placement config includes `implementation`
- path resolution helpers understand and validate the new bucket
- directory-conflict validation treats `implementation` like the other managed directories

If rundown manages local workspace directories, `implementation/` must exist after Start.

### 2. Start materializes `implementation/`

When `rundown start` creates the starter scaffold, it must create `implementation/` the same way it creates the other starter directories.

- create it when missing
- leave it untouched when it already exists
- do not require migrations or bootstrap inference for it to appear

This should apply regardless of whether the target directory is otherwise empty or already contains files.

### 3. Clean-start semantics still apply

This migration does not reintroduce seeded migrations or other bootstrap content.

- Start still creates a clean project scaffold
- `migrations/` may remain empty per migration 175
- adding `implementation/` is a layout fix, not a behavior reversal on migration seeding

## Scope

In scope:

- Add `implementation` to the managed workspace directory model.
- Persist the new bucket in generated config.
- Make Start create `implementation/` alongside the other starter directories.
- Add regression coverage for Start in empty and non-empty directories.

Out of scope:

- Bootstrapping implementation contents.
- Moving existing source files into `implementation/` automatically.
- Inferring implementation roots from linked workspaces.

## Implementation boundaries

Primary files likely involved:

- `implementation/src/application/workspace-paths.ts`
- `implementation/src/application/start-project.ts`
- any config serialization helpers used by Start
- tests covering workspace directory defaults and Start scaffolding

The root change should be made in the workspace bucket model first, then consumed by Start.
