# 196. Simplify Discuss Into A File Discussion Session

## Problem

`discuss` currently exposes two different mental models:

- `rundown discuss <source>` for discussing the next unchecked task from source.
- `rundown discuss --run <id|latest>` for discussing one saved finished run.

That split makes the feature more complicated than the user intent.

From the user's point of view, the intent is simple: open a TUI session to discuss a file and see the relevant artifact history for that file.

The current design leaks implementation detail into the interface:

- there are effectively two prompt modes,
- artifact context is asymmetric across those modes,
- the user has to understand run targeting,
- and the `discuss-finished` concept exists because the system is organized around runs rather than around file discussion.

This is over-specified for the job the user is trying to do.

## Desired Behavior

After this migration, `discuss` should have one user-facing form:

`rundown discuss <file.md>`

That command opens a TUI session to discuss the file.

The rendered prompt should be file-first and simple:

- identify the file being discussed,
- include the current source snapshot,
- include a single list of related saved artifacts for that file,
- and let the worker inspect whichever artifacts are relevant during the conversation.

There should be no separate user-facing finished-run discuss mode.

There should be no requirement to preselect a run.

The prompt should not try to be clever about one target run versus another. It should show the file and a plain related-artifacts list, then let the discussion use that information naturally.

## Behavior Contract

### 1. `discuss` becomes file-oriented

`rundown discuss <file.md>` discusses the file itself, not the next unchecked task and not one preselected run.

The file path is the primary input.

The command should work even when the file has no unchecked tasks, because the discussion target is the file, not only the next runnable task.

### 2. There is one prompt shape

The product should use one discuss prompt model for file discussion.

That prompt should contain:

- file path,
- current source snapshot,
- any useful lightweight task context derived from the file,
- and one related-artifacts list for the file.

The prompt should not branch into a separate "finished" discussion mode.

### 3. Related artifacts are shown as one plain list

The prompt should show all related saved artifacts for the discussed file in one simple list.

The list should be file-based, not target-run-based.

An artifact is related when its persisted task or source metadata resolves to the discussed file.

The list should remain lightweight. It should prefer a compact row or bullet format with fields such as:

- run id,
- originating command,
- status,
- timestamps when available,
- artifact directory path,
- and a short note or label when useful.

The prompt should not embed deep phase summaries for every artifact by default.

If the worker wants execution, verification, repair, reverify, or previous discuss details, it can read the corresponding artifact files during the session.

### 4. `--run` is removed from the user-facing model

`rundown discuss --run <id|latest>` should no longer be the primary interaction model.

The product should guide users toward file discussion instead of run targeting.

If compatibility needs to be preserved temporarily, the old option may be translated internally or deprecated with a clear redirect to `rundown discuss <file.md>`, but the intended design is a single file-oriented command.

### 5. `discuss-finished` stops being a distinct product concept

The separate `discuss-finished` concept should disappear from the user-facing behavior.

If some internal artifact statuses or trace event names need to remain temporarily for compatibility, they should be treated as implementation detail only.

The user-facing feature is simply `discuss a file`.

## Simplicity Rules

This migration intentionally prefers a simpler model over a more specialized one.

- Do not require the user to choose a run before discussion starts.
- Do not introduce separate prompt templates for source discussion versus finished-run discussion unless there is a hard technical blocker.
- Do not make the initial prompt perform heavy artifact summarization.
- Do not make the file discussion dependent on unchecked-task selection.
- Do keep the artifact list visible and easy to scan.

## Scope

In scope:

- reduce `discuss` to one file-oriented workflow,
- make `<file.md>` the primary and sufficient input,
- show a single related-artifacts list for that file,
- remove or de-emphasize `--run` as a product surface,
- collapse prompt behavior into one file discussion contract,
- and simplify documentation/help around discuss.

Out of scope:

- changing how `run`, `verify`, `repair`, or `reverify` generate artifacts,
- changing artifact retention policy,
- changing worker TUI mechanics,
- or adding sophisticated artifact ranking heuristics.

## Implementation Shape

The intended implementation direction is:

1. Resolve `discuss` from an explicit file path rather than from next-task selection.
2. Load the file source and derive any lightweight task context directly from the file.
3. Collect all saved artifacts whose persisted metadata maps back to that file.
4. Render one discuss prompt with the source snapshot and a compact related-artifacts list.
5. Let the worker inspect specific artifact directories as needed during the TUI session.
6. Remove the separate finished-run prompt path from the product design.

## Implementation Boundaries

Primary areas likely involved:

- `implementation/src/application/discuss-task.ts`
- `implementation/src/domain/defaults.ts`
- `implementation/src/presentation/cli.ts`
- `implementation/docs/cli-discuss.md`
- prompt/template loading for discuss
- tests covering discuss CLI behavior and prompt rendering

Areas that should simplify or disappear:

- explicit user-facing dependence on `--run` for discuss,
- the separate `discuss-finished.md` product role,
- special-case prompt branching whose only purpose is finished-run discussion.

## Acceptance Criteria

- `rundown discuss <file.md>` opens a TUI session for discussing the file.
- The prompt includes the current file content and one related-artifacts list for that file.
- Related artifacts are discovered by file association, not by a selected target run.
- Users do not need `--run` to get artifact-aware discussion.
- The discuss experience is described everywhere as file-oriented, not split into source-mode and finished-run mode.
- The product no longer presents `discuss-finished` as a distinct user-facing concept.

## Notes

This migration is intentionally a simplification migration.

The goal is not to make discuss smarter by adding more branches.

The goal is to make discuss easier to understand:

- one command,
- one file-focused prompt,
- one list of related artifacts,
- one TUI discussion flow.
