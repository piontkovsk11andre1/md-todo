# 178. Design Revisions Live Under design/revisions

## Problem

The canonical design workspace currently mixes mutable draft state and immutable released revisions as siblings directly under `design/`:

- `design/current/`
- `design/rev.N/`

That layout works, but it makes the design root noisier over time and blurs the distinction between the active draft and the released revision archive. As projects accumulate snapshots, `design/` becomes a mixed bucket containing both the current working tree and every historical revision directory.

The desired shape is clearer:

- `design/current/` stays the mutable working draft
- `design/revisions/rev.N/` becomes the canonical home for immutable released revisions

This change affects real runtime behavior, not just naming. The current revision save/discovery/diff logic and user-facing docs still assume the flat `design/rev.N/` layout.

## Desired behavior

After this migration, the canonical managed design layout is:

- `design/current/`
- `design/revisions/rev.N/`

Revision lifecycle commands must treat `design/revisions/` as the canonical revision root for:

- discovering released revisions
- saving the next immutable snapshot
- resolving metadata sidecars
- computing revision diffs and migrate context
- describing the feature in CLI help, docs, prompts, and tests

Older workspaces that still store snapshots directly under `design/rev.N/` must remain readable as a compatibility path so existing projects do not break immediately.

## Behavior contract

### 1. Canonical revision root moves to `design/revisions/`

The managed design workspace keeps `design/current/` unchanged, but released snapshots are no longer stored directly under the design root.

Canonical structure becomes:

- `design/current/`
- `design/revisions/rev.0/`
- `design/revisions/rev.1/`
- ...

This applies to both revision directory discovery and creation of new snapshots.

If the workspace uses a configured design bucket other than the default name, the same structure applies within that configured design root:

- `<design>/current/`
- `<design>/revisions/rev.N/`

### 2. Revision save and diff flows use the new archive root

`rundown design release` must write the next immutable snapshot into `<design>/revisions/rev.N/`.

`rundown design diff` and migrate revision context must enumerate released revisions from the canonical revisions root, not from the top level of `<design>/`.

That means the owning design revision helpers should stop treating the design bucket root itself as the revision archive root. They should distinguish between:

- the design workspace root (`design/`)
- the current draft directory (`design/current/`)
- the revisions archive root (`design/revisions/`)

### 3. Metadata sidecars move with the revision archive root

Revision metadata sidecars should live alongside the canonical revision archive entries under the revisions root.

Examples:

- `design/revisions/rev.3/`
- `design/revisions/rev.3.meta.json`

Metadata lookup, creation, and planned-state updates must all resolve against the canonical revisions root.

The metadata naming scheme does not need to change; only the containing directory changes.

### 4. Existing flat `design/rev.N/` workspaces remain readable as compatibility-only fallback

This migration changes the canonical write path, but it should not strand older repositories.

Compatibility requirements:

- existing `design/rev.N/` directories remain discoverable for read/diff/migrate flows when canonical `design/revisions/` entries are absent
- existing `design/rev.N.meta.json` sidecars remain readable for those legacy revision directories
- legacy `docs/current/` and `docs/rev.N/` compatibility behavior remains intact unless a more specific migration changes it later

Canonical writes should go to `design/revisions/` once the new layout is available. The fallback exists for compatibility and transition, not as a second canonical location.

### 5. External `design/current` overrides keep revisions inside the managed design root

When `workspace.design.currentPath` points to an external directory, that directory still acts only as `design/current`.

Released revisions must continue to live under the managed design bucket inside the workspace, now at:

- `<design>/revisions/rev.N/`

They must not be written into the external current-path directory.

### 6. User-facing docs and prompts describe `design/revisions/`

Help text, docs, prompt defaults, and tests must stop describing canonical revisions as `design/rev.N/` or `design/rev.*/**`.

Canonical wording should become:

- `design/current/**`
- `design/revisions/rev.*/**`

Legacy flat `design/rev.*/**` should only be mentioned as a compatibility-only fallback where that detail matters.

## Scope

In scope:

- Change canonical design revision discovery/save/metadata logic to use `<design>/revisions/`.
- Preserve compatibility reads for existing flat `design/rev.N/` layouts.
- Update CLI help, docs, prompt defaults, and tests that describe canonical revision paths.
- Add regression coverage for save, discovery, diff, and migrate context under the new layout.

Out of scope:

- Physically migrating existing user repositories from `design/rev.N/` to `design/revisions/rev.N/` automatically.
- Changing revision naming from `rev.N` to another scheme.
- Changing the contents of revision metadata beyond path relocation.
- Removing legacy `docs/` compatibility behavior.
