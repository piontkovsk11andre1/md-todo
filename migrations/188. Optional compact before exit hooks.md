# 188. Optional Compact Before Exit Hooks

## Problem

An explicit `rundown compact` command is necessary, but it is not sufficient for projects that want storage hygiene to happen as part of normal command execution.

Once archive-aware compaction exists for design revisions and migration payloads, users will still have to remember to invoke it manually after commands that grow those stores.

That is easy to forget in workflows such as:

- repeated `rundown design release` snapshots,
- repeated `rundown migrate` planning across many revisions,
- and broader lifecycle commands where teams want compact cleanup to run opportunistically at the end of successful commands.

The runtime currently has no shared post-command compaction hook, no per-command compact flag, and no config-driven way to opt into automatic compaction.

## Desired behavior

After this migration, supported CLI commands should be able to invoke compaction automatically at the end of a successful run through a shared post-command hook.

The target product behavior is:

- commands can opt into automatic compaction with an explicit CLI flag,
- the hook runs only after successful command completion,
- the hook reuses the same compact policy and archive-aware logic introduced by 187,
- automatic compaction is optional and disabled by default unless the user explicitly enables it by flag or config,
- and hook failures are reported clearly without silently mutating unrelated command semantics.

This migration is about orchestration and opt-in UX, not about redefining the archive storage contract itself.

## Behavior contract

### 1. Automatic compaction is a post-success hook, not pre-exit shutdown magic

The runtime should invoke automatic compaction from normal command completion flow after the owning command has already determined its result.

That means:

- no compaction on process shutdown handlers,
- no compaction during partial failure cleanup,
- no compaction when the command exits early in dry-run, print-only, or no-work modes unless explicitly defined otherwise,
- and no compaction before primary command artifacts or summaries are finalized.

The hook should behave like a deliberate follow-up action, not an implicit destructor.

### 2. Supported commands expose an explicit compact flag

Commands that opt into the hook should expose a consistent user-facing flag such as:

- `--compact-before-exit`

The initial goal is shared flag semantics across eligible lifecycle commands rather than one-off per-command naming.

The flag means:

- if the command completes successfully,
- run the compact use case with configured/default compact policy,
- and print a compact summary after the main command summary.

### 3. Automatic compaction reuses the same policy engine as `rundown compact`

The hook must not reimplement archive selection or movement rules locally inside each command.

Instead:

- the explicit `rundown compact` use case remains the single owner of candidate selection and archive execution,
- hook callers pass normalized compact options into that shared use case,
- and command-specific code only decides whether the hook should run.

This keeps explicit and automatic compaction behavior identical for the same policy inputs.

### 4. Automatic compaction is opt-in by flag or config and disabled by default

The runtime should not silently begin compacting history after unrelated commands unless the user explicitly requested that behavior.

Supported enablement mechanisms should be:

- per-invocation CLI flag,
- and optional config defaults for teams that want persistent behavior.

If config support is added, the default should still remain off until the user sets it.

### 5. Command outcome rules must be strict

The hook should run only when the primary command reaches a clear success state.

It should not run when:

- the primary command fails,
- verification/repair fails,
- the command returns no-work,
- the command is dry-run or print-prompt only,
- or the command is interrupted or cancelled.

The goal is to avoid compacting state after ambiguous or incomplete operations.

### 6. Hook failures are surfaced, but do not rewrite primary command semantics invisibly

If automatic compaction runs and fails, the user must see:

- that the primary command succeeded,
- that the follow-up compaction step failed,
- and what compact scope/policy was attempted.

The runtime should choose one consistent exit policy and document it clearly. The recommended contract is:

- primary command success remains visible,
- auto-compact failure is surfaced as a secondary failure summary,
- and the final exit code reflects failure because the requested follow-up action did not complete.

That policy is stricter, but it keeps scripting behavior honest when users explicitly asked for compaction.

### 7. Broad command availability does not imply broad storage scope

The user may want `--compact-before-exit` to exist on multiple commands, but the hook should still only target stores covered by the compact feature.

In the first automatic slice, that means:

- design revisions,
- root/thread migration payloads,
- and no `.rundown/runs/` compaction.

Finished-run `discuss` must remain compatible with its current direct artifact scan behavior because run artifacts are not being compacted here.

### 8. Shared output should be concise and deterministic

Hook output should summarize:

- whether compaction was skipped,
- whether it ran in dry-run or execute mode,
- targeted stores,
- retained counts/policy,
- and moved item counts.

That output should come after the primary command summary so users can distinguish command work from storage follow-up work.

## Scope

In scope:

- Add an optional `--compact-before-exit` style flag to eligible commands.
- Add shared post-success hook orchestration.
- Reuse the compact use case from 187 instead of duplicating policy logic.
- Optionally add config-backed defaults for auto-compact behavior.
- Add tests for flag parsing, hook invocation rules, and failure reporting.

Out of scope:

- Defining the archive storage contract itself.
- Compacting `.rundown/runs/`.
- Enabling auto-compact by default for all users.
- Running compaction during generic process shutdown or crash handling.
- Adding compression support.

## Implementation boundaries

Primary files likely involved:

- `implementation/src/presentation/cli.ts`
- `implementation/src/presentation/cli-command-actions.ts`
- `implementation/src/create-app.ts`
- command application entrypoints that should expose the hook, especially:
  - `implementation/src/application/migrate-task.ts`
  - design release entrypoints under the design lifecycle flow
  - any shared command orchestration helper chosen for broader lifecycle support
- config parsing and defaults plumbing if persistent auto-compact settings are added
- `implementation/__tests__/presentation/cli-command-actions.test.ts`
- command-focused application tests covering hook invocation conditions

Likely implementation shape:

- Add one normalized auto-compact option shape at the CLI/action boundary.
- Thread that option into a shared post-command helper near app creation or command orchestration glue.
- Invoke the compact use case only after a successful primary result.
- Keep the compact use case itself independent of the hook caller.
