Start supports mounted workspace bootstrap

## Intent

Extend `rundown start` so it can bootstrap a new rundown workspace elsewhere while attaching existing user directories to logical rundown paths through an explicit mount-oriented CLI surface.

This migration defines the start/bootstrap product behavior for mounted workspaces, including bare-control-repo scenarios.

## Problem

Today `rundown start` can:

- start in the current directory or `--dir <path>`
- scaffold managed local directories under the target workspace
- link invocation and target workspaces through `workspace.link`
- attach one special external directory as `design/current` via `--from-design`

It cannot express the broader onboarding shapes users want, such as:

- start a new rundown folder elsewhere and treat the current directory as `implementation`
- start elsewhere and treat the current directory as `specs`
- start elsewhere and treat the current directory as `implementation/something`
- start a bare rundown control directory and selectively map user directories into `design`, `implementation`, `specs`, or `migrations`
- attach only some subdirectories from the invocation tree while leaving other logical rundown paths local to the new workspace

That makes `start` too greenfield-oriented and prevents mounted workspace adoption from being a first-class bootstrap flow.

## Desired behavior

`rundown start` should support repeatable mount declarations as part of project bootstrap.

Recommended CLI surface:

- `--mount <logical-path=target-path>` repeatable

Examples:

```bash
rundown start "adopt existing app" --dir ../control --mount implementation=. -- opencode run
rundown start "adopt existing app" --dir ../control --mount specs=. -- opencode run
rundown start "adopt existing app" --dir ../control --mount implementation/generated=./generated -- opencode run
rundown start "adopt existing app" --dir ../control --mount implementation=. --mount migrations=../control/migrations -- opencode run
```

CLI contract:

- logical path is a canonical rundown path prefix
- target path may be absolute or relative
- relative target paths are resolved from the invocation directory for CLI ergonomics during adoption of an existing current directory
- normalized mount targets are persisted in workspace config so later commands resolve identically regardless of invocation location

## Bootstrap semantics

The new `start` flow should support both of these modes:

### 1. Bare control workspace

The target rundown directory may contain only `.rundown/` and any other control files that are truly local to rundown.

Managed content directories do not need to be created inside the target workspace when mounts route them elsewhere.

### 2. Mixed local + mounted workspace

Some logical paths may remain local to the new workspace while others are mounted into external user directories.

Example:

- `implementation -> .` (current directory)
- `specs -> ./.specs`
- `migrations -> <target workspace>/migrations`
- `design -> <target workspace>/design`

`start` should create only the local managed directories that are still owned by the target workspace.

## Linking and discoverability contract

When `start` bootstraps a workspace elsewhere but the invocation directory is mounted into that workspace, commands launched from the invocation directory still need to discover the satellite rundown workspace deterministically.

Required behavior:

- invocation directory receives or updates `.rundown/workspace.link` so commands can resolve the controlling workspace root
- target workspace receives compatible link metadata back to the invocation workspace when that behavior remains useful for current multi-workspace flows
- mounted bootstrap must not break existing `--workspace` selection semantics

This migration does not redefine `workspace.link`; it requires `start` to keep bootstrap discoverability intact for the new mounted layouts.

## Compatibility contract

Keep old bootstrap flags working during the transition:

- `--design-dir`, `--specs-dir`, `--migrations-dir`
- placement flags
- `--from-design`

Compatibility rules:

- old flags may normalize into mount records internally
- conflicting old/new CLI combinations must fail with actionable errors
- docs/help should clearly state which flags are legacy shorthands versus the new authoritative mount surface

## Scope

In scope:

- define the `start` CLI surface for mount declarations
- define normalization/validation rules for mount arguments
- adapt start scaffolding so only locally owned logical paths are created inside the target workspace
- preserve discoverability through `workspace.link` for mounted invocation directories
- persist normalized mount config to `.rundown/config.json`
- document supported adoption examples for current directory and subdirectory attachments

Out of scope:

- moving existing user files automatically
- inferring mounts without explicit CLI declarations
- implementing a separate interactive attach wizard

## Validation rules

Validate at bootstrap time:

- mount argument must be in `logical=target` form
- logical path must be a valid normalized rundown logical prefix
- target path must resolve to a directory path
- external targets that are expected to adopt existing user content must already exist
- conflicting mount declarations must fail early
- bootstrap must never silently reinterpret the invocation directory root as a mounted target without explicit CLI input

Scaffolding rules:

- if a logical path is mounted externally, `start` does not create that directory under the target workspace
- if a logical path remains local, `start` creates it when missing
- re-running `start` must not overwrite mounted external content
