Prompt and docs move from placement to resolved mounts

## Intent

Align worker prompts, template variables, and user documentation with the new mount-based workspace model so agents and users are shown the actual resolved routing contract rather than the old placement abstraction.

## Problem

Current runtime prompt context and docs expose:

- bucket directory names
- `workspace*Placement` values (`sourcedir` / `workdir`)
- derived absolute bucket paths

That was appropriate for the placement model, but it becomes misleading once runtime resolution is mount-based.

In a mounted workspace:

- a logical path may not belong to either `workspaceDir` or `invocationDir`
- nested logical prefixes may route to different targets
- `workspaceDesignPlacement=sourcedir` no longer fully explains where `design/current` lives

Agents need the resolved authoritative paths, not a stale explanation of how old placement math would have worked.

## Desired behavior

Prompts and docs should describe workspace routing in terms of resolved mounts.

Prompt/runtime principles:

- resolved absolute paths are authoritative
- agents must not reconstruct filesystem layout from older placement heuristics
- when useful, prompt context may expose which logical mount matched a path and whether the workspace is linked
- prompt/docs examples should include mounted workspace scenarios, not only local/placement scenarios

Recommended template surface direction:

- keep `invocationDir`, `workspaceDir`, `workspaceLinkPath`, and `isLinkedWorkspace`
- keep resolved absolute path variables for common logical roots (`workspaceDesignPath`, `workspaceImplementationPath`, etc.)
- add a structured workspace mount summary variable when needed for advanced/custom templates
- de-emphasize or deprecate `workspace*Placement` variables once compatibility strategy is in place

## Scope

In scope:

- update runtime workspace context to speak in terms of resolved mounts
- update default prompt templates so they instruct agents to use resolved paths as authoritative
- update template docs and CLI docs with mount-oriented explanations and examples
- document compatibility semantics for legacy placement variables if they remain temporarily available

Out of scope:

- redesigning the entire prompt system
- exposing every internal mount resolver detail when a concise resolved summary is enough

## Documentation contract

Update docs so they clearly distinguish:

- invocation directory
- effective workspace/control directory
- linked workspace discovery
- logical rundown paths
- resolved filesystem targets via mounts

Required documentation scenarios:

- local default workspace
- linked workspace
- bare control workspace with all major content mounted elsewhere
- nested mount override such as `design/current` or `implementation/generated`

## Prompt contract

Default prompts should state, in substance:

- workspace path variables are already resolved and authoritative
- do not recompute layout from `workspaceDir + directory name`
- if a workspace mount summary is present, treat it as the canonical routing map

Prompt examples and runtime metadata should include `implementation` consistently alongside `design`, `specs`, `migrations`, and `prediction` where those buckets remain part of the model.
