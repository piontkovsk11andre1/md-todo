# 182. Migrate Supports Thread Scoped Drafting

## Problem

`rundown migrate` currently plans and drafts migrations as one linear stream against the current design diff.

- It has no concept of optional migration threads declared by the project.
- It cannot specialize the same design diff for different thread-specific concerns.
- All migrate output still targets the single root `migrations/` directory.
- The staged drafting flow from 181 works for one shared migration lane, but not for multiple isolated thread lanes.

That prevents projects from breaking one design revision into separately guided migration tracks when they need different thread-specific interpretations of the same design change.

## Desired behavior

After this migration, `rundown migrate` should support optional thread-scoped drafting driven by `.rundown/threads/*.md`.

- If `.rundown/threads/` does not exist, or contains no markdown files, migrate behaves exactly as it does after 181.
- If one or more thread markdown files exist, migrate treats each one as an independent migration lane.
- For each thread, migrate translates the current design diff through that thread document before drafting migrations.
- The translated thread result becomes part of that thread's migrate prompt context.
- Each thread drafts, verifies, repairs, and promotes its own migrations independently.
- Thread migrations live under `migrations/threads/<thread-name>/` and use numbering local to that thread directory.

The result should be that a single released design revision can produce multiple thread-specific migration histories without forcing those histories into one shared numbering stream.

## Behavior contract

### 1. Thread mode is optional and auto-detected

Migrate should inspect `.rundown/threads/` before planning.

- Only `*.md` files in `.rundown/threads/` participate.
- If no such files exist, migrate must preserve the existing non-threaded behavior with the root `migrations/` directory.
- If thread files exist, migrate enters thread mode for that revision and plans each thread separately.

Thread mode is a project opt-in based on files on disk, not a new required CLI flag.

### 2. Each thread translates the design diff before drafting

For every discovered thread document, migrate should derive thread-specific prompt context by running the `translate` flow against the current design diff.

- The translate input should be the current unreconciled design diff for the target revision.
- The translate instruction should be the thread markdown file itself.
- The translated output should be written into the current run artifacts, not into `migrations/`.
- The thread translation result should then be embedded into that thread's migrate drafting prompt.

This translation step lets one design diff yield different migration plans for different thread concerns without rewriting the base design diff itself.

### 3. Thread prompts stay isolated

Each thread drafting run should only see:

- the shared design diff and design context,
- migration history relevant to that thread directory,
- the translated thread brief for that thread,
- and any shared migrate invariants that apply to all threads.

It must not merge migration history from sibling thread directories into the current thread prompt.

### 4. Thread migrations live in dedicated directories

When thread mode is active, migrate should promote files into thread-scoped directories.

- Each thread gets `migrations/threads/<thread-slug>/`.
- Drafted migration files inside that directory still use canonical filenames: `N. Title.md`.
- Numbering is determined from the highest migration number already present in that thread directory, not from the root `migrations/` directory and not from sibling threads.
- The app should create missing thread directories as needed.

The root `migrations/` directory remains the target only for non-threaded migrate runs.

### 5. Staged drafting and repair happen per thread

The staging model introduced in 181 should be preserved, but scoped per thread.

- Each thread gets its own staged drafting directory under the current run artifacts.
- Verification evaluates only the staged drafts for that thread against the shared design diff plus that thread's translated brief.
- Repair mutates only the staged drafts for that thread.
- Promotion for one thread must not require another thread to succeed first.

If one thread fails verification or exhausts repair, its staged artifacts must be preserved for inspection without forcing partial drafts into the real migrations tree for that thread.

### 6. Explore runs on promoted thread files only

After a thread's staged drafts pass verification and are promoted, `explore` should run only on the promoted migration files for that thread.

- Failed staged thread drafts must not be explored.
- A thread's explore run must target that thread's promoted files, not sibling thread files.

### 7. Revision planning tracks thread outputs explicitly

When migrate marks a revision as planned in thread mode, the recorded migration paths should reflect the promoted thread-scoped files.

- The recorded paths must preserve the `threads/<thread>/...` location so later tooling can distinguish thread histories.
- If no thread files were promoted for a given thread because drafting failed, that thread contributes no promoted files to planned state.

The revision should be considered planned only when the migrate run finishes under the app's existing success rules for the chosen threading model.

### 8. Thread lanes are isolated and parallel-safe by construction

The directory layout and prompt boundaries should make thread migration lanes independent.

- rundown does not need to guarantee actual parallel execution in this migration.
- However, it should avoid cross-thread shared numbering or shared staged draft directories.
- Nothing in the app contract should assume sibling threads must serialize because of shared mutable migration state.

This keeps the design compatible with future parallel execution even if the initial implementation processes threads sequentially.

## Scope

In scope:

- Detect optional thread files under `.rundown/threads/`.
- Translate the current design diff through each thread file before drafting.
- Feed translated thread briefs into migrate planning and drafting prompts.
- Create and validate thread-scoped staged draft directories.
- Promote thread migrations into `migrations/threads/<thread>/` with local numbering.
- Preserve the existing non-threaded root migration flow when no thread files exist.
- Record promoted thread file paths in revision planning state.

Out of scope:

- Guaranteeing concurrent execution of thread lanes.
- Defining cross-thread merge, dependency, or ordering semantics.
- Changing how non-threaded migrations are numbered.
- Allowing thread documents outside `.rundown/threads/*.md`.
- Reworking the general `translate` command UX beyond the migrate integration needed here.

## Implementation boundaries

Primary files likely involved:

- `implementation/src/application/migrate-task.ts`
- `implementation/src/application/translate-task.ts`
- `implementation/src/domain/defaults.ts`
- `implementation/src/domain/migration-parser.ts`
- `implementation/src/infrastructure/runtime-artifacts.ts`
- `implementation/__tests__/application/migrate-task.test.ts`
- `implementation/__tests__/integration/migrate-task.integration.test.ts`

Likely implementation shape:

- Add thread discovery for `.rundown/threads/*.md` near the migrate loop entry.
- Build a per-thread migration state reader rooted at `migrations/threads/<thread>/`.
- Reuse the translate pipeline, or a shared internal helper derived from it, to materialize thread briefs into run artifacts.
- Extend the migrate prompt template with thread-specific variables such as thread name, thread source path, and translated thread brief.
- Reuse the staged drafting, verification, repair, and promotion flow from 181 per thread lane.
- Keep the existing root migration path untouched when no thread documents are present.
