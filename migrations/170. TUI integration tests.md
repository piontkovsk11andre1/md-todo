# 170. TUI integration tests

## Problem

The TUI now spans nine scenes (main menu, Continue, New Work, Workers, Profiles, Settings, Help, Tools, Health) plus probes, bridges, and a navigation stack. The only test coverage so far is the per-scene snapshot from 160. Without a higher-level harness, regressions like "Esc from Health no longer returns to Workers" or "Settings scope cycle skips global" can only be caught by manual play-testing.

`rundown` already has structured `ApplicationOutputEvent` flowing through every command path (we wired the `progress` events for exactly this kind of consumption). The TUI is a deterministic function from input keys + emitted events to rendered frames. That makes it cheaply testable.

## Goal

Stand up an integration harness for the TUI that drives scenes through scripted key sequences and asserts on the rendered frame. Use it to cover one happy path per scene plus the cross-scene navigation graph.

## Model

Harness shape (under `__tests__/presentation/tui/harness.ts`):

```ts
interface HarnessOptions {
  workspaceFiles?: Record<string, string>;  // virtual fs seeded for the test
  configJson?: object;                      // .rundown/config.json contents
  workerHealthJson?: object;                // .rundown/worker-health.json contents
  initialScene?: SceneId;
}

interface HarnessResult {
  press(keys: string | string[]): Promise<void>;
  emit(event: ApplicationOutputEvent): Promise<void>;
  frame(): string;          // last-rendered frame as plain text (ANSI stripped)
  rawFrame(): string;       // last-rendered frame with ANSI
  sceneStack(): SceneId[];
  pendingProbes(): string[];
}

function createTuiHarness(opts: HarnessOptions): Promise<HarnessResult>;
```

The harness:

- builds an in-memory file system (already a port in rundown — reuse `MemoryFileSystem`-style adapter or create one).
- constructs `App` with all real domain code, only swapping the file system, clock, and worker executor for in-memory adapters.
- injects a fake stdout/stderr that captures rendered frames; key input goes through the same input handler the real TUI uses.
- exposes `emit()` so tests can simulate worker events without invoking real workers.

Test surface (one test file per scene plus one cross-scene file):

- `__tests__/presentation/tui/main-menu.test.ts` — six rows render; navigation up/down wraps; status probes called once on entry.
- `__tests__/presentation/tui/continue.test.ts` — preview shows next task; `Enter` → running; emitted phase events flip the badge through `scan → execute → verify → repair → finalize`; `done` summary shows correct counts.
- `__tests__/presentation/tui/new-work.test.ts` — missing `agent.md` shows recovery panel; `g` writes the template; ready path enters agent run view.
- `__tests__/presentation/tui/workers.test.ts` — pool, per-command, routing summary, and health column all render against a fixture config; `[H]` opens Health; `Esc` returns to Workers.
- `__tests__/presentation/tui/profiles.test.ts` — profiles list with workspace usage scan returns correct counts on a seeded workspace.
- `__tests__/presentation/tui/tools.test.ts` — custom tools discovered from `toolDirs` with shadowing; built-in catalog visibility toggles.
- `__tests__/presentation/tui/health.test.ts` — entries render; `[r]` resets; empty health file renders gracefully.
- `__tests__/presentation/tui/settings.test.ts` — scope cycle `effective → local → global → effective`; provenance shown only on effective; `[e]` blocked on effective.
- `__tests__/presentation/tui/help.test.ts` — local docs discovered; AGENTS.md template renders in pager; `[k]` overlay renders.
- `__tests__/presentation/tui/navigation.test.ts` — main → Workers → Health → Esc → Workers → Esc → main; arbitrary depth; quit works from any scene.

Snapshot strategy:

- Use `expect(frame).toMatchSnapshot()` for full-frame assertions on stable scenes (main menu, Workers, Settings).
- Use targeted substring assertions for dynamic scenes (Continue during a run) so snapshots don't churn on cosmetic changes.

## Implementation boundaries

Files expected to change:

- new `implementation/__tests__/presentation/tui/harness.ts`.
- new `implementation/__tests__/presentation/tui/<scene>.test.ts` per scene.
- new `implementation/__tests__/presentation/tui/__snapshots__/` directory.
- if the TUI input or render path is not already factored as pure functions, extract them in this migration so the harness can drive them without a real TTY. The smallest viable change: make the render loop a pure `(state, frame_n) → string` and the input handler a pure `(state, event) → state`.

Out of scope:

- Visual regression across terminals.
- Property-based tests.
- Tests that spawn real child processes for worker invocations. All worker calls go through the in-memory executor.
