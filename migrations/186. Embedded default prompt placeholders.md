Embedded default prompt placeholders

## Intent

Allow a project prompt override in `.rundown/` to extend its bundled default template instead of replacing it wholesale.

The concrete goal is to support a dedicated placeholder inside a project template file, so a user can write custom guidance before and after the built-in prompt body without copying the entire default prompt into the repo.

Recommended placeholder for this migration: `{{defaultTemplate}}`.

## Problem

Current template override behavior is all-or-nothing.

- `src/application/project-templates.ts` loads `.rundown/<name>.md` and falls back to the matching `DEFAULT_*_TEMPLATE` only when the file is missing, unreadable, or empty.
- If an override file exists and contains content, that file fully replaces the bundled default.
- This forces users to copy large default prompts just to add a short repository-specific preface or appendix.
- Existing prompt composition is ad hoc rather than general: root help explicitly concatenates `agent.md` and `help.md`, and plan guidance uses dedicated prepend/append files.

That makes prompt maintenance noisy and brittle:

- local prompt copies drift from upstream defaults
- small customizations require large duplicated files
- updating built-in prompts becomes harder for repositories with minor local tweaks

## Desired behavior

Support self-default embedding for project prompt overrides loaded from `.rundown/`.

Behavior contract:

1. If a project template file contains `{{defaultTemplate}}`, rundown replaces that token with the bundled default template for that same file.
2. If a project template file does not contain `{{defaultTemplate}}`, current full-override behavior remains unchanged.
3. If a project template file is missing, unreadable, or effectively empty after trim, current fallback behavior remains unchanged.
4. Replacement is single-file and single-pass: `execute.md` may embed the bundled `execute` default, `plan.md` may embed the bundled `plan` default, and so on.
5. Multiple occurrences of `{{defaultTemplate}}` in the same file are allowed and each occurrence expands to the same bundled default template string.

Example intended usage:

```md
You are working in a regulated repo. Follow these extra constraints first.

{{defaultTemplate}}

Before finishing, summarize any compliance-impacting changes.
```

## Scope

In scope:

- project templates loaded through `src/application/project-templates.ts`
- self-default embedding only
- documentation for the new placeholder contract
- tests covering fallback, composition, and unchanged runtime placeholder behavior

Out of scope:

- cross-template includes such as embedding `help.md` inside `execute.md`
- recursive include systems or arbitrary prompt inheritance trees
- `.rundown/tools/*.md` tool templates
- changing runtime `renderTemplate()` behavior in `src/domain/template.ts`
- introducing a general-purpose template language beyond the single placeholder above

## Non-goals and constraints

- Do not add `defaultTemplate` to runtime `TemplateVars`.
- Do not make `renderTemplate()` responsible for include resolution.
- Do not recursively re-process the expanded output for more template includes.
- Do not change root help ordering semantics; `agent.md` followed by `help.md` remains a separate composition rule.
- Do not change `rundown init` scaffolding unless there is a deliberate follow-up decision to seed example usage.

This should remain a loader-level feature, not a new global rendering rule.

## Implementation direction

Primary implementation point:

- `src/application/project-templates.ts`

Recommended refactor shape:

- replace the current simple helper that chooses `loadedTemplate || fallback`
- detect the three meaningful states for each file:
  - no override available -> return bundled default
  - override available without `{{defaultTemplate}}` -> return override as-is
  - override available with `{{defaultTemplate}}` -> replace all occurrences with bundled default and return the expanded string

Recommended boundary:

- keep all substitution local to project-template loading
- do not alter downstream prompt rendering or command-specific prompt assembly logic unless tests show a small compatibility gap

## Edge cases

- Missing override file: bundled default is used exactly as today.
- Whitespace-only override file: bundled default is used exactly as today.
- Override contains `{{defaultTemplate}}` plus normal runtime placeholders like `{{task}}`: only `{{defaultTemplate}}` is resolved at load time; runtime placeholders remain intact for later rendering.
- Override contains multiple `{{defaultTemplate}}` occurrences: all occurrences expand.
- Override contains an unknown placeholder such as `{{parentTemplate}}`: no special handling is introduced by this migration.
- Root help mode using `.rundown/help.md`: embedded default expansion in `help.md` should still participate correctly when root help later composes `agent` + `help`.

## Documentation contract

Update `docs/templates.md` so it explains:

- template overrides are still full replacements by default
- `{{defaultTemplate}}` is a special loader-supported escape hatch for extending the bundled default of the same file
- this feature is self-default only, not a general include system
- root no-arg help still has its separate `agent.md` + `help.md` composition contract

Documentation should include one short example showing how to extend `execute.md` or `plan.md` without copying the whole bundled template.

## Testing expectations

Primary test surface:

- `__tests__/application/project-templates.test.ts`

Required coverage:

- missing override still falls back to bundled default
- whitespace-only override still falls back to bundled default
- override without `{{defaultTemplate}}` remains a full override
- override with prefix and suffix around `{{defaultTemplate}}` expands correctly
- override with multiple `{{defaultTemplate}}` occurrences expands all of them
- ordinary runtime placeholders remain unresolved until normal render time
- existing per-template fallback behavior continues to work for unrelated templates

Optional narrow regression coverage:

- add or adjust help-task tests only if needed to confirm expanded `help.md` still composes under the existing root-help prompt order
