# 164. Tools scene

## Problem

Custom tool prefixes live as `.md` and `.js` files under `<config-dir>/tools/` (or any directory listed in `toolDirs`). Built-in handlers (`verify:`, `memory:`, `fast:`, `optional:`, `quit:`/`exit:`/`end:`/`break:`/`return:`, `include:`, `force:`, `profile=`) ship with rundown. Today there is no single place to:

- enumerate available custom tools,
- see and edit their prompt template,
- discover the built-in catalog and prefix-chain syntax.

## Goal

Build a Tools scene that lists custom tools (with their resolved file path) and shows the built-in catalog as read-only reference. `[e] edit prompt` opens a custom tool's file in `$EDITOR`.

## Model

Scene layout:

```
┌─ Tools ────────────── toolDirs: [tools, shared-tools] ─────────────┐
│                                                                    │
│  Custom (project)                                                  │
│    post-on-gitea     tools/post-on-gitea.md                        │
│        commands.tools.post-on-gitea overrides worker for this prefix│
│    summarize         tools/summarize.md                            │
│    triage-issue      shared-tools/triage-issue.js                  │
│                                                                    │
│  Built-in (read-only)                                              │
│    [b] toggle visibility                                           │
│                                                                    │
│  [↵] inspect template   [e] edit prompt   [r] reload tool dirs     │
└────────────────────────────────────────────────────────────────────┘
```

Behavior:

- Custom tool discovery walks each entry in `toolDirs` (resolved per [docs/configuration.md](implementation/docs/configuration.md): relative to `<config-dir>` unless absolute, in listed order). For each `.md` and `.js` file, the tool name is the file basename minus extension. If the same tool name appears in multiple directories, the first one wins (matches existing precedence) and the others are listed dimmed below as "shadowed".
- A tool that has a matching `commands.tools.<name>` override is annotated with the per-tool worker override summary on a sub-line.
- `Enter` on a `.md` tool opens a paginated read-only view of the template. `Enter` on a `.js` tool opens its source the same way.
- `[e]` opens the file in `$EDITOR` (uses `editor-launch.js` from 162).
- `[b]` toggles visibility of the built-in catalog. Hidden by default after first run; first-run state has it visible so new users see the built-ins.

Built-in catalog (read-only, hand-maintained list mirroring the docs):

```
Verify-only         verify: confirm: check:
Memory capture      memory: memorize: remember: inventory:
Fast execution      fast: raw: quick:
Conditional skip    optional: skip:
Terminal stop       quit: exit: end: break: return:
Include another md  include:
Outer retry wrapper force:
Modifier            profile=
```

Each row links to the relevant docs anchor.

## Implementation boundaries

Files expected to change:

- new `implementation/src/presentation/tui/scenes/tools.js`.
- new `implementation/src/presentation/tui/components/pager.js` — minimal less-style pager for read-only file content (reused by Help scene in 167).
- reuse `editor-launch.js` from 162.

Out of scope:

- Running a tool with a synthetic payload from the TUI.
- Generating new tool templates from a wizard.

## Acceptance

- Opening the scene on a workspace without `tools/` shows only the built-in catalog.
- Custom tools appear with correct file paths and `commands.tools.<name>` annotations when present.
- Shadowed tools render dimmed under the winning entry.
- `[e]` opens a custom tool's file in `$EDITOR` and reloads on return.
- `[b]` hides/shows built-ins; the choice persists for the current TUI session.
