Replace workspace placement with workspace mounts

## Intent

Generalize workspace path resolution so rundown no longer models layout primarily as:

- bucket directory name
- plus `sourcedir | workdir` placement
- plus one special external override for `design/current`

Instead, every filesystem lookup should resolve from a single authoritative mount table that maps logical rundown path prefixes to physical filesystem targets.

This migration is the core runtime/config change that enables:

- bare rundown directories that only hold `.rundown/` and related control files
- mounting user directories into any logical rundown path prefix
- treating `implementation`, `implementation/<subpath>`, `specs`, or `design/current` as external targets
- deterministic longest-prefix routing for nested workspace mappings

## Problem

The current workspace model is split across three different concepts:

- `workspace.link` chooses the effective workspace root
- `workspace.directories.<bucket>` renames top-level buckets
- `workspace.placement.<bucket>` selects either `sourcedir` or `workdir`
- `workspace.design.currentPath` special-cases one nested external mount

That model is sufficient for top-level bucket naming and coarse root selection, but it cannot express the real target behavior:

- map the invocation directory itself to `implementation`
- map the invocation directory itself to `specs`
- map `implementation/generated` to a different external tree than `implementation`
- map only selected subdirectories of the invocation tree while the rundown workspace remains elsewhere
- describe all of the above with one consistent resolution rule

It also leaves prompt/docs/runtime behavior describing placement semantics instead of the actual resolved path routing model.

## Desired behavior

Introduce a first-class mount table keyed by logical rundown path prefixes.

Conceptual config shape:

```json
{
  "workspace": {
    "mounts": {
      "design": "design",
      "design/current": "../notes/current-design",
      "implementation": "../app",
      "implementation/generated": "../shared/generated",
      "specs": "../tests"
    }
  }
}
```

Resolution contract:

- logical paths remain canonical rundown paths
- physical filesystem access must always resolve through the workspace mount table
- the most specific matching logical prefix wins
- mount targets may be project-relative or absolute
- resolved paths are normalized to absolute filesystem paths before use

Examples:

- `implementation/foo.ts` resolves through `implementation`
- `implementation/generated/schema.ts` resolves through `implementation/generated`
- `design/current/Target.md` resolves through `design/current`, while `design/rev.3/Target.md` still resolves through `design`

## Scope

In scope:

- define the authoritative mount-table config model
- add resolver utilities for logical path prefix matching and path materialization
- normalize existing workspace config surfaces into mounts for backward compatibility
- migrate path-sensitive command flows to consume resolved mounts instead of placement-derived paths
- replace special-case `design/current` handling with normal mount resolution
- update validation rules from overlap rejection to mount-aware ambiguity rejection

Out of scope:

- redesigning worker execution or artifact storage
- changing the meaning of `workspace.link` as workspace-root selection metadata
- moving existing user files automatically between directories
- removing old config keys in the same change; compatibility should remain during transition

## Compatibility contract

Legacy config must remain readable during the transition.

Compatibility normalization rules:

- `workspace.directories.<bucket>` + `workspace.placement.<bucket>` normalize into top-level bucket mounts
- `workspace.design.currentPath` normalizes into the `design/current` mount
- absence of explicit config produces the current default local layout mounts
- runtime consumers must prefer normalized mount resolution over recomputing `workspaceDir + directory + placement`

Compatibility goal:

- existing projects behave exactly as today when they do not opt into explicit mounts
- existing CLI/docs/tests using placement continue to pass until follow-up migrations move public surfaces

## Resolver contract

Add one authoritative resolver layer responsible for:

- loading normalized mount definitions
- validating logical mount keys
- validating target path syntax and normalization
- resolving a logical rundown path to an absolute filesystem path via longest-prefix match
- surfacing the resolved mount record that won, for diagnostics and prompt context when needed

Suggested invariants:

- logical mount keys use normalized forward-slash rundown paths
- empty keys are invalid
- duplicate logical keys are invalid
- ambiguous prefix shapes are allowed only when longest-prefix resolution is deterministic
- path traversal is allowed in relative targets only when explicitly intended by config; resolved absolute targets are the authority
- conflict validation applies to impossible or misleading configurations, not to intentional parent/child mount prefixes

## Runtime touchpoints

Primary consumers that should stop depending on placement math and consume mount resolution instead:

- workspace path resolution helpers in `implementation/src/application/workspace-paths.ts`
- design workspace resolution in `implementation/src/application/design-context.ts`
- runtime workspace context/template vars in `implementation/src/application/runtime-workspace-context.ts`
- path-sensitive command flows (`migrate`, `design`, `test`, `plan`, `research`, `run`, and related helpers)

All of those flows should operate on resolved absolute paths, not enum placement values.

## Validation and edge cases

Validate these cases explicitly:

- bare workspace root with all managed content mounted elsewhere
- nested mount override (`design/current` under `design`)
- invocation directory mounted as `implementation`
- invocation directory mounted as `specs`
- invocation directory mounted as workspace root with selective subpath mounts (`migrations` external, rest local)
- sibling and parent/child mount targets that resolve to overlapping physical paths intentionally
- stale absolute and relative mount targets with actionable errors

Rules to enforce:

- longest-prefix resolution must be deterministic
- no duplicate logical mount keys
- invalid target types fail fast with actionable config errors
- prompt/runtime metadata must report resolved absolute paths, not guessed placement semantics
