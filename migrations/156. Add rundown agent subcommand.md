# 156. Add rundown agent subcommand

## Problem

Today, running `rundown` with no arguments opens the interactive agent (via `createHelpCommandAction` at [src/presentation/cli.ts](implementation/src/presentation/cli.ts) and [src/presentation/cli-command-actions.ts](implementation/src/presentation/cli-command-actions.ts)). We are about to repurpose the no-args entry point to open the new TUI (see migration 157). Before swapping the root behavior, the existing agent flow must keep working under an explicit name so users and scripts have a stable invocation.

## Goal

Add a new `rundown agent` subcommand that is **functionally identical** to the current no-args invocation. Specifically, it must:

- accept the same flags currently mounted on the root: `-c, --continue`, `--agents`, `--config-dir <path>`, `--trace`,
- delegate to `app.helpTask(...)` with the same options the root action uses today,
- behave identically when stdout is non-TTY (print canonical agents text for `--agents`, otherwise fall through to interactive mode the same way the root currently does — non-TTY behavior parity),
- forward the worker pattern the same way (`--worker <pattern>` / `-- <command>` separator).

## Non-goal

- The root `rundown` (no-args) command keeps its current behavior in this migration. The swap to the TUI lands in 157.
- No change to `helpTask` itself.
- No change to documentation about `rundown` (no-args). 157 will update those.

## Model

`createHelpCommandAction` becomes the implementation for both:

1. The current root `.action(...)` at [src/presentation/cli.ts](implementation/src/presentation/cli.ts) — unchanged in this migration.
2. A new subcommand:

   ```ts
   program
     .command("agent")
     .description("Open the interactive rundown agent.")
     .option("-c, --continue", "Resume the previous interactive agent session")
     .option("--agents", "Print canonical AGENTS.md guidance and exit")
     .action(withCliAction(createHelpCommandAction({ ... })));
   ```

The factory does not need to change. Both call sites pass the same dependencies (`getApp`, `getWorkerFromSeparator`, `getInvocationArgv`, `outputHelp`, `cliVersion`).

`resolveInvocationCommand(invocationArgv)` in `createHelpCommandAction` currently treats only the no-args case as the agent path and falls back to `outputHelp()` otherwise. That logic must continue to work for the root path **and** must accept `agent` as the invocation command. Update the helper (or its check) so `agent` is recognized as an interactive invocation.

## Implementation boundaries

Files expected to change:

- `implementation/src/presentation/cli.ts` — add the `agent` subcommand definition.
- `implementation/src/presentation/cli-command-actions.ts` — extend `resolveInvocationCommand`/`createHelpCommandAction` so the `agent` invocation path is treated like the no-args invocation (TTY check, `--agents` short-circuit, `--continue` handling, `helpTask` delegation).
- `implementation/docs/cli-help.md` (or equivalent) — add a one-paragraph entry for `rundown agent`.
- `implementation/__tests__/presentation/...` — add a test asserting:
  - `rundown agent --agents` prints the canonical AGENTS.md text and exits 0,
  - `rundown agent` (no TTY) falls back to help-output exit 0,
  - `rundown agent` invokes `app.helpTask` with the resolved worker pattern.

Out of scope:

- Removing or deprecating the root no-args agent path (that happens in 157).

## Acceptance

- `rundown agent --agents` matches the current `rundown --agents` byte-for-byte.
- `rundown agent -c` resumes the previous agent session exactly like `rundown -c` does today.
- `rundown agent -- claude -p` and `rundown agent --worker '<pattern>'` route the worker the same way the root command does today.
- All pre-existing tests for `createHelpCommandAction` still pass without modification.
