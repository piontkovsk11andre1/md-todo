# 165. Health scene

## Problem

Worker health is a first-class concern in rundown ([docs/cli-worker-health.md](implementation/docs/cli-worker-health.md), `healthPolicy` in [docs/configuration.md](implementation/docs/configuration.md)). Today the CLI surfaces it via `rundown worker-health [--json]`. The TUI has no equivalent and no way to act on health entries (reset, manual probe).

## Goal

Build a Health scene that wraps `rundown worker-health` output, shows the `healthPolicy` summary, and offers actions to reset a single entry or trigger a probe.

## Model

Scene layout:

```
┌─ Health ────────────── unavailableReevaluation: cooldown ─────────┐
│                                                                   │
│  Workers                                                          │
│    default     opencode             ✓ ready                       │
│    fallback#1  claude               ✓ ready                       │
│    fallback#2  codex                ⚠ cooling_down  until 03:47   │
│                                       failure: usage_limit         │
│                                                                   │
│  Policy                                                           │
│    cooldown by failure class:                                     │
│      usage_limit             120s                                 │
│      transport_unavailable   0s                                   │
│      execution_failure_other 0s                                   │
│    failover budget:                                               │
│      maxFailoverAttemptsPerTask  0                                │
│      maxFailoverAttemptsPerRun   0                                │
│    fallbackStrategy:           strict_order                       │
│    unavailableReevaluation:                                       │
│      mode                  cooldown                               │
│      probeCooldownSeconds  300                                    │
│                                                                   │
│  [↵] view recent failures   [r] reset entry   [p] probe now       │
│  [e] edit healthPolicy in config.json                             │
└───────────────────────────────────────────────────────────────────┘
```

Behavior:

- Workers list reads from the same persisted health source as `rundown worker-health`.
- `[r]` on a selected entry calls the existing reset path (whatever `rundown worker-health` exposes for clearing entries; if no programmatic API exists, this migration adds a tiny port for it).
- `[p]` triggers a manual probe of the selected worker by attempting a no-op invocation and updating the entry on outcome. If a probe API does not exist yet, this action is gated behind a TODO and shows "probe not yet supported"; the rest of the scene still works.
- `[↵]` opens a paginated view of recent failure messages for the selected entry.
- Policy section is read-only here. `[e]` defers to the standard "edit `.rundown/config.json` in `$EDITOR`" pattern.

## Implementation boundaries

Files expected to change:

- new `implementation/src/presentation/tui/scenes/health.js`.
- `implementation/src/presentation/tui/bridges/health-bridge.js` (created in 162) — extend with reset and probe operations if the existing port allows.
- if no programmatic reset/probe exists today, add minimal additions to the relevant ports / use cases under `src/application/` and `src/infrastructure/adapters/`. Keep additions surgical; the goal is wrapping existing CLI semantics, not designing a new health subsystem.

Out of scope:

- Editing `healthPolicy` through a form.
- Bulk reset of all entries (useful, but follow-up).

## Acceptance

- Health scene shows entries identical in semantics to `rundown worker-health` output for the same config dir.
- `[r]` clears the selected entry and the row updates in place.
- `[e]` opens config in `$EDITOR` and reloads after edit.
- The scene gracefully renders with an empty health file (no entries yet).
